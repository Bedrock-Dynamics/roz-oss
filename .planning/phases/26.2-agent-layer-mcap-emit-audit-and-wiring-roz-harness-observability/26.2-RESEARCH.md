## Summary

**What the audit actually found (matters — CONTEXT.md framing understates scope):**

This is **not** a minor "cleanup" phase. Three of the six D-10 required variants (`ModelCallCompleted`, `ReasoningTrace`, `ToolCallStarted`, `ToolCallFinished`) have **zero production emit sites** today. `ModelCallCompleted` and `ReasoningTrace` have only builder structs (`roz-agent/src/observability.rs`) that are never called from the agent loop. The in-process tool-dispatch path (`agent_loop/dispatch.rs`) emits Phase-24 WAL checkpoint signals, not `SessionEvent`s — only the **remote** tool-call channel emits `ToolCallRequested` (session_runtime/mod.rs:1133, :1489). Asserting these variants in MCAP requires **new emit sites**, not just a mock provider.

**Core architectural finding:** `agent_loop.rs` is a misnomer in CONTEXT.md — the file does not exist; the module is `crates/roz-agent/src/agent_loop/` with six submodules. Almost every "agent-loop" `SessionEvent` is actually emitted by `crates/roz-agent/src/session_runtime/mod.rs`, which drains `StreamChunk`s + `PresenceSignal`s from the loop and translates them to `SessionEvent`s. The correct mental model is **`AgentLoop` (loop + provider + tool dispatch) → `SessionRuntime` (event translator) → `EventEmitter::emit` → broadcast → `spawn_cloud_ingestors` subscriber → `emit_session_event` → MCAP `/roz/session/events`**. Coverage-matrix emit-site columns MUST reflect this; CONTEXT.md's "emit from `agent_loop.rs`" framing is wrong.

**Primary recommendation:** Plan in 4 waves. (1) Audit-only — write the coverage matrix as a CONTEXT.md supplement. (2) Mock provider + mock tool as `test-helpers`-gated module in `roz-test`. (3) Wiring fixes — add `emit_session_event`-bound emit calls for `ModelCallCompleted` / `ReasoningTrace` in session_runtime at the point where `StreamChunk::Done` / assistant-parts land, and add in-process `ToolCallStarted`/`Finished` emits in `agent_loop/dispatch.rs` (adjacent to the existing `checkpoint_signal.tool_call_started/completed` Phase-24 calls). (4) Integration test + SC5 extension (either inline both drives or extract a shared helper — flag for planner).

### Clarification on silent-drop mechanics (CONTEXT.md #6 is imprecise)

`encode_session_event_proto` (`ingest_cloud.rs:225`) calls `event_envelope_to_session_response` which always returns `Response::SessionEvent(envelope_proto)` (`event_mapper.rs:557` → `:524`). ## Landmines

### 1. `#[cfg(test)]` does not propagate to integration tests

Already called out at `agent_loop/mod.rs:172-174`: `#[cfg(test)]` attributes don't transfer to integration-test binary builds. Use `test-helpers` feature flag + `#[cfg(feature = "test-helpers")]` for anything that needs to be reachable from `tests/*.rs`. See existing `#[doc(hidden)] pub` pattern (`test_mcap_args`) for harness entry points that shouldn't be production-callable.

### 2. `required-features` on test targets

New `[[test]] name = "mcap_agent_session_live" required-features = ["test-helpers"]` entry MUST be added to `crates/roz-server/Cargo.toml`. Without it, `cargo clippy -p roz-server --all-targets -- -D warnings` breaks when run without `--features test-helpers` (the existing `export_roundtrip` target at line 102-104 is the template).

### 3. Testcontainers Docker requirement

`pg_container()` (`roz-test/src/pg.rs:25`) starts a PostgreSQL:16-alpine container unless `DATABASE_URL` is set. CI must have Docker. The new test MUST be `#[ignore]`-gated so default `cargo test` doesn't fail in environments without Docker.

### 4. Container lifecycle: `std::mem::forget(guard)`

SC5 uses `std::mem::forget(guard)` after extracting the URL (`export_roundtrip.rs:76`) to prevent container auto-teardown at scope exit. The new test needs the same pattern OR must keep the guard alive via binding.

### 5. Broadcast-channel lag semantics

`tokio::sync::broadcast::Receiver` drops messages on subscriber lag. `spawn_cloud_ingestors` task 1 handles `RecvError::Lagged` explicitly (`ingest_cloud.rs:69-71`) with a warn log — in the new test, assertion on exact message COUNT is brittle if the MCAP ingestor lags behind the runtime's emission rate. Ensure the writer mpsc buffer is large enough (default in `spawn_writer` is presumably fine; check `mcap_archive.rs` if needed).

### 6. `chrono::Utc::now()` at two layers

- `EventEmitter::emit` stamps `envelope.timestamp` with `Utc::now()` (events.rs:40).
- `emit_session_event` overrides with `now_wall_clock_ns()` for MCAP `log_time_ns` (ingest_cloud.rs:136).

The envelope's chrono timestamp is INSIDE the protobuf payload (`SessionEventEnvelope.timestamp`), preserved. The MCAP file-level `log_time_ns` is wall-clock-at-emit. Tests that care about ordering must not mix the two.

### 7. Clippy workspace lints (pedantic + nursery)

New test files with long match arms / assertion tables frequently trip `clippy::too_many_lines`. Follow SC5's pattern at line 35-38:
```rust
#![allow(
    clippy::too_many_lines,
    reason = "integration test carries unavoidable fixture scaffolding"
)]
```

### 8. `async_trait` on Model

`Model` trait uses `#[async_trait]` (`types.rs:302`). Mock provider implementations must import `use async_trait::async_trait;` and mark `impl Model for ... { #[async_trait] async fn ... }`.

### 9. `SessionRuntime` construction is non-trivial

Look at existing call sites (e.g., `grpc/agent.rs` around line 2660+) for the constructor argument list. The current signature includes memory_store, blueprint_version, permissions, edge_state, snapshot, etc. Some can be defaulted (e.g., `MemoryStore::noop()`), others need explicit seeding. If this balloons, consider whether Path B's `AgentLoop` + `SessionRuntime` is still cheaper than Path A's `AgentService`.

### 10. Mock tool injection

The mock `hello_world` tool must be registered in `ToolDispatcher` before `AgentLoop::new(..)`. The dispatcher constructor is `crate::dispatch::ToolDispatcher` — check `crates/roz-agent/src/dispatch/mod.rs` for the registration API. Tool names exposed to the model come from `dispatcher.schemas()` (`agent_loop/core.rs:348`). Tool execution goes through `ToolExecutor` trait impls.

## Open Questions

### 1. Emit-channel design for agent-loop → session-runtime

**What we know:** `AgentLoop` owns `chunk_tx: mpsc::Sender<StreamChunk>` and `presence_tx: mpsc::Sender<PresenceSignal>` — both drained by `SessionRuntime::run_turn_streaming`. Neither channel carries raw `SessionEvent`s. `CheckpointSignal` (Phase 24) is a `dyn`-trait handle threaded via `AgentLoop::with_checkpoint_signal`.
**What's unclear:** Should the agent-event hook be:
- (a) a new `SessionEvent` channel `event_tx: mpsc::Sender<SessionEvent>` drained by SessionRuntime and re-emitted?
- (b) a `dyn AgentEventHook` trait handle like CheckpointSignal, with methods `on_model_call_completed(..)`, `on_tool_call_started(..)`, etc.?
- (c) expose `EventEmitter` directly on `AgentLoop` via `with_event_emitter(emitter)`?
**Recommendation:** (b) — matches the existing Phase 24 pattern, doesn't add channel plumbing, keeps the loop decoupled from the emitter type.

### 2. Full-fidelity SC5 vs. minimal SC5

**What we know:** D-11 mandates the SC5 fixture invoke "one scripted agent turn via the same mock provider path" — ambiguous between (a) full `SessionRuntime::run_turn_streaming` and (b) single `AgentLoop::run` call.
**What's unclear:** The LOC delta between the two. Full SessionRuntime fires additional `SessionStarted`/`ActivityChanged`/etc. which inflates the total-message-count assertion and needs tracking.
**Recommendation:** Extract a shared `drive_mock_agent_turn(&tx) -> AgentTurnCounts` helper used by both SC5 extension and the new integration test. Keep the SC5 counts honest (including the ActivityChanged noise) with a named constant `AGENT_TURN_SESSION_EVENTS = N`.

### 3. Collapse `ToolCallRequested` + `ToolCallStarted` for in-process non-approval tools?

**What we know:** For physical tools that go through approval, Requested → await approval → Started (after approval) → Finished — 3 distinct lifecycle transitions. For pure in-process tools with no approval, Requested and Started fire on the same stack frame with no intervening state.
**What's unclear:** Does downstream (substrate viewer) benefit from seeing both, or is one sufficient?
**Recommendation:** Emit both. Over-emission is better than under-emission for observability, and D-10 explicitly requires both.

### 4. `ModelCallCompleted` latency measurement

**What we know:** `latency_ms: u64` is a field on `ModelCallCompletedPayload`. No timer infrastructure exists around the model call today.
**What's unclear:** Does the latency include retry loops? Token streaming duration? Just the first-byte?
**Recommendation:** Wall-clock duration between the model invocation entry and the first-successful response — matches the spirit of existing OTel GenAI conventions. Use `std::time::Instant::now()` bracketing.

### 5. Does the `turn_event_rx` subscriber at `grpc/agent.rs:1635` filter variants?

**What we know:** The subscriber loops on the broadcast and calls `cloud_event_envelope_to_response` for every envelope (line 1660). No `match` filter — every variant goes out on the gRPC stream.
**What's unclear:** Are there OTHER subscribers in this path that might filter?
**Verified:** No — grep for `subscribe_events` shows only three sites: the turn-scoped relay at 1635, the session-scoped relay at 2682, and the MCAP ingestor at 1234. None filter. Open question RESOLVED ✅.

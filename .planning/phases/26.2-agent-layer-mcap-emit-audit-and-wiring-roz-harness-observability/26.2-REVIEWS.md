# Cross-AI Plan Review — Phase 26.2

## Codex Review

**Model:** gpt-5.4 (xhigh reasoning)

### Summary
The plan set is well structured and mostly respects the phase boundary: audit first, add a narrow emit hook, wire known gaps, then prove MCAP output. However, I would not execute it as-is. There are several high-risk issues around the mock-provider dependency location, tool-call lifecycle semantics, duplicate remote-tool events, and async MCAP test flakiness. These are fixable, but they should be corrected before Wave 2/3.

### Strengths
- The wave ordering is sound: Plan 01 audit, Plan 02 seam, Plan 03 mock provider, Plan 04 wiring, Plans 05/06 verification.
- The canonical event string risk is mostly handled. `ModelCallCompleted` correctly maps to `"model_call"` in `crates/roz-core/src/session/event.rs:170`.
- Plan 02's hook seam follows the existing `CheckpointSignal` pattern and keeps `roz-agent` decoupled from server runtime internals.
- Plans 05/06 correctly use `required-features = ["test-helpers"]` and `#[ignore]` for Docker-backed integration tests.
- Scope control is mostly good: no proto/schema/channel/build.rs changes, and D-17 lossy proto fixes stay deferred.

### Concerns

- **HIGH — Plan 03's dependency change is invalid or at least unproven.**
  `crates/roz-agent/Cargo.toml` already has `roz-test` as a dev-dependency at line 55. Plan 03 adds `roz-agent` as a normal dependency of `roz-test`, while its own acceptance says `grep -q 'roz-test' crates/roz-agent/Cargo.toml` should fail. It will not. This may create a dev-dependency cycle under `--all-targets`.

- **HIGH — Plan 03's mock provider does not satisfy its own non-streaming claim.**
  `StreamingMockModel::complete()` returns `"streaming mock complete() fallback"` with default token usage at `crates/roz-agent/src/model/types.rs:391`, not the D-06 canned response. The plan comment says non-streaming `complete()` returns the same response, but the proposed implementation does not.

- **HIGH — Plan 04's `ToolCallStarted` placement is semantically wrong for physical/approval-gated tools.**
  The existing checkpoint signal fires before safety evaluation at `crates/roz-agent/src/agent_loop/dispatch.rs:108`, explicitly to capture intent. `SessionEvent::ToolCallStarted` should not reuse that timing for physical tools. Emit `Requested` before safety, but emit `Started` only once execution actually begins after safety/approval.

- **HIGH — Plan 04 may duplicate `ToolCallRequested` for remote tools.**
  Remote tools already produce `ToolCallRequested` through `SessionRuntime`'s `tool_call_rx` drain at `crates/roz-agent/src/session_runtime/mod.rs:1489`. Adding a blanket dispatch-level `ToolCallRequested` will double-emit for `RemoteToolExecutor`, which sends `RemoteToolCall` at `crates/roz-agent/src/dispatch/remote.rs:175`.

- **MEDIUM — Plan 04 under-specifies `ReasoningTrace`.**
  `ReasoningTraceBuilder::new` requires a `turn_index` at `crates/roz-agent/src/observability.rs:14`, but `AgentLoop` does not currently know the `SessionRuntime` turn index. Creating a fresh builder per model call also resets `cycle_index` to 0.

- **MEDIUM — Plan 04's tool event template omits a required field.**
  `ToolCallStarted` requires `category` at `crates/roz-core/src/session/event.rs:247`. The plan notes caveats, but the main snippet is not compilable as shown.

- **MEDIUM — Plans 05/06 are vulnerable to MCAP finalize races.**
  `spawn_cloud_ingestors` runs in background tasks and sends `WriteCommand::Event` asynchronously at `crates/roz-server/src/observability/ingest_cloud.rs:63`. Finalizing immediately after `run_turn_streaming` can enqueue `Finalize` before all ingested event writes reach the writer.

- **LOW — Plan 04 adds `event_emitter_clone()` even though `event_emitter()` already exists.**
  `SessionRuntime::event_emitter()` already clones the emitter at `crates/roz-agent/src/session_runtime/mod.rs:625`. Adding an alias is harmless but unnecessary API churn.

### Suggestions
- Revise Plan 03. Do not put the mock provider in `roz-test` unless the dev-dependency cycle is explicitly validated. Prefer `roz-agent::model::mock_provider_v1` behind a `test-helpers` feature, or a new dedicated test-support crate that does not create a cycle.
- Make the mock provider implement both `complete()` and `stream()` with the exact D-06 response. Add a unit test that asserts chunks, parts, stop reason, `input_tokens=42`, `output_tokens=13`, tool id/name, and final text.
- In Plan 04, separate tool lifecycle semantics:
  - `ToolCallRequested`: emit once per non-remote local dispatch intent.
  - `ToolCallStarted`: emit only after safety/approval permits execution.
  - `ToolCallFinished`: emit for success, blocked, denied, timeout, and executor error.
- Add a `ToolSource`/metadata flag to `ToolDispatcher` entries, or otherwise skip dispatch-level `ToolCallRequested` for `RemoteToolExecutor` to avoid duplicates.
- For `ReasoningTrace`, either defer it per D-10 warning status or pass runtime turn metadata into the executor/agent input so `turn_index` and `cycle_index` are correct.
- In Plans 05/06, add a deterministic drain barrier before finalizing MCAP. A fixed sleep is weaker; better is to wait until a side subscriber observes `TurnFinished` plus all required gap events, then yield until the ingestor has had a chance to enqueue before sending `Finalize`.
- In Plan 05, use the actual proto path: likely `roz_server::grpc::roz_v1::SessionEventEnvelope`, not `roz_server::proto::roz::v1`.
- Add a small Plan 04 unit test for `SessionRuntimeEventHook` routing into `EventEmitter`; this catches the adapter independently of the heavy MCAP tests.

### Risk Assessment
**Overall risk: HIGH until revised.** The design direction is good, but Plan 03 can break the dependency graph, Plan 04 can emit misleading or duplicate tool lifecycle events, and Plans 05/06 can be flaky because MCAP finalization races background ingest. After those corrections, the risk drops to medium: the remaining complexity is mainly integration-test scaffolding and Docker-backed determinism.

### Agreed Concerns (single-reviewer — treat as "worth investigating")

**Top-priority fixes before execute:**
1. **HIGH — Dependency graph (Plan 03):** `roz-test` already depends on `roz-agent` via dev-deps, or vice versa. Verify direction; cycle risk under `--all-targets`. Consider relocating `mock_provider_v1` to `roz-agent::model` behind a `test-helpers` feature instead.
2. **HIGH — Mock provider `complete()` path (Plan 03):** `StreamingMockModel::complete()` has a fallback that does NOT return the D-06 response. The new mock must override `complete()` explicitly OR the phase commits to streaming-only usage everywhere.
3. **HIGH — Tool lifecycle semantics (Plan 04):** `ToolCallStarted` must fire AFTER safety/approval gates, not at dispatch intent. Current Plan 04 placement conflates `Requested` and `Started`.
4. **HIGH — Double-emit for remote tools (Plan 04):** `SessionRuntime` already emits `ToolCallRequested` for remote tools at `session_runtime/mod.rs:1489`. Dispatch-level emit must be skipped for `RemoteToolExecutor` or the test will see duplicate events.
5. **MEDIUM — `ReasoningTrace` requires `turn_index` (Plan 04):** `ReasoningTraceBuilder::new(turn_index)` — agent loop does not carry this. Either plumb it or defer ReasoningTrace per D-10 WARNING-not-BLOCKER.
6. **MEDIUM — `ToolCallStarted.category` field missing in Plan 04 snippets** — plan won't compile as shown.
7. **MEDIUM — MCAP finalize race (Plans 05/06):** background `spawn_cloud_ingestors` may not flush before `Finalize`. Add a drain barrier (observe terminal event + yield) not a fixed sleep.
8. **LOW — `event_emitter_clone()` is API churn:** `SessionRuntime::event_emitter()` already exists at `session_runtime/mod.rs:625` and clones. Use the existing method.

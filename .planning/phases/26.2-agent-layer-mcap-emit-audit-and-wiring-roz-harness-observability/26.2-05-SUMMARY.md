**Test drive (Path B — direct AgentLoop + SessionRuntime):**

**Drain barrier (REVIEWS.md M3):**
```
async fn await_turn_drain(
    mut rx: broadcast::Receiver<EventEnvelope>,
    timeout: Duration,
) -> DrainReport
```
Polls the side subscriber with `tokio::time::timeout(remaining, rx.recv())` in a loop; breaks early once `TurnFinished` AND all 5 gap events are observed. Tail of 8x `tokio::task::yield_now()` gives the cloud ingestor's background task scheduler ticks to enqueue its collected `WriteCommand::Event` mpsc sends. `DrainReport` exposed in every failure message so regressions surface the specific gap from the broadcast (not from MCAP readback, which can't show what the ingestor never received).

**Finalize + readback:**
`WriteCommand::Finalize { reason: SessionCompleted }` → drop sender → 50x 100ms SQL-row-transition poll → `std::fs::read(mcap_file)` → `MessageStream::new(&data)` iteration decoding `roz_v1::SessionEventEnvelope` on `/roz/session/events`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Corrected MockProviderV1 turn semantics to unblock tool dispatch**

- **Found during:** Task 2 integration test first run.
- **Issue:** Plan 03's `MockProviderV1` returned a single `CompletionResponse` with `ContentPart::ToolUse` AND `stop_reason = EndTurn`. But `agent_loop/core.rs:476` has `if resp.stop_reason == StopReason::EndTurn || resp.tool_calls().is_empty() { break; }` — the loop short-circuits on `EndTurn` BEFORE `dispatch_tool_calls` runs. Result: Plan 04's in-process `ToolCallRequested` / `ToolCallStarted` / `ToolCallFinished` emit sites in `agent_loop/dispatch.rs` were unreachable, and the test's D-10 BLOCKING assertions failed:
  ```
  drain report: DrainReport { saw_tool_call_requested: false, saw_tool_call_started: false,
                              saw_tool_call_finished: false, ... }
  ```
- **Fix:** Made `MockProviderV1` stateful with an `AtomicU32` call counter. Call #1 returns `Thinking + ToolUse` with `stop_reason = ToolUse` (drives dispatch). Call #2 returns `Text "Done."` with `stop_reason = EndTurn` (terminates the turn). This matches real-provider (Anthropic / OpenAI) turn semantics and preserves D-06 canonical token usage (42 in / 13 out on every call, so `ModelCallCompleted` fidelity assertions still hold).
- **Files modified:** `crates/roz-agent/src/model/mock_provider.rs` (also updated 3 unit tests → 4 tests to exercise both calls + the streaming chunk sequence for each).
- **Commit:** `4f07cf9`
- **Rationale:** Rule 1 (bug in Plan 03's mock semantics, not architectural). Blast radius: 1 file, no production code change, no new trait or dep, no schema change. The corrected mock still satisfies every D-05/D-06 semantic clause in CONTEXT.md (deterministic, instant, `test-mock-v1` id, hello_world tool with `{"name": "world"}` input, final "Done." text, `stop_reason = end_turn` on the terminal response).

**2. [Adjustment - not a deviation] Integration test asserts TWO model_call events (consequence of Rule 1 fix)**

The stateful 2-call mock emits two `ModelCallCompleted` envelopes per turn. The test's fidelity assertion was updated to:
- Loop over `mcc_events` and assert every envelope carries `model_id="test-mock-v1"`, `input_tokens=42`, `output_tokens=13`.
- Find **exactly one** `ModelCallCompleted` with `stop_reason="end_turn"` (terminal check).
- Prove **exactly one** `ModelCallCompleted` with `stop_reason="tool_use"` exists (regression fence — if someone collapses back to a single model_call event, this surfaces the regression).

The plan's `must_haves.truths` line "ModelCallCompleted carries model_id='test-mock-v1', input_tokens=42, output_tokens=13" is ambiguous about "which one"; terminal is the defensible reading, and the test now asserts the payload on every envelope so both are covered.

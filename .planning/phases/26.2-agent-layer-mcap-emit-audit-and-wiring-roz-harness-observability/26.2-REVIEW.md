# Phase 26.2 Code Review Report

**Reviewed:** 2026-04-22
**Depth:** standard (per-file analysis; emit-site audit across 5 agent-loop files; cross-file hook-installation audit across all 8+ `AgentLoop::new` production sites)
**Files Reviewed:** 14 source + 1 new integration test
**Status:** has_issues (1 medium, 2 low, 4 info — no blockers; phase is mergeable)

## Summary

Phase 26.2 closes the agent-loop MCAP emit gaps on the cloud-gRPC session path in a way that is internally correct, well-documented, and well-tested. The `AgentEventHook` seam is narrowly scoped (single method, `Send + Sync`, fire-and-forget) and mirrors the Phase 24 `CheckpointSignal` pattern; `SessionRuntimeEventHook` is a thin adapter over an already-cloneable `EventEmitter`. All six SC3 required variants (`TurnStarted`, `ModelCallCompleted`, `ToolCall{Requested,Started,Finished}`, `TurnFinished`) are verified end-to-end by `mcap_agent_session_live.rs`.

The core issue is one of **phase-scope boundary**, not correctness: the hook is installed only at `crates/roz-server/src/grpc/agent.rs:1639`, leaving worker-side (`roz-worker`) and local-runtime (`roz-local`, `roz-cli`) `AgentLoop` constructions with the default `NoopAgentEventHook`. For worker in particular, which has its own MCAP writer pipeline, this means agent-loop-originated events (ModelCallCompleted, in-process ToolCall{Requested,Started,Finished}) currently go nowhere from edge sessions. The phase plan does not list this as in-scope, and `deferred-items.md` doesn't call it out — this is a scope gap worth a follow-on phase.

All other observations are low-severity: a Pure-branch emit-ordering subtlety that downstream consumers pairing events by adjacency must account for, a `Started → dispatch.await → Finished` pattern that can drop `Finished` on mid-await future cancellation, and two Info-level notes on tool-input secret exposure and `mem::forget(pg_container_guard)`.

Feature gating for the `test-helpers` mock provider is airtight. `AtomicU32` counter usage is correct. No proto changes. No `unsafe_code`. Pre-commit (`cargo fmt`, `cargo clippy -D warnings`) compliance preserved.

## Medium Issues

### MR-01: AgentEventHook installed only on the cloud-gRPC path — worker and local runtimes silently drop agent-loop session events

**File:** `crates/roz-worker/src/main.rs:1210-1214`, `crates/roz-worker/src/session_relay.rs:441-443`, `crates/roz-local/src/runtime.rs:406,1230`, `crates/roz-cli/src/commands/non_interactive.rs:183`, `crates/roz-cli/src/tui/mod.rs:1178`

**Issue:** `AgentLoop::new(...)` is called in at least 8 production sites across the workspace. Only `crates/roz-server/src/grpc/agent.rs:1634-1639` chains `.with_agent_event_hook(Arc::new(SessionRuntimeEventHook::new(...)))`; every other production site defaults to `NoopAgentEventHook` at `agent_loop/mod.rs:111`.

For `roz-worker` in particular this is load-bearing: worker builds its own `SessionRuntime` (same `session_relay.rs` call) and wires an MCAP emission pipeline through `session_event_tx` / `spawn_session_event_forwarder` (`crates/roz-worker/src/main.rs:636-637, 1941, 2330-2358`). The forwarder only receives events published to `session_mpsc_tx`; because the worker-side AgentLoop holds a `NoopAgentEventHook`, every `ModelCallCompleted` and in-process `ToolCall{Requested,Started,Finished}` is silently dropped before it can reach the broadcast bus that feeds worker-side MCAP. The SC3-fixed gaps therefore remain open on the edge-execution path even after this phase merges.

The phase plan and `deferred-items.md` do not mention this. `CONTEXT.md:120` names only `grpc/agent.rs` as the forwarding layer covered by this phase. This is a **scope gap**, not a correctness defect in the code that was written — but it is a user-visible "MCAP contents differ between cloud and edge sessions" regression risk that should be explicitly tracked, not discovered later.

**Fix:** Either (a) install the hook symmetrically on the worker-side `AgentLoop` in `session_relay.rs:441` and `worker/src/main.rs:1210`, using the same `session_runtime.event_emitter()` pattern as `grpc/agent.rs:1630-1633`; or (b) add an entry to `deferred-items.md` declaring worker + local AgentLoop hook installation as a follow-on phase (likely 26.2.1 or 26.3 scope), so downstream integration testers know MCAP parity between cloud and edge does not land in 26.2.

Minimal worker-side snippet (matches grpc/agent.rs pattern):

```rust
// In worker/src/session_relay.rs::build_agent_loop, after constructing
// session_runtime but before AgentLoop::new:
let agent_event_emitter = self.session_runtime.event_emitter();

let agent = AgentLoop::new(model, dispatcher, safety, spatial)
    .with_extensions(extensions)
    .with_approval_runtime(approval_runtime)
    .with_agent_event_hook(std::sync::Arc::new(
        SessionRuntimeEventHook::new(agent_event_emitter),
    ));
```

Recommendation: pick (b) for 26.2 (preserve phase scope) and file 26.2.1 immediately so the gap does not drift.

## Low Issues

### LR-01: Started → dispatch.await → Finished can drop Finished on mid-await cancellation

**File:** `crates/roz-agent/src/agent_loop/dispatch.rs:123-150` (Pure branch), `:194-208` (Physical Approved branch), `crates/roz-agent/src/agent_loop/approvals.rs:242-269` (granted approval branch)

**Issue:** Each branch emits `SessionEvent::ToolCallStarted`, then awaits `dispatcher.dispatch(...)`, then emits `ToolCallFinished`. There is no structured `Drop`-guard or `scopeguard`-style finalizer around the await; if the enclosing future is dropped mid-dispatch (for example, if the gRPC session stream is closed by the client while a physical tool is executing), `Started` has already been broadcast but `Finished` will never be emitted. Downstream consumers that pair Started/Finished by `call_id` will see dangling Starteds.

Note that the cooperative cancellation check at `core.rs:152-160` only runs at the top of each cycle loop, so this is specifically a future-drop concern (drop → outer `tokio::spawn` abort, `tokio::select!` loser, client disconnect), not the in-loop cancellation path.

**Fix:** Low-severity because (a) broadcast-bus subscribers can reasonably reconcile by call_id correlation even without Finished, and (b) real-world cancellation rarely lands inside a single tool dispatch window. If a fix is desired, wrap the dispatch in a scope-guard struct that emits a `Finished { result_summary: "cancelled before completion" }` in `Drop`. Example:

```rust
struct FinishedGuard<'a> {
    hook: &'a dyn AgentEventHook,
    call_id: String,
    tool_name: String,
    fired: bool,
}
impl Drop for FinishedGuard<'_> {
    fn drop(&mut self) {
        if !self.fired {
            self.hook.on_agent_event(SessionEvent::ToolCallFinished {
                call_id: std::mem::take(&mut self.call_id),
                tool_name: std::mem::take(&mut self.tool_name),
                result_summary: "cancelled before completion".to_owned(),
            });
        }
    }
}
```

Defer to Phase 26.3 — not a 26.2 blocker.

### LR-02: Pure-branch Started emits are not adjacent to their Finished emits on the wire

**File:** `crates/roz-agent/src/agent_loop/dispatch.rs:93-153`

**Issue:** All Pure-segment `ToolCallStarted` events fire inside the `iter().map()` closure at `:123` — **before** any future has been awaited. `join_all(pure_futures)` then runs all dispatches concurrently and the Finished emits happen in `join_all` completion order at `:146`. This means the broadcast ordering for a segment of N Pure calls is:

```
Started(c1), Started(c2), ..., Started(cN),
[concurrent dispatch]
Finished(c_i), Finished(c_j), ..., Finished(c_k)  // completion order, not call order
```

Downstream consumers that expected `Started(c_i) → Finished(c_i) → Started(c_{i+1})` adjacency (a reasonable default for a serial Physical branch) will see no such pairing for Pure segments. `call_id` correlation always works, but a naive MCAP consumer pairing by event-stream adjacency will mis-pair.

**Fix:** Not a code change — this is a documentation note. Either (a) document the ordering contract on `SessionEvent::ToolCallStarted` / `ToolCallFinished` doc-comments, calling out that Pure-segment dispatch interleaves; or (b) move the Started emit inside the `async move {}` block at `:128` so Started and Finished are at least both inside the per-call future and race on scheduling. Option (b) is safer for consumers but has a subtle timing cost (Started is delayed by one scheduler yield). Recommend (a).

## Info Items

### IR-01: ToolCallRequested.parameters contains raw tool inputs — potential secret leakage into MCAP

**File:** `crates/roz-agent/src/agent_loop/dispatch.rs:112-118, :177-183`

**Issue:** Both in-process emit sites clone `c.params` (the raw JSON tool input) directly into `SessionEvent::ToolCallRequested::parameters`, which rides the broadcast bus into MCAP and, for cloud sessions, eventually into the offsite archive pipeline. If any tool accepts secrets in its input (API keys, bearer tokens, credentials for pass-through tools), those secrets land in session archives.

This is not newly introduced by Phase 26.2 — the remote-path emit at `session_runtime/mod.rs:1489` has the same shape — but the new in-process sites widen the exposure surface without any redaction policy.

**Fix:** Out of 26.2 scope. Worth filing as a follow-on hardening item: define a `ToolParameterRedactor` trait or per-tool `redact_for_telemetry` hook, and apply it uniformly at both emit sites. For now, any tool author bundling secrets in tool inputs is already exposed via the pre-existing remote path, so the phase does not materially regress security posture.

### IR-02: MockProviderV1 call counter is per-instance; ok for current tests but verify on reuse

**File:** `crates/roz-agent/src/model/mock_provider.rs:57-68, :117-124`

**Issue:** `call_count: AtomicU32` is per-`MockProviderV1` instance. Because `mock_provider_v1()` returns a freshly-constructed `Box<dyn Model>` each call, each test gets its own counter — concurrent test parallelism is fine. `Ordering::SeqCst` is stronger than needed (a single-thread increment would be fine with `Relaxed`, multi-thread-safe with `AcqRel`), but it is correct.

The one edge case to be aware of: if a future test ever reuses a single `MockProviderV1` instance across multiple `AgentLoop::run` invocations (e.g., multi-turn test fixture), call #3+ will keep returning `end_turn_response()`, which silently skips the `ToolCallRequested` emit for turn 2. Not a defect; a usage foot-gun worth a doc-comment note on `MockProviderV1::new`.

**Fix:** Add a doc-comment on `MockProviderV1::new`: *"Intended for single-turn fixtures. Reusing a single instance across multiple `AgentLoop::run` calls will return `end_turn_response()` on every call after the second, skipping the ToolCallRequested emit path."*

### IR-03: Model-call error paths emit no SessionEvent

**File:** `crates/roz-agent/src/agent_loop/core.rs:403-407`

**Issue:** When `complete_with_retry` or `stream_and_forward_with_retry` propagate an error via `?`, no `ModelCallCompleted` is emitted and there is no `ModelCallFailed` variant in `SessionEvent`. A turn that fails inside the model call leaves a visible `TurnStarted` with no corresponding `ModelCallCompleted` on the bus. Downstream MCAP readers computing cycle counts from `ModelCallCompleted` events will undercount on failure paths.

This is consistent with current `SessionEvent` shape (no `ModelCallFailed` variant) — plan scope explicitly bars proto changes. Filing purely as a tracking note for Phase 26.3 or a targeted event-taxonomy follow-on.

**Fix:** None in 26.2. Add a `ModelCallFailed` variant in a future phase; emit on the error path in `core.rs:406-407`. Alternatively, document that `ModelCallCompleted`-absence with `TurnFinished { status: failure }` is the canonical failure-cycle signature.

### IR-04: `std::mem::forget(guard)` in integration test leaks testcontainer on process crash

**File:** `crates/roz-server/tests/mcap_agent_session_live.rs:215`

**Issue:** `std::mem::forget(guard)` is used to keep the Postgres testcontainer alive past the guard's scope. This is a common pattern for testcontainer fixtures but will leak a Docker container if the test process is killed mid-test (e.g. CI timeout, `panic::set_hook` abort). Not a correctness defect; a CI-hygiene note.

**Fix:** None required. If CI flakes on leaked containers, switch to an explicit `OnceCell<Guard>` held in a static `LazyLock` so Docker's reaper gets a proper Drop on normal termination.

## Recommendations

1. **Before merging:** Decide on the MR-01 scope resolution. Recommended path: amend `deferred-items.md` with a 26.2.1 follow-on entry ("worker + local AgentLoop hook parity") and merge 26.2 as-is. Do **not** open-bake the worker-side install into 26.2 — it widens the surface area of an already-verified phase.

2. **Before Phase 26.3 starts:** Address LR-01 (scope-guard around Started/dispatch.await/Finished). This closes a latent "dangling Started" edge case that will become more visible as downstream consumers rely on paired events.

3. **Document after merge:** Add a Rustdoc note to `SessionEvent::ToolCallStarted` / `ToolCallFinished` describing the Pure-segment interleave (LR-02) and a doc-comment on `MockProviderV1::new` warning against multi-turn reuse (IR-02).

4. **Separate hardening phase:** IR-01 (raw tool-parameter leakage into telemetry) deserves a dedicated small phase — it is a pre-existing issue but the in-process emits add new call sites that would benefit from a redactor pattern established once.

5. **No action needed:** IR-03 and IR-04 are tracking notes only. The mock-provider counter (IR-02 concurrency angle) is fine as-is.

## What Was Reviewed and Found Clean

- **Hook seam design** (`roz-core/src/agent_event_hook.rs`): narrow trait, `Send + Sync`, fire-and-forget contract, well-documented. Mirror of `CheckpointSignal`. No issues.
- **SessionRuntimeEventHook adapter** (`session_runtime/mod.rs:2582-2610`): thin forward to cloneable `EventEmitter`; three unit tests cover happy-path routing and no-subscriber non-panic behavior.
- **Feature gating** (`roz-agent/Cargo.toml:14` + `roz-server/Cargo.toml:22, :108-124`): `test-helpers` is declared, mock module is `#[cfg(feature = "test-helpers")]`, re-export is same-gated, integration tests use `required-features`. Mock provider cannot leak into production builds.
- **Emit-site completeness for NON-error paths**: all six D-10 required variants verified present by `mcap_agent_session_live.rs`. `ToolCallFinished` fires on every return path (`SafetyResult::Approved`, `Blocked`, `NeedsHuman`+runtime-missing, granted approval, denied approval, timeout, channel-closed). `ToolCallStarted` correctly gated to post-safety / post-approval paths per REVIEWS.md H3.
- **Remote vs in-process dispatch split** (`dispatch/mod.rs:67-69` + `dispatch/remote.rs:157-159`): `ToolExecutor::is_remote()` default-`false` with `RemoteToolExecutor` override is a minimal, correct seam. Prevents double-emit cleanly.
- **MockProviderV1 atomic counter**: per-instance, `AtomicU32` with `SeqCst`. No test-parallelism hazard.
- **No proto changes**: verified via the file list — no `.proto` modifications.
- **No `unsafe_code`**: workspace lint still denies; no new `#[allow(unsafe_code)]` introduced.
- **Integration test determinism** (`mcap_agent_session_live.rs`): `await_turn_drain` subscriber-based barrier replaces any fixed sleep for the broadcast-drain step; the only remaining `sleep` is a polling loop for SQL status transition AFTER `Finalize`, which is correct per REVIEWS.md M3.

# Phase 26.2: Agent-layer MCAP emit audit and wiring — Context

**Gathered:** 2026-04-21
**Status:** Ready for planning
**Source:** Post-Phase-26.1 architectural refinement. Context captured autonomously per `--auto` discuss-phase mode after multi-turn scoping conversation that covered MCAP/ULOG/copper/RRD format strategy and the "Roz agent-harness observability for robotics" thesis.

<domain>
## Phase Boundary

**What this phase delivers:** Verify or make true that every `SessionEvent` variant representing agent-loop activity actually fires from the production `crates/roz-agent/src/agent_loop.rs` code path (and its downstream emit sites) into the `/roz/session/events` MCAP channel. Deliver a coverage matrix, close any wiring gaps (no new schemas), and ship an integration test that drives a real 1-turn agent session against a mock model provider and asserts the expected SessionEvent variant set appears in the resulting MCAP with correct payloads.

**What this phase does NOT deliver:**
- No new proto schemas. `SessionEventEnvelope`, `ToolCallEvent`, and `TaskLifecycleEvent` are already correct — the protos carry the right shape; this phase is wiring-only.
- No new channels. The existing six MCAP channels (`/tf`, `/roz/telemetry/pose`, `/roz/log`, `/roz/session/events`, `/roz/task/lifecycle`, `/roz/tool/calls`) stay as-is.
- No trace-context fields (that's Phase 26.3).
- No metadata index table (that's Phase 26.4).
- No new camera/pointcloud channels (that's Phase 26.5).
- No viewer integrations — substrate is the downstream consumer; this phase emits bytes, substrate renders them.

**Scope anchor:** This phase anchors the "Roz agent-harness observability for robotics" thesis — every agent-loop action must become a first-class MCAP event that substrate can render alongside physical state. The phase ends when every `SessionEvent` variant listed in the coverage matrix below is either verified-emitting or has a wiring fix landed.
</domain>

<decisions>
## Implementation Decisions

### Audit strategy (locked)
- **D-01:** Audit-first workflow. Before any code change, produce a **coverage matrix** in this CONTEXT.md supplement (separate section added during planning) with one row per `SessionEvent` variant × production emit site × fixture coverage status.
- **D-02:** Source of truth for "should emit" is the variant list in `crates/roz-core/src/session/event.rs::SessionEvent` (~60 variants — see `<canonical_refs>` for exact file/line). Every variant gets a row.
- **D-03:** Classify each variant as one of:
  - `agent-loop` (core reasoning/turn lifecycle — MUST emit from `agent_loop.rs`)
  - `tool-dispatch` (tool call path — MUST emit from `agent-loop::execute_tool_call` or `bt/action.rs`)
  - `approval` (human-gated — already verified working by Phase 26 SC5 fixture)
  - `safety` (edge safety filter — MUST emit from copper safety filter path or `roz-worker` safety filter)
  - `memory` (memory subsystem)
  - `controller` (WASM controller lifecycle — load/shadow/promote/rollback)
  - `telemetry` (telemetry status, trust posture, transport degradation)
  - `infrastructure` (resume/snapshot/failure — lifecycle plumbing)
- **D-04:** For each row, record: `variant_name | classification | emit_site(s) found by grep | is_emit_actually_fired_in_a_test | is_variant_landing_in_MCAP_via_fixture`.

### Mock model provider (locked)
- **D-05:** Build a **deterministic** mock model provider in `crates/roz-test/src/mock_provider.rs`. Canned `AgentTurn` with scripted tool calls + text + stop_reason. No network. Reusable for phases 26.3, 26.4, 26.6, and any future test that wants "1 turn, no LLM vendor".
- **D-06:** Canned response shape:
  - `model_id = "test-mock-v1"`
  - 1 thinking block (exercises `ThinkingDelta` + `ReasoningTrace` if the loop emits)
  - 1 tool_use block: tool name `"hello_world"`, parameters `{"name": "world"}`
  - 1 text block: `"Done."`
  - `stop_reason = "end_turn"`
  - Token counts: input=42, output=13, cache_read=0, cache_creation=0 (deterministic)
  - Latency: instant return (no artificial delay)
- **D-07:** Mock provider implements the same trait the real Anthropic/OpenAI providers implement (whatever that trait is — verify during audit step). Returns `AgentTurn` via a single function call or an async stream, matching however the real providers return.
- **D-08:** Tool call execution path uses an injected mock tool that returns a canned success (no actual side-effect work). Defined inline in the test, not in `roz-test`.

### Integration test shape (locked)
- **D-09:** New integration test file `crates/roz-server/tests/mcap_agent_session_live.rs` — one test that:
  1. Spins up testcontainers Postgres (reuse `roz_test::pg_container`).
  2. Builds an `AppState` with the mock provider wired into `roz-agent::AgentLoop`.
  3. Starts a session via the gRPC AgentService (or directly via AgentLoop if simpler — judgment call left to planner).
  4. Submits one user prompt, drives one turn.
  5. Waits for `SessionCompleted` or equivalent terminal event.
  6. Finalizes the MCAP, re-reads via `mcap::MessageStream`.
  7. Asserts the `/roz/session/events` channel contains messages with each of the expected variants (see D-10).
- **D-10:** Expected variant presence (assert at least 1 message of each kind, payloads match mock response):
  - `TurnStarted { turn_index: 0 }`
  - `ModelCallCompleted { model_id: "test-mock-v1", input_tokens: 42, output_tokens: 13, stop_reason: "end_turn", ... }`
  - `ReasoningTrace` (if the agent loop produces a thinking summary — classify this as "emit if active" not mandatory; leave as WARNING not BLOCKER if absent in v1)
  - `ToolCallRequested { tool_name: "hello_world", parameters: {name: "world"}, ... }`
  - `ToolCallStarted { tool_name: "hello_world", call_id: <same id>, ... }`
  - `ToolCallFinished { tool_name: "hello_world", result_summary: <non-empty>, ... }`
  - `TextDelta` OR `TurnFinished` — at least one of these carries the final text output. Both should be emitted if the loop produces them.
  - `TurnFinished { stop_reason: "end_turn", input_tokens: 42, output_tokens: 13, ... }`

### SC5 fixture extension (locked)
- **D-11:** Extend `crates/roz-server/tests/export_roundtrip.rs` (the SC5 fixture) to invoke one scripted agent turn via the **same mock provider path** used by the new integration test — not via direct `WriteCommand::Event` injection. This closes the fidelity gap the Phase 26.1 conversation exposed (session events bypassed the production encoder path before).
- **D-12:** Update SC5 message-count assertions to reflect the new emits. Preserve existing assertions (30s telemetry, approvals, task lifecycle, tool-call triplets).
- **D-13:** Keep SC5's deterministic pinned-epoch property where possible; if the agent-turn path forces wall-clock timestamps (e.g. `emit_session_event` internal `now_wall_clock_ns()`), accept that and document.

### Wiring gap closure (locked)
- **D-14:** If the audit finds a variant classified `agent-loop`, `tool-dispatch`, `safety`, or `memory` that is NOT emitted from the production path, add the minimal emit call at the semantic correct site. Prefer to emit via whatever mechanism the other variants at that site already use (likely the session event broadcast channel consumed by `emit_session_event`).
- **D-15:** Do NOT add emits for variants classified `infrastructure` unless explicitly directed. Some are legitimately not-yet-wired because the subsystem doesn't emit anything at the boundary yet (e.g. some resumability events may only matter post-reconnection flows that are not exercised here).
- **D-16:** Every emit-site addition must be accompanied by an assertion in either the new integration test or the extended SC5 fixture.

### Out of scope (locked)
- **D-17:** Zero proto changes in this phase. If a variant's current proto mapping in `crates/roz-server/src/grpc/event_mapper.rs` is found to be lossy (fields drop, truncation), document in the audit but defer fix to a separate targeted phase.
- **D-18:** Zero changes to the six existing channel names, schemas, or retention behavior.
- **D-19:** No build.rs changes (no new proto files vendored).
- **D-20:** No multimedia channels (Phase 26.5 covers that).
- **D-21:** No trace context fields (Phase 26.3 covers that — this phase runs first and leaves trace_id=None throughout).

### Claude's Discretion
- Exact audit format (Markdown table shape, grouping). Recommend one big table sorted by classification.
- Whether the mock provider returns `AgentTurn` synchronously or via an async stream — match whatever the real providers do.
- How to name the injected mock tool (`"hello_world"` is the decision; impl is free).
- Whether the new integration test file uses the gRPC AgentService or drives `AgentLoop` directly — pick whichever has the smaller test-scaffolding surface. Preference: gRPC path because it's closer to production and exercises the full emit chain.
- Test feature flag strategy — the existing pattern is `test-helpers` for ignored integration tests. Follow it.
- Fixture extension strategy — inline the mock-provider agent turn in `export_roundtrip.rs`, or spin up a mini helper module. Either is fine; simplest wins.
- Clippy lint posture: match the existing file's posture. The SC5 fixture already has `#![allow(clippy::too_many_lines, reason = "...")]`; reuse the same annotation style if needed.
</decisions>

## Existing Code Insights

### Reusable Assets
- **SessionEvent + EventEnvelope types** (`crates/roz-core/src/session/event.rs`): Comprehensive variant set already defined. No need to invent; just verify emit paths.
- **`emit_session_event` funnel** (`crates/roz-server/src/observability/ingest_cloud.rs:132`): Already dual-writes to `/roz/log` (summary) and `/roz/session/events` (proto). Adding coverage means ensuring variants reach this funnel, not building a new funnel.
- **`event_mapper::event_envelope_to_session_response`** (`crates/roz-server/src/grpc/event_mapper.rs`): Authoritative proto-encoding for every variant. If a variant isn't there, `encode_session_event_proto` returns `None` and the message silently drops — audit-critical.
- **`emit_session_event_for_tests`** (same file, ~line 184): Already exists as a test-harness entry. The new integration test may use this OR spin a real Session depending on whether we want to exercise the broadcast channel path.
- **`pg_container`** (`crates/roz-test/src/pg.rs`): Testcontainers Postgres scaffold — reuse verbatim.
- **SC5 fixture** (`crates/roz-server/tests/export_roundtrip.rs`): Already drives MCAP end-to-end with testcontainers; extend rather than clone.

### Established Patterns
- **Dual-emit pattern**: `emit_session_event` sends to BOTH `/roz/log` (human-readable summary via `format!("{envelope:?}")`) AND `/roz/session/events` (structured proto). Any new emit site should follow.
- **`pub(crate)` emit internals + `pub` test wrapper**: `emit_session_event` is `pub(crate)`; `emit_session_event_for_tests` is the public wrapper behind the `test-helpers` feature. Follow this pattern for any new test-only surfaces.
- **Ignored integration tests**: Tests requiring testcontainers are `#[ignore]` with the reason `"requires testcontainers Postgres + --features test-helpers"`. The new test follows suit.
- **Session-scoped MCAP**: Per-session `WriterActor`, channels registered up-front in `channels.rs`, events written as `WriteCommand::Event`. No changes needed here.

### Integration Points
- **Broadcast channel**: Session events flow from agent loop → session broadcast → subscribers including `emit_session_event`. If a variant is *emitted* in the agent loop but the subscriber path drops it, the audit catches it.
- **`log_level_for_event` as implicit coverage list**: The match arms in this function (lines 244–267) enumerate most variants by name for severity classification. Use this match as a cross-check during the audit — any variant NOT mentioned here is either defaulting to `Info` or was added after the match was last updated.
</code_context>

## Specific Ideas

- **Coverage matrix format:** big Markdown table, grouped by classification (agent-loop / tool-dispatch / safety / memory / controller / telemetry / approval / infrastructure), sorted alphabetically within each group. Columns: `Variant | Classification | Emit sites (file:line) | In production? | In fixture? | Status (✅/⚠️/❌) | Notes`.

- **Integration test timing:** prefer the gRPC AgentService path because it exercises the full broadcast → emit chain that a direct AgentLoop drive would bypass. But if the gRPC scaffolding balloons the test to 300 LOC, fall back to direct AgentLoop drive — it still catches the emit gaps even if it misses the gRPC forward path.
- **Feature-flag the mock provider behind `test-helpers`** — same pattern as `emit_session_event_for_tests`. Production builds never include it.
- **Do not touch the real Anthropic/OpenAI provider code.** The mock provider is additive.
</specifics>

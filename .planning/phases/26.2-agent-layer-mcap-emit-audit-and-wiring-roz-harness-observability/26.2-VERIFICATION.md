### O1: Edge-side AgentLoop parity drift (documented, not in deferred-items.md)

`crates/roz-server/src/grpc/agent.rs:1639` is the only production `AgentLoop` constructor that chains `.with_agent_event_hook(...)`. The following sites still default to `NoopAgentEventHook`, so agent-loop-originated events from worker and local-runtime sessions do not reach their session MCAP:

- `crates/roz-worker/src/main.rs:1210`
- `crates/roz-worker/src/session_relay.rs:441`
- `crates/roz-local/src/runtime.rs:406, :1230`
- `crates/roz-cli/src/commands/non_interactive.rs:183`
- `crates/roz-cli/src/tui/mod.rs:1178`

This is explicitly flagged in `26.2-REVIEW.md` MR-01 as a **scope gap**, not a correctness defect. The reviewer's recommendation (`REVIEW.md:78`) is to pick option (b) — "preserve phase scope" and file a follow-on (likely 26.2.1). As of this verification, `deferred-items.md` contains only the pre-existing fmt issue and the `roz-copper --all-features` build failure — it does NOT yet capture this edge-parity drift. **Recommend** adding a deferred-items entry or filing phase 26.2.1 before 26.3 kicks off, so downstream integration testers know MCAP contents currently differ between cloud and edge sessions.

This observation does not block SC3 / SC4 / SC6 because those tests all drive the cloud-gRPC hook install site, and the phase goal as written targets "agent-loop activity firing into `/roz/session/events`" without explicitly constraining edge-vs-cloud parity.

### O2: ReasoningTrace deferred to Phase 26.3 (per D-10 + REVIEWS.md M1)

`ReasoningTrace` is a D-10 WARN-not-BLOCK variant. Plan 04 explicitly defers the emit (core.rs:225, :399 carry `// DEFERRED: ReasoningTrace per Phase 26.3` comments) because `ReasoningTraceBuilder::new` requires a `turn_index` that `AgentLoop` does not currently carry. Both integration tests (`mcap_agent_session_live.rs:436-447`, `export_roundtrip.rs:582-588`) tolerate ReasoningTrace absence with a WARN-only print. This is the expected steady state for 26.2 and the deferred work is in scope for 26.3.

### O3: `MockProviderV1` is stateful (Plan 05 Rule-1 auto-fix)

CONTEXT.md D-06 originally described a single `CompletionResponse` with `ToolUse` + `EndTurn`. That shape would trigger the `agent_loop/core.rs:476` `EndTurn` short-circuit and make the entire tool-dispatch emit chain unreachable. Plan 05 auto-fixed this by making the mock stateful via `AtomicU32`: call #1 returns ToolUse (`stop_reason=ToolUse`), call #2+ returns `"Done."` (`stop_reason=EndTurn`). Both calls preserve canonical D-06 token usage (42/13), so `ModelCallCompleted` payload assertions remain deterministic. Foot-gun noted in `MockProviderV1` module docstring: reusing a single instance across multi-turn fixtures will keep returning `end_turn_response()` on call #3+.

### O4: Plan 03 `required-features` feature propagation (now fixed)

Plan 03's original must-haves claimed downstream tests "already have `roz-agent/test-helpers` in their feature graph." Plan 05 discovered and fixed this (`crates/roz-server/Cargo.toml:22` now propagates `test-helpers = ["roz-agent/test-helpers"]`). Phase 26.2 ends in the correct steady state; flagging only so future phases that add new roz-server integration tests know to include `required-features = ["test-helpers"]` on their `[[test]]` entries.

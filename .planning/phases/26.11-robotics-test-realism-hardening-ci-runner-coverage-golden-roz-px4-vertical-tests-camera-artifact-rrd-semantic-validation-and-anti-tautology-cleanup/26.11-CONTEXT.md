# Phase 26.11: Robotics Test Realism Hardening - Context

**Gathered:** 2026-04-26
**Status:** Ready for planning
**Source:** Conversation audit after Phase 26.10 verification

<domain>
## Phase Boundary

Harden the test suite so Roz's robotics and manipulator-class claims are proven by real, vertical, non-tautological tests instead of disconnected subsystem checks. This phase is not new product functionality. It is test infrastructure, CI routing, and acceptance coverage that proves the existing Phase 24-27 robotics stack works through production boundaries.

Terminology boundary: Roz is the agentic robotics harness, inspired in part by reference manipulator's agent-harness ideas. The reference manipulator and deterministic fake backend are Roz fixtures; they are not reference manipulator hardware support, reference manipulator protocol compatibility, or a claim that Roz is an reference manipulator distribution.

The phase must convert the audit findings into executable test work:

1. Make high-value tests actually run in CI/nightly with the required features.
2. Add a golden reference manipulator vertical path that proves user/agent command -> tool dispatch -> worker -> Copper -> fake manipulator backend -> telemetry/session evidence.
3. Add flight-command tool-routing coverage for the exact Phase 27 `flight_command` design.
4. Add or harden MAVLink/PX4 fixture and readiness replay tests without auto-mutating checked-in fixtures.
5. Add safety-policy vertical coverage from API policy push to physical command clamp/reject to archived session evidence.
6. Add camera archival coverage through worker NATS relay -> server ingest -> MCAP -> RRD.
7. Add artifact/bundle coverage for copper segments and manifest self-verification.
8. Upgrade RRD tests from file-header smoke checks to semantic entity/timeline validation where the current Rerun API allows it.
9. Remove or quarantine tautological/fragile test patterns such as `std::process::exit(0)` in normal MAVLink tests and robotics tests that pass without physical/simulated state change.

Out of scope:

- Rewriting product behavior outside what is needed to expose or test the production path.
- Adding real hardware requirements to default CI. hardware bench remains env-gated.
- Auto-updating MAVLink fixtures from nightly.
- Adding broad line-coverage thresholds. The project uses scenario coverage, not numeric coverage gates.
</domain>

<decisions>
## Locked Implementation Decisions

### Test Runner + CI Scope

- Add dedicated CI/nightly routing for tests that are currently present but skipped by default because of features or `#[ignore]`.
- The new routing must cover `roz-server --features test-helpers`, `roz-cli --features export-rrd`, and robotics/container tests that require NATS/Postgres/PX4/fake manipulator backends.
- Keep expensive SITL/hardware bench paths nightly or env-gated. Do not add PX4 or manipulator hardware bench to PR-gated default `cargo test --workspace`.
- Remove `std::process::exit(0)` from normal test binaries or isolate those tests behind a separate process harness so one test cannot terminate the rest of the binary.

### Golden manipulator-Inspired Manipulator Test

- Add a golden robotics acceptance test that proves Roz's reference manipulator path through production code, not just a fake isolated unit.
- The test must exercise: deterministic agent/tool invocation, worker task execution, physical tool registration, `EmbodimentRuntime`, `IoFactory`, `CopperHandle`, deterministic fake manipulator-class manipulator backend, telemetry/state observation, session event evidence, and failure/safety variants.
- The test must not assert actual upstream reference manipulator hardware behavior or upstream protocol compatibility.
- The test must prove state changed in the fake backend. A test that only asserts "command sent" or "no error" is not sufficient.
- Negative variants must be included: missing sink/backend returns a structured failure, and safety reject/clamp prevents unsafe physical output.

### Flight Command Tool Routing

- Use the Phase 27 decision: one `flight_command` tool with a command variant arg.
- Test the exact tool-routing path from agent-visible tool schema to dispatcher to `DiscreteCommandSink<FlightCommand>` using the concrete backend/extension pattern planned for Phase 27.
- Missing sink must produce a visible failed/unavailable tool result. Denied MAVLink ACK must surface as failed tool outcome.

### MAVLink / PX4 Fixtures

- PX4 fixture generation is verify-only. Nightly may record fresh `.tlog` files to tempdir, but fixture changes require a reviewed PR.
- Readiness assertions should use exact equality on the full `ReadinessState` at TAKEOFF and LAND checkpoints.
- ArduPilot variants remain deferred until an ArduPilot SITL container exists.
- Existing MAVLink tests that use process exit must be reworked before they are considered reliable CI evidence.

### Safety Vertical

- Safety tests must cover the full chain: server API policy write -> signed NATS push -> worker verify/apply -> policy cache/hot policy -> Copper/fake manipulator clamp or reject -> SafetyIntervention/SafetyViolation/session evidence -> metadata/index query.
- The test must assert physical unsafe values do not reach the actuator sink.
- Include anti-tautology inversion where feasible: permissive policy allows the same command that restrictive policy clamps/rejects.

### Camera / MCAP / RRD Vertical

- Camera archival must use the worker relay path: TestPatternSource or deterministic encoded frame -> worker `mcap_relay` -> NATS subject -> server `ingest_edge` -> `WriterActor` -> MCAP.
- Direct `WriteCommand::Event` camera injection can remain as a schema unit/integration test, but it is not the acceptance test for archival.
- Use real decodable or at least structurally valid H.264 Annex-B bytes from the existing encoder where possible, not hand-rolled arbitrary bytes, for vertical tests.
- RRD validation must check semantic output beyond the `RRF2` magic header where the `rerun` APIs permit: expected entity paths, timelines, and channel-derived objects.

### Artifacts / Bundle

- Test copper segment upload as one artifact row per segment.
- Test success cleanup and soft-fail retention behavior.
- Test actual CLI/server bundle export where feasible: MCAP rows from `roz_session_mcap_archives`, artifacts from `roz_session_artifacts`, `manifest.json` first, all digests verified.

### Test Quality Bar

- Tests must be vertical where the feature is vertical. Slices are allowed only when they are named as slices and paired with a vertical acceptance test.
- Every robotics acceptance test must prove a state transition, persisted evidence, or externally observable protocol message, not only an `Ok` return.
- Every newly added ignored test must be included in a documented nightly or manual run path.
- Avoid sleeps-only synchronization; prefer bounded timeouts on observable events.
- Prefer `crates/roz-test` helpers and existing `testcontainers` patterns over bespoke setup code.
</decisions>

<specifics>
## Specific Test Plan Seed

The planning output should break this phase into executable plans roughly along these seams:

1. CI/test-runner hardening and MAVLink process-exit cleanup.
2. reference manipulator golden vertical and flight-command tool-routing tests.
3. Safety vertical tests.
4. Camera archival and MCAP/RRD semantic tests.
5. Artifact/bundle vertical tests.
6. MAVLink fixture/readiness replay hardening.

The planner may split differently if dependencies demand it, but every item above must be covered by at least one plan objective and acceptance criteria.

Audit facts to preserve:

- The suite had about 145 integration-test files, about 569 test functions, and about 269 ignored tests at the time of audit.
- Several critical observability tests are feature-gated behind `test-helpers` or `export-rrd` and are not proven by default workspace tests.
- `mcap_camera_roundtrip` explicitly bypasses the NATS relay path.
- `mcap_to_rrd` currently validates mostly `RRF2` magic, not semantic recording contents.
- Some MAVLink tests call `std::process::exit(0)`, which can hide later tests in the same binary.
- `drone_wasm_velocity` can print a known issue and pass without actual drone movement; acceptance tests must not preserve that pattern.
</specifics>

<deferred>
## Deferred Ideas

- Optional manipulator hardware bench remains env-gated and should not block default CI. reference manipulator remains design inspiration only; upstream hardware/protocol compatibility is out of scope unless a later phase explicitly asks for it.
- ArduPilot fixture parity remains deferred until an ArduPilot SITL container exists.
- Full Rerun visual rendering in a viewer may remain a human gate if no stable read API exists, but semantic checks should still be added for what can be inspected programmatically.
- Numeric line/branch coverage reporting remains out of scope.
</deferred>

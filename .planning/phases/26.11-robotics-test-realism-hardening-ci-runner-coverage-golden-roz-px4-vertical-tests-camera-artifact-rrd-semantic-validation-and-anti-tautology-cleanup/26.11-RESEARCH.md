## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Test Runner + CI Scope

- Add dedicated CI/nightly routing for tests that are currently present but skipped by default because of features or `#[ignore]`.
- The new routing must cover `roz-server --features test-helpers`, `roz-cli --features export-rrd`, and robotics/container tests that require NATS/Postgres/PX4/fake manipulator backends.
- Keep expensive SITL/hardware bench paths nightly or env-gated. Do not add PX4 or manipulator hardware bench to PR-gated default `cargo test --workspace`.
- Remove `std::process::exit(0)` from normal test binaries or isolate those tests behind a separate process harness so one test cannot terminate the rest of the binary.

#### Golden manipulator-Inspired Manipulator Test

- Add a golden robotics acceptance test that proves Roz's reference manipulator path through production code, not just a fake isolated unit.
- The test must exercise: deterministic agent/tool invocation, worker task execution, physical tool registration, `EmbodimentRuntime`, `IoFactory`, `CopperHandle`, deterministic fake manipulator-class manipulator backend, telemetry/state observation, session event evidence, and failure/safety variants.
- The test must not assert upstream reference manipulator hardware behavior or upstream reference manipulator protocol compatibility.
- The test must prove state changed in the fake backend. A test that only asserts "command sent" or "no error" is not sufficient.
- Negative variants must be included: missing sink/backend returns a structured failure, and safety reject/clamp prevents unsafe physical output.

#### Flight Command Tool Routing

- Use the Phase 27 decision: one `flight_command` tool with a command variant arg.
- Test the exact tool-routing path from agent-visible tool schema to dispatcher to `DiscreteCommandSink<FlightCommand>` using the concrete backend/extension pattern planned for Phase 27.
- Missing sink must produce a visible failed/unavailable tool result. Denied MAVLink ACK must surface as failed tool outcome.

#### MAVLink / PX4 Fixtures

- PX4 fixture generation is verify-only. Nightly may record fresh `.tlog` files to tempdir, but fixture changes require a reviewed PR.
- Readiness assertions should use exact equality on the full `ReadinessState` at TAKEOFF and LAND checkpoints.
- ArduPilot variants remain deferred until an ArduPilot SITL container exists.
- Existing MAVLink tests that use process exit must be reworked before they are considered reliable CI evidence.

#### Safety Vertical

- Safety tests must cover the full chain: server API policy write -> signed NATS push -> worker verify/apply -> policy cache/hot policy -> Copper/fake manipulator clamp or reject -> SafetyIntervention/SafetyViolation/session evidence -> metadata/index query.
- The test must assert physical unsafe values do not reach the actuator sink.
- Include anti-tautology inversion where feasible: permissive policy allows the same command that restrictive policy clamps/rejects.

#### Camera / MCAP / RRD Vertical

- Camera archival must use the worker relay path: TestPatternSource or deterministic encoded frame -> worker `mcap_relay` -> NATS subject -> server `ingest_edge` -> `WriterActor` -> MCAP.
- Direct `WriteCommand::Event` camera injection can remain as a schema unit/integration test, but it is not the acceptance test for archival.
- Use real decodable or at least structurally valid H.264 Annex-B bytes from the existing encoder where possible, not hand-rolled arbitrary bytes, for vertical tests.
- RRD validation must check semantic output beyond the `RRF2` magic header where the `rerun` APIs permit: expected entity paths, timelines, and channel-derived objects.

#### Artifacts / Bundle

- Test copper segment upload as one artifact row per segment.
- Test success cleanup and soft-fail retention behavior.
- Test actual CLI/server bundle export where feasible: MCAP rows from `roz_session_mcap_archives`, artifacts from `roz_session_artifacts`, `manifest.json` first, all digests verified.

#### Test Quality Bar

- Tests must be vertical where the feature is vertical. Slices are allowed only when they are named as slices and paired with a vertical acceptance test.
- Every robotics acceptance test must prove a state transition, persisted evidence, or externally observable protocol message, not only an `Ok` return.
- Every newly added ignored test must be included in a documented nightly or manual run path.
- Avoid sleeps-only synchronization; prefer bounded timeouts on observable events.
- Prefer `crates/roz-test` helpers and existing `testcontainers` patterns over bespoke setup code.

## Summary

Phase 26.11 is a test realism phase, not a broad product feature phase. The planning work should focus on making existing high-value feature-gated or ignored tests run in PR/nightly, then adding a small set of vertical acceptance tests that prove production boundaries: agent/tool dispatch, worker task execution, Copper/fake manipulator IO, signed NATS, server ingest, MCAP archives, RRD export, artifacts, and metadata queries.

Terminology boundary: Roz is the agentic robotics harness, inspired in part by reference manipulator's agent-harness ideas. The reference manipulator and deterministic fake backend are Roz fixtures; they are not shorthand for reference manipulator hardware behavior or protocol compatibility.

The strongest current gap is not lack of code everywhere; it is that many tests either do not run under current CI profiles, bypass the production seam they claim to prove, or assert file existence/header success instead of semantic behavior. The research found explicit examples: `mcap_camera_roundtrip` drives `WriterActor` directly instead of worker NATS relay, `mcap_to_rrd` mostly checks `RRF2` bytes, MAVLink tests call `std::process::exit(0)`, and `drone_wasm_velocity` can pass after printing a known issue without movement.

**Primary recommendation:** Plan this phase as six low-conflict waves: CI/nightly routing and process-exit cleanup first, then reference manipulator/flight-command test seams, safety vertical, camera/RRD semantics, artifact/bundle vertical, and MAVLink/PX4 readiness fixtures.

### Pattern 1: Vertical Acceptance Means Production Boundary Coverage

**What:** A vertical test must drive the same production seam that the feature uses. It can be deterministic and local, but it must observe state change, persisted evidence, or a protocol message outside the immediate function under test.

**When to use:** Golden reference manipulator tests, safety, camera archival, artifact/bundle, and flight-command routing.

**Example acceptance shape:**

```rust
// Source: local production pattern in crates/roz-worker/src/main.rs and io_backends.rs.
// The test should create a real OodaReAct invocation with EmbodimentRuntime,
// allow execute_task to select factory_for("manipulator"), then observe both:
// 1. fake manipulator-class manipulator state changed through CopperHandle IO
// 2. session/runtime evidence was emitted or archived.
assert!(observed_joint_position_delta.abs() > f64::EPSILON);
assert!(session_events.iter().any(|event| event.kind == "ToolCallFinished"));
```

### Pattern 2: Keep Slices, But Name Them as Slices

**What:** Existing tests such as `manipulator_dispatch_path`, `fake_manipulator_tick_loop`, and `telemetry_proto_reaches_mcap` are useful focused slices. They should remain, but not be the only evidence for vertical claims.

**When to use:** Backend determinism, pure protocol mapping, schema compatibility, or test diagnostics.

**Planning rule:** Every slice added in this phase should either be a prerequisite for one vertical test in the same plan or explicitly documented as non-acceptance coverage.

### Pattern 3: CI Routing Mirrors Feature Gates

**What:** If a test target has `required-features`, the workflow must run that crate with those features. If a test is `#[ignore]`, it must appear in a named `nextest` profile or be documented as manual/env-gated.

**Current gaps found:**

- `roz-server --features test-helpers` targets exist but PR CI does not run them as a feature job.
- `roz-cli --features export-rrd` exists but PR CI does not run it.
- `roz-worker --features test-fixtures` is not routed in PR/nightly despite fake manipulator-class manipulator wiring being feature-gated there.
- `roz-mavlink --features test-helpers` is only reached indirectly by some worker dev-deps, not as explicit MAVLink evidence.
- `.config/nextest.toml` does not allowlist several Phase 26.x observability tests now listed as canonical references.

### Pattern 4: Use Concrete Extensions for Physical Sinks

**What:** `roz_agent::dispatch::Extensions` stores concrete `TypeId` keys. It does not support retrieving a trait object inserted under another type. Phase 27's flight-command test should insert and retrieve a concrete `Arc<MavlinkBackend>` or a dedicated concrete wrapper.

**When to use:** `flight_command` tool-routing tests and missing-sink negative tests.

**Example:**

```rust
// Source: crates/roz-agent/src/dispatch/mod.rs Extensions TypeId map.
let mut extensions = roz_agent::dispatch::Extensions::new();
extensions.insert(Arc::clone(&mavlink_backend));

let backend = context
    .extensions
    .get::<Arc<roz_mavlink::MavlinkBackend>>()
    .ok_or_else(|| anyhow::anyhow!("flight_command unavailable: MAVLink sink missing"))?;
```

### Anti-Patterns to Avoid

- **Acceptance through direct backend calls:** Calling `fake_manipulator_pair` or `ActuatorSink::send` alone proves backend math, not worker/reference manipulator production routing.
- **Header-only RRD validation:** `RRF2` proves an RRD file was opened, not that expected entities or timelines exist.
- **Ignored test without a runner:** `#[ignore]` plus no `nextest` allowlist is equivalent to manual-only coverage and cannot support acceptance.
- **Skip-on-missing-dependency as pass:** A test that prints a known issue and succeeds must not count as robotics evidence.
- **Process-wide exit in test binaries:** `std::process::exit(0)` prevents later tests in the same binary from running and hides failures.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| Postgres/NATS/Restate setup | Custom Docker shell scripts and sleeps | `crates/roz-test` helpers and existing `testcontainers` patterns | Existing helpers encode ports, readiness, cleanup, and CI behavior. |
| MCAP parsing | Byte scanning for topics/schemas | `mcap::MessageStream` and summaries | MCAP has schema/channel/message structure; parsing bytes misses semantic mismatches. |
| RRD validation | `RRF2` magic checks only | `re_log_encoding::DecoderApp` plus `re_sorbet::ChunkBatch` | Entity paths, timelines, and component columns are the actual contract. |
| reference manipulator simulation | New fake manipulator implementation | `roz_copper::fake_manipulator::fake_manipulator_pair` through `FakeManipulatorFactory` | The existing fake is deterministic and already wired behind `roz-worker/test-fixtures`; this is not upstream reference manipulator simulation. |
| Flight-command transport | New MAVLink encoder in tests | `roz_mavlink::FlightCommandDispatcher` and `MavlinkBackend` | Existing code maps `FlightCommand` to `MAV_CMD_*` and ACK semantics. |
| Signed NATS in acceptance tests | Unsigned direct publish when testing signed paths | `WorkerSigningContext`, signed helpers, and server ingest paths | Unsigned shortcuts bypass production security behavior. |
| Camera H.264 bytes | Random hand-rolled byte arrays | `TestPatternSource` plus existing `SwEncoder`, or known valid Annex-B fixtures | Arbitrary bytes prove only storage, not realistic camera archival. |
| Synchronization | Fixed sleeps | Bounded timeouts on events, DB rows, channels, or file finalization | Sleeps create flakes and do not prove the observable condition occurred. |

**Key insight:** The hard parts here are not algorithms; they are proving that independent production components meet at the correct boundary. Reusing the real components is more valuable than adding broad helper abstractions.

## Common Pitfalls

### Pitfall 1: Feature-Gated Tests Orphaned From CI

**What goes wrong:** A test target with `required-features = ["test-helpers"]` or `export-rrd` compiles only when manually run with features, so default workspace CI never proves it.

**Why it happens:** `cargo test --workspace` skips required-feature test targets when features are absent.

**How to avoid:** Add explicit CI jobs for `cargo test -p roz-server --features test-helpers`, `cargo test -p roz-cli --features export-rrd`, and `cargo test -p roz-worker --features test-fixtures` where tests are non-ignored and PR-safe. Add nightly `nextest` jobs for ignored/container-heavy cases.

**Warning signs:** A `[[test]] required-features` block exists but no workflow command mentions the crate/feature pair.

### Pitfall 2: Direct WriterActor Camera Injection

**What goes wrong:** Camera MCAP tests pass while the worker relay, signed NATS subject, server subscription, and first-sighting `RegisterCamera` logic are broken.

**Why it happens:** `crates/roz-server/tests/mcap_camera_roundtrip.rs` currently drives `WriteCommand::RegisterCamera` and `WriteCommand::Event` directly.

**How to avoid:** Keep that test as schema coverage, but add a vertical test using `spawn_mcap_relay` -> NATS `camera.{worker}.{session}.{camera}` -> `spawn_edge_ingestors`/`ingest_edge` -> `WriterActor` -> MCAP.

**Warning signs:** The test imports `WriteCommand` but never starts NATS or worker relay code.

### Pitfall 3: RRD Header Checks

**What goes wrong:** RRD tests pass when entity paths, timelines, or channel-derived objects are absent or misnamed.

**Why it happens:** Existing `mcap_to_rrd` and exporter module tests check `RRF2` magic and non-empty output.

**How to avoid:** Decode `.rrd` in tests and assert expected entity paths, component columns, and timelines.

**Warning signs:** Assertions only inspect `&bytes[..4] == b"RRF2"`.

### Pitfall 4: MAVLink Tests Exiting the Process

**What goes wrong:** One test terminates the whole test process and masks later tests in the same binary.

**Why it happens:** Several MAVLink tests use `std::process::exit(0)` as a teardown shortcut.

**How to avoid:** Replace with drop-based cleanup, explicit task cancellation, or isolate the scenario behind a subprocess harness where the parent asserts exit status and output.

**Warning signs:** `std::process::exit(0)` in `crates/roz-mavlink/tests/*` or normal worker tests.

### Pitfall 5: Manipulator Test That Does Not Prove Movement

**What goes wrong:** A robotics test passes after a command path returns `Ok`, but the fake backend state never changes.

**Why it happens:** Tests stop at command dispatch rather than reading state from the backend/Copper state.

**How to avoid:** Assert joint position or velocity changes via fake manipulator-class sensor state, `CopperHandle` state snapshots, and persisted session event evidence.

**Warning signs:** Assertions only check "command sent", "status ok", or "no error".

### Pitfall 6: `Extensions` Trait-Object Mismatch

**What goes wrong:** `flight_command` tool tests insert one type and retrieve another, causing missing-sink failures unrelated to MAVLink logic.

**Why it happens:** `Extensions` is keyed by concrete `TypeId`.

**How to avoid:** Store `Arc<MavlinkBackend>` or a concrete `FlightCommandSinkHandle` wrapper, then retrieve that same concrete type. If multiple implementations are needed later, add a wrapper enum/newtype rather than storing `dyn DiscreteCommandSink`.

### Pitfall 7: Fixture Mutation in Nightly

**What goes wrong:** Nightly PX4 runs rewrite checked-in `.tlog` fixtures and create unreviewed drift.

**Why it happens:** Recording and verification are coupled.

**How to avoid:** Nightly may record fresh logs only to tempdir; checked-in fixture updates require a reviewed PR. The fixture replay test should be verify-only.

## Code Examples

Verified patterns from local production sources:

### Semantic RRD Inspection

```rust
use std::{collections::BTreeSet, fs::File, io::BufReader, path::Path};

use re_log_encoding::DecoderApp;
use re_log_types::LogMsg;
use re_sorbet::ChunkBatch;

fn rrd_entities_timelines_components(
    path: &Path,
) -> anyhow::Result<(BTreeSet<String>, BTreeSet<String>, BTreeSet<String>)> {
    let file = BufReader::new(File::open(path)?);
    let mut entities = BTreeSet::new();
    let mut timelines = BTreeSet::new();
    let mut components = BTreeSet::new();

    for msg in DecoderApp::decode_lazy(file) {
        if let LogMsg::ArrowMsg(_, arrow_msg) = msg? {
            let batch = ChunkBatch::try_from(&arrow_msg.batch)?;
            entities.insert(batch.entity_path().to_string());
            for (descriptor, _) in batch.index_columns() {
                timelines.insert(descriptor.timeline_name().to_string());
            }
            for (descriptor, _) in batch.component_columns() {
                components.insert(descriptor.component_path().to_string());
            }
        }
    }

    Ok((entities, timelines, components))
}
```

Planner note: method names above are verified against local `re_log_encoding`, `re_log_types`, and `re_sorbet` 0.31.3 sources at research time, but the exact direct dev-dep compile should be a Wave 0 task because Rerun internals are version-sensitive.

### Bounded Observable Wait

```rust
async fn wait_until<T, Fut>(
    timeout: std::time::Duration,
    mut probe: impl FnMut() -> Fut,
) -> anyhow::Result<T>
where
    Fut: std::future::Future<Output = anyhow::Result<Option<T>>>,
{
    let deadline = tokio::time::Instant::now() + timeout;
    loop {
        if let Some(value) = probe().await? {
            return Ok(value);
        }
        if tokio::time::Instant::now() >= deadline {
            anyhow::bail!("condition was not observed before timeout");
        }
        tokio::time::sleep(std::time::Duration::from_millis(25)).await;
    }
}
```

Use this shape for DB row visibility, NATS subscription messages, MCAP finalization, and fake-backend state changes. Do not use unbounded sleeps.

- The test must use a concrete extension type (`Arc<MavlinkBackend>` or wrapper) because `Extensions` is concrete `TypeId` storage.

### MAVLink / PX4 Fixtures

- `crates/roz-mavlink/src/readiness.rs` exposes `ReadinessBuilder::snapshot()` returning full `ReadinessState`; tests should assert exact full equality at TAKEOFF and LAND checkpoints.
- Process-exit teardown exists in `crates/roz-mavlink/tests/log_fanout.rs`, `send_log_erase.rs`, `autopilot_kind.rs`, `qgc_coexistence.rs`, and `crates/roz-worker/tests/mavlink_backend_null_key.rs`.
- `qgc_coexistence.rs` already comments that `std::process::exit(0)` terminates the whole test process; this is not acceptable CI evidence.
- PX4 fixture generation should write new `.tlog` files only to tempdir during nightly/manual verification.

### Safety Vertical

- `crates/roz-server/tests/phase24_policy_crud_live.rs` already demonstrates server API -> DB -> signed NATS -> worker verify/apply -> hot policy/pre-dispatch rejection.
- `crates/roz-copper/tests/phase24_policy_tick_clamp.rs` is a good anti-tautology clamp pattern because permissive vs restrictive behavior would invert the outcome, and it asserts captured command frames.
- Missing acceptance coverage is the final physical segment: restrictive policy must prove unsafe values do not reach the fake manipulator actuator sink and that SafetyViolation/SafetyIntervention evidence is archived and queryable.

### Camera / MCAP / RRD

- Worker production path: `crates/roz-worker/src/camera/mcap_relay.rs` subscribes to camera frames, encodes `CompressedVideo`, signs/publishes to NATS camera subjects, and enforces frame-size/keyframe rules.
- Server production path: `crates/roz-server/src/observability/ingest_edge.rs` subscribes to `camera.{worker}.{session}.*`, verifies signatures, sends first-sighting `RegisterCamera`, then writes camera `Event`s to `WriterActor`.
- Current camera test `crates/roz-server/tests/mcap_camera_roundtrip.rs` is schema/archive coverage only because it directly sends `WriteCommand`.
- `crates/roz-worker/tests/camera_integration.rs` already has real TestPatternSource -> openh264 encoder -> StreamHub slices. Reuse that source/encoder for realistic vertical camera bytes.
- Current RRD tests in `crates/roz-cli/tests/mcap_to_rrd.rs` and exporter modules mostly assert `RRF2` magic. Add semantic decode assertions for entity paths/timelines/components.

### Artifacts / Bundle

- `crates/roz-worker/src/copper_archive.rs::finalize_copper_archive` walks session `*.copper` files, uploads one artifact per segment, soft-fails per file, and conditionally removes the local session dir after complete success.
- `crates/roz-server/tests/artifact_copper_roundtrip.rs` proves ArtifactService upload/download digest behavior, including tampered-digest rejection. It does not prove worker finalizer segmentation or CLI bundle export.
- `crates/roz-cli/src/commands/session.rs::export_bundle` fetches MCAP rows and artifact rows, writes `manifest.json` first, assembles tar, and calls `verify_bundle_integrity` before writing.
- Existing bundle unit tests assert manifest-first and tamper detection; add a live server/CLI-backed bundle test where feasible.

## Open Questions

1. **Should 26.11 implement a minimal `flight_command` tool seam or only test the Phase 27 seam once it exists?**
   - What we know: `MavlinkBackend` implements `DiscreteCommandSink<FlightCommand>`, but worker/agent tool registration is explicitly deferred to Phase 27.
   - What's unclear: Whether this test-hardening phase is allowed to add the thin tool-routing code needed for green tests.
   - Recommendation: Plan a narrow test-first seam aligned with Phase 27's one-tool/variant-arg design. Keep product behavior minimal and avoid broader MAVLink IoFactory work.

2. **How deep should the live CLI bundle export test go?**
   - What we know: server ArtifactService and CLI bundle integrity internals are separately tested; `export_bundle` is private and config-driven.
   - What's unclear: Whether to drive the actual `roz` binary against a live test server or expose a narrower test helper.
   - Recommendation: Prefer a live server-backed CLI/integration test if setup stays bounded. Otherwise extract a crate-private/test-only helper that still fetches real MCAP/artifact rows through service clients.

3. **Will Rerun internals stay stable enough for direct semantic tests?**
   - What we know: workspace pins `rerun = "=0.31.3"` and `Cargo.lock` includes matching `re_log_encoding`, `re_log_types`, and `re_sorbet` 0.31.3.
   - What's unclear: Whether future Substrate/Rerun lockstep changes will require test helper rewrites.
   - Recommendation: Add direct exact-pinned dev-deps and keep the semantic helper small. Treat compile failures as intentional lockstep drift signals.

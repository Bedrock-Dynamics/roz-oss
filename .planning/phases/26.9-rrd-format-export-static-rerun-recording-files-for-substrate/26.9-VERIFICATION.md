### Documented Implementation Corrections (verified at source)

These are documented inline in source code and prevent a reviewer comparing against CONTEXT/RESEARCH text from flagging them as deviations:

1. **B-2 method-name correction** (`recording.rs:9-19` module docs) — CONTEXT D-09 prescribes `RecordingStream::set_time_seconds(name, f64)`; the method does not exist in `rerun = 0.31.3`. The verified-correct API is `set_timestamp_secs_since_epoch(timeline, secs)`. Intent (f64-second precision dual timelines) is preserved.
2. **AnyValues demotion option A** (`text_logs.rs:7-19` module docs) — CONTEXT D-10 row 4 prescribes `TextLog + AnyValues` for tool-call params; `rerun = 0.31.3`'s `AnyValues` lacks typed-value setters per RESEARCH §Topic 5. Demoted to JSON-stringified params in TextLog body. The verification prompt's SC3 wording acknowledges this demotion.
3. **`VideoSample` → `VideoStream::with_sample` correction** (`camera.rs:23-45` module docs) — CONTEXT D-11 prescribes `rec.log(path, &VideoSample::from(data))`; `rerun::components::VideoSample` does not impl `AsComponents` in `re_sdk_types-0.31.3`. The verified-correct pattern is `VideoStream::new(codec).with_sample(data)` first sighting + `VideoStream::update_fields().with_sample(data)` per frame (canonical Rerun 0.31.3 per-frame video logging pattern). Same on-disk `rerun.components.VideoSample` payload either way.
4. **EncodedImage typo correction (ROADMAP SC3 → CONTEXT D-10)** — ROADMAP SC3 row 7 says `rr.EncodedImage` for camera channels; CONTEXT D-10 documents this is wrong (EncodedImage = JPEG/PNG still images, not H.264 video). The verification prompt's SC3 row 7 already reflects the corrected mapping `rr.EncodedImage / rr.Points3D / rr.Boxes3D` qualified by "(when present)" — i.e., the corrected H.264 path ships, and Points3D/Boxes3D producers don't exist yet (CONTEXT `<deferred>`).

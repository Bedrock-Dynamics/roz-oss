### Acceptance Criterion vs Reality Mismatch (no fix needed)

The corrected implementation does NOT call `VideoSample::from` — it uses `VideoStream::new(codec).with_sample(data)` and `VideoStream::update_fields().with_sample(data)` because `rerun::components::VideoSample` does not implement `AsComponents` in `re_sdk_types-0.31.3`. - `cv.data` (`Vec<u8>` Annex B NALU buffer for one frame) is passed unchanged to `with_sample(...)`.

- The recorded `.rrd` therefore carries the same per-frame `rerun.components.VideoSample` payload bytes whether logged via the Plan-prescribed pattern or the API-corrected pattern.

## API Correction Applied

CONTEXT D-11 / RESEARCH §Topic 3 / RESEARCH Code Example 4 all prescribe:

```rust
use rerun::components::{VideoCodec, VideoSample};
let sample = VideoSample::from(message.data.to_vec());
rec.log(format!("/world/cameras/{name}"), &sample)?;
```

Both RESEARCH and CONTEXT claim "`Component` impls `AsComponents` so `&sample` works in `rec.log()`." Empirically (verified against `re_sdk_types-0.31.3/src/components/video_sample.rs` on 2026-04-24), `VideoSample` is a `WrapperComponent` (transparent newtype around `Blob`) and does NOT have an `AsComponents` blanket impl in 0.31.3. Attempting to log it directly produces:

```
error[E0277]: the trait bound `VideoSample: AsComponents` is not satisfied
```

The verified-correct Rerun 0.31.3 API for per-frame video logging — documented inline on `VideoStream` itself (re_sdk_types-0.31.3/src/archetypes/video_stream.rs lines 27-30) — is:

> "All components except `sample` are typically logged statically once per entity. `sample` is then logged repeatedly for each frame on the timeline."

with the per-frame builder pattern using `with_sample(data: impl Into<VideoSample>)`. The corrected implementation:

```rust
if state.seen_camera_videostream_logged.insert(camera_name.to_string()) {
    // First sighting: codec + first sample together.
    rec.log(entity_path, &VideoStream::new(VideoCodec::H264).with_sample(cv.data))?;
} else {
    // Subsequent frames: only the sample, codec stays as previously-logged static.
    rec.log(entity_path, &VideoStream::update_fields().with_sample(cv.data))?;
}
```

Both branches log a `VideoStream` archetype (which DOES impl `AsComponents`), but with different field sets — the `update_fields()` branch leaves the codec field unset so the static codec is not re-logged per frame. This is the canonical Rerun 0.31.3 pattern and produces the same on-disk `.rrd` payload (`rerun.components.VideoSample` per frame) that the original D-11 prescription intended.

## Dedupe Design (State Mutation Order)

The `malformed_payload_errors_gracefully` test asserts that a decode failure leaves `state.seen_camera_videostream_logged` empty. To satisfy this, the implementation orders operations as:

1. `CompressedVideo::decode(msg.data.as_ref()).context("decode foxglove.CompressedVideo")?` — fail-fast on bad payload, before any state read or mutation.
2. `state.seen_camera_videostream_logged.insert(camera_name.to_string())` — `HashSet::insert` returns `true` iff the camera was newly inserted; this drives the once-per-camera codec logging branch.
3. `rec.log(...)` — only after both decode and insert succeed.

If decode fails at step 1, control returns to caller via `?` with state untouched. If insert returns `true` (first sighting), the impl logs `VideoStream::new(VideoCodec::H264).with_sample(data)` once. If insert returns `false` (subsequent frame), the impl logs `VideoStream::update_fields().with_sample(data)`. Either way, the per-frame Annex B `cv.data` rides through `with_sample(data: impl Into<VideoSample>)` byte-identically.

Multi-camera dedupe is automatic from `HashSet<String>` semantics: front and rear cameras hash to different bucket entries, so each camera's `VideoStream`-with-codec is logged exactly once independently. Verified by `multi_camera_dedupe_tracks_each_independently`.

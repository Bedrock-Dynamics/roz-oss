## AnyValues Demotion Decision

CONTEXT D-10 row 4 specifies `TextLog + AnyValues` for tool-call parameters. RESEARCH ┬žTopic 5 confirmed that `rerun = 0.31.3`'s `AnyValues` API has no typed-value convenience methods (no `.with_string(name, value)` / `.with_int(name, value)`). Three options were evaluated:

| Option | Approach | Status |
|--------|----------|--------|
| A | JSON-stringify the `parameters` Struct into the TextLog body | **shipped** ÔÇö simple, visually scannable in Rerun viewer's log panel |
| B | Walk the Struct and emit one `with_component` per top-level key | deferred ÔÇö adds API surface without UX win for current substrate consumer |
| C | Build Arrow arrays directly via `with_component_from_data` | deferred ÔÇö heaviest implementation, blocked on a clearer UX requirement |

Option A is documented in the module doc-comment (lines 7-19). Future phases can revisit when substrate's UX team has a column-typed rendering requirement.

### Plan-Acceptance Mismatch (no fix needed)

PLAN.md's prescribed code wraps `TextLogLevel::*` constants with `.to_string()` (e.g., `with_level(level.to_string())`). The actual `with_level` signature is `pub fn with_level(self, level: impl Into<TextLogLevel>)`, and `TextLogLevel::ERROR` is `&'static str`; the blanket `impl<T: Into<Utf8>> From<T> for TextLogLevel` lets `&'static str` flow through directly. Dropping the `.to_string()` is cleaner and still satisfies the `grep -c 'TextLogLevel::*'` acceptance criterion (10 matches, Ôëą5 required). PLAN line 512 explicitly authorised this: "If rerun 0.31.3's actual signature accepts `&'static str` directly, omit the `.to_string()`."

## Threat Flags

None ÔÇö both new threats enumerated in the plan's `<threat_model>` (T-26.9-16 and T-26.9-17) are honored:

- **T-26.9-16** (mitigate): JSON injection via tool-parameter values ÔÇö `prost_value_to_json_string` calls `serde_json::to_string` on every string value (handles escaping for embedded quotes/backslashes/control characters). The fallback `format!("\"{s}\"")` only fires if `serde_json::to_string` fails, which would require a non-finite floating-point value masquerading as a string (impossible at the type boundary).

NaN/Infinity in `Kind::NumberValue` are emitted as `null` rather than the JSON-illegal `NaN` / `Infinity` literals ÔÇö small Rule-2 hardening beyond the original plan.

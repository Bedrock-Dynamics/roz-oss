# Phase 26.9: Code Review Report

**Reviewed:** 2026-04-24
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

Phase 26.9 introduces a pure CLI format converter (`roz mcap to-rrd`) that streams MCAP messages, classifies channels by `(topic, schema_name)`, and emits Rerun archetypes to a `.rrd` file via `RecordingStream::save()`. The implementation is well-structured, feature-gated cleanly behind `export-rrd`, has good per-channel test coverage, and follows the project's `anyhow`/`tracing` error-handling conventions. No critical security or panic risks were found.

Three warnings concern correctness/UX: a glob with no matches succeeds silently, the camera-name classifier accepts nested paths (`/roz/camera/front/sub`), and the `cv.format` field is decoded but never validated against the hardcoded `H264` codec. Info items cover lower-impact concerns: parent-frame loss in `/tf` (intentional per CONTEXT D-10), hand-rolled JSON encoding, fragile `Debug`-string parsing for fallback enum names, partial `.rrd` files left on disk after bulk-mode failures, and a few minor naming/sanitisation gaps.

## Warnings

### WR-01: `export_bulk` succeeds silently when glob matches zero files

**File:** `crates/roz-cli/src/commands/mcap/export.rs:202-269`
**Issue:** When `glob::glob(pattern)` returns an empty iterator (typo'd path, wrong cwd, no matching files), `total = 0`, `fail = 0`, the loop body never executes, the summary prints `"0/0 succeeded, 0 failed"`, and the function returns `Ok(())`. A user invoking `roz mcap to-rrd --bulk 'sessions/*.mcap' --output-dir out` against the wrong directory gets exit 0 with no `.rrd` files produced — silent no-op. Bulk mode's continue-on-error contract (D-05) implies "at least one file failed → nonzero exit" but does not address "zero matches".
**Fix:**
```rust
let entries: Vec<_> = glob::glob(pattern)
    .with_context(|| format!("expand glob: {pattern}"))?
    .collect();

if entries.is_empty() {
    anyhow::bail!("glob matched zero files: {pattern}");
}

let total = entries.len();
// ...rest unchanged
```

### WR-02: `classify_channel` accepts nested camera paths and produces unintended entity hierarchies

**File:** `crates/roz-cli/src/commands/mcap/export.rs:93-98`
**Issue:** `topic.strip_prefix("/roz/camera/")` returns `Some("front/sub")` for `/roz/camera/front/sub`. The `!name.is_empty()` guard does not catch slashes inside `name`. The resulting `ChannelKind::Camera("front/sub")` flows into `format!("/world/cameras/{name}")` in `camera::emit_camera`, producing `/world/cameras/front/sub` — an unintended Rerun entity sub-hierarchy that may collide with sibling entities and complicate substrate-side rendering. There is also an existing test `classify_empty_camera_name_is_unknown` covering the empty case but no nested-path test.
**Fix:**
```rust
if let Some(name) = topic.strip_prefix(TOPIC_CAMERA_PREFIX)
    && schema_name == SCHEMA_COMPRESSED_VIDEO
    && !name.is_empty()
    && !name.contains('/')
{
    return ChannelKind::Camera(name.to_string());
}
```
Add a corresponding test:
```rust
#[test]
fn classify_nested_camera_path_is_unknown() {
    assert_eq!(
        classify_channel("/roz/camera/front/sub", "foxglove.CompressedVideo"),
        ChannelKind::Unknown
    );
}
```

### WR-03: `cv.format` is decoded but never validated against hardcoded `H264`

**File:** `crates/roz-cli/src/commands/mcap/camera.rs:97-117`
**Issue:** The decoded `CompressedVideo.format` field is bound (via the prost struct on line 67 with `format: String`) but the emit path hardcodes `VideoCodec::H264` regardless of what the producer wrote in `format` (e.g. `"h264"`, `"h265"`, `"av1"`). CONTEXT D-11 explicitly scopes 26.9 to H.264 pass-through, so a non-H264 producer is out of contract — but if the worker ever writes a non-H264 format, the resulting `.rrd` would carry incorrect codec metadata that produces a corrupt-looking video stream in Rerun (silent data corruption, not a panic). At minimum, validate `format` and warn-or-error on mismatch so the failure mode is observable rather than silent.
**Fix:**
```rust
let cv = CompressedVideo::decode(msg.data.as_ref()).context("decode foxglove.CompressedVideo")?;

// Phase 26.9 D-11 scopes us to H.264 only. Refuse to emit other codecs
// rather than silently mis-tagging them with VideoCodec::H264.
if !cv.format.is_empty() && !cv.format.eq_ignore_ascii_case("h264") {
    anyhow::bail!(
        "unsupported camera codec for {camera_name}: {:?} (Phase 26.9 supports h264 only)",
        cv.format
    );
}
```
Alternative (warn-once + skip) if continue-on-error is preferred for forward-compat, mirroring the unknown-channel pattern.

## Info

### IN-01: `/tf` emit drops `parent_frame_id` and flattens kinematic chains

**File:** `crates/roz-cli/src/commands/mcap/transforms.rs:37-62`
**Issue:** `emit_tf` logs `Transform3D` at `/world/{child_frame_id}` without consulting `tf.parent_frame_id`. Rerun composes transforms via entity-tree hierarchy, so a kinematic chain (e.g., `parent="base_link"`, `child="tool0"`) is rendered as if `tool0` were a direct child of `world`, losing the parent relationship. CONTEXT.md D-10 row 1 explicitly mandates this exact path mapping, so this is a deliberate design decision — but worth flagging here because future producers writing chained transforms will silently render incorrectly. Consider scheduling a follow-up phase for proper hierarchy reconstruction.
**Fix:** Documented design choice; no change needed in 26.9. If hierarchy support is added later, the path becomes `format!("/world/{}/{}", tf.parent_frame_id, tf.child_frame_id)` (with sanitisation; see IN-07).

### IN-02: Hand-rolled JSON encoder reinvents `serde_json`

**File:** `crates/roz-cli/src/commands/mcap/text_logs.rs:271-302`
**Issue:** `prost_struct_to_json_string` and `prost_value_to_json_string` manually emit JSON syntax (`{`, `}`, `[`, `]`, `,`, `:`, `null`, `true`, `false`) and only use `serde_json::to_string` for string escaping. `serde_json::Value` already covers this domain trivially: convert `prost_types::Value` → `serde_json::Value` once, then call `serde_json::to_string`. The current implementation has subtle gaps (numeric `f64` formatting via `n.to_string()` doesn't match JSON's preferred shortest-round-trip representation; nested struct order is implementation-defined per the comment).
**Fix:**
```rust
fn prost_struct_to_json_string(s: &prost_types::Struct) -> String {
    let v: serde_json::Value = serde_json::Value::Object(
        s.fields.iter().map(|(k, v)| (k.clone(), prost_value_to_json(v))).collect()
    );
    serde_json::to_string(&v).unwrap_or_else(|_| "{}".to_string())
}

fn prost_value_to_json(v: &prost_types::Value) -> serde_json::Value {
    use prost_types::value::Kind;
    match &v.kind {
        None | Some(Kind::NullValue(_)) => serde_json::Value::Null,
        Some(Kind::NumberValue(n)) if n.is_finite() => serde_json::json!(n),
        Some(Kind::NumberValue(_)) => serde_json::Value::Null,
        Some(Kind::StringValue(s)) => serde_json::Value::String(s.clone()),
        Some(Kind::BoolValue(b)) => serde_json::Value::Bool(*b),
        Some(Kind::StructValue(inner)) => serde_json::from_str(&prost_struct_to_json_string(inner))
            .unwrap_or(serde_json::Value::Null),
        Some(Kind::ListValue(lv)) => {
            serde_json::Value::Array(lv.values.iter().map(prost_value_to_json).collect())
        }
    }
}
```

### IN-03: `session_event_variant_and_summary` fallback parses `Debug` output for entity-path segments

**File:** `crates/roz-cli/src/commands/mcap/text_logs.rs:155-164`
**Issue:** The `other =>` arm extracts a variant name via `format!("{other:?}")` followed by `split_whitespace().next()`. The Rust `Debug` impl format for enum variants is convention but not guaranteed stable across rustc versions or prost-build releases. If prost ever emits a different `Debug` shape (e.g., wraps in a tuple struct that prints with leading whitespace), the resulting Rerun entity paths will silently change — breaking any downstream substrate-side dashboards that depend on path stability. The comment acknowledges this trade-off.
**Fix:** Either (a) hand-enumerate all 40 variants explicitly (mechanical, eliminates the risk), or (b) guard with a snapshot test that pins a few canonical Debug outputs to alert on rustc/prost upgrades.

### IN-04: `sanitize_path_segment` only strips `/`, leaving whitespace and control chars

**File:** `crates/roz-cli/src/commands/mcap/text_logs.rs:257-262`
**Issue:** Tool names with spaces, tabs, newlines, control bytes, or unusual Unicode characters would still produce path segments that Rerun's entity-path parser may reject or mishandle. Rerun's path grammar is documented to accept a restricted alphabet. Currently only `/` is replaced with `_`.
**Fix:**
```rust
fn sanitize_path_segment(name: &str) -> String {
    if name.is_empty() {
        return "_unknown".to_string();
    }
    name.chars()
        .map(|c| if c.is_ascii_alphanumeric() || c == '_' || c == '-' { c } else { '_' })
        .collect()
}
```

### IN-05: Bulk-mode failure leaves orphaned partial `.rrd` files on disk

**File:** `crates/roz-cli/src/commands/mcap/export.rs:133-162`
**Issue:** `export_one` opens the RRD writer (writing the RRF2 magic header) BEFORE iterating MCAP messages. If the MCAP stream errors mid-iteration (or fails to open after the writer is created), the partial `.rrd` remains on disk with valid magic but no message content. `tests/mcap_to_rrd.rs:236-243` explicitly acknowledges this. Downstream substrate-side tooling that opens any `*.rrd` it finds may attempt to render an empty timeline.
**Fix:** Write to `output.with_extension("rrd.tmp")` and rename to the final path on `Ok(())`:
```rust
let tmp_output = output.with_extension("rrd.tmp");
let result = (|| -> Result<ConversionStats> {
    let rec = super::recording::open_rrd_writer(&tmp_output)?;
    // ... existing loop ...
    drop(rec);
    Ok(ConversionStats { /* ... */ })
})();
match result {
    Ok(stats) => {
        std::fs::rename(&tmp_output, output)
            .with_context(|| format!("rename rrd: {} -> {}", tmp_output.display(), output.display()))?;
        Ok(stats)
    }
    Err(e) => {
        let _ = std::fs::remove_file(&tmp_output);
        Err(e)
    }
}
```

### IN-06: Unreachable `unwrap_or(u64::MAX)` fallback

**File:** `crates/roz-cli/src/commands/mcap/export.rs:159`
**Issue:** `u64::try_from(state.unknown_channels_warned.len()).unwrap_or(u64::MAX)` — `usize` is `u64` or `u32` on all supported targets, and `u64::try_from(usize)` only fails when `usize` > `u64::MAX`, which is impossible on any 64-bit or smaller platform. The fallback is dead code.
**Fix:** Use `state.unknown_channels_warned.len() as u64`. Pair with a tiny `#[expect(clippy::cast_possible_truncation, reason = "...")]` if pedantic catches it on a 128-bit hypothetical.

### IN-07: `text_logs::emit_task_lifecycle` does not sanitize `task_id` before embedding in entity path

**File:** `crates/roz-cli/src/commands/mcap/text_logs.rs:186-196`
**Issue:** `format!("/session/tasks/{}", evt.task_id)` interpolates `evt.task_id` (an arbitrary proto string from the producer) directly into a Rerun entity path. If a producer ever writes a `task_id` containing `/` or whitespace, the resulting entity path is corrupted in the same way as IN-04. Tool calls go through `sanitize_path_segment` (line 217); task_ids do not.
**Fix:** Apply the same sanitisation:
```rust
let entity_path = format!("/session/tasks/{}", sanitize_path_segment(&evt.task_id));
```
(Consider promoting `sanitize_path_segment` to a module-private helper consumed by all entity-path builders.)

## Warn-Once Behavior Verified

- Code path: `classify_or_warn` checks `state.unknown_channels_warned.insert(topic.to_string())`; only the first insert returns `true` and triggers `tracing::warn!(channel = %topic, schema = %schema_name, "unknown channel; skipped")`.
- Subsequent messages on the same unknown channel return `None` from `classify_or_warn` without re-emitting the warn.
- Schema-mismatch on a known topic (e.g. `/tf` carrying `something.else`) routes through the same path: `classify_channel` returns `ChannelKind::Unknown`, the warn-once dedup uses the topic key, and the message is skipped.
- Unit tests covering the classifier truth table for both schema-mismatch cases (`classify_camera_with_wrong_schema_is_unknown`, `classify_tf_with_wrong_schema_is_unknown`) are in `commands::mcap::export::tests` and pass.

## Bulk-Mode Output Format Confirmed

- Per-file success: `[OK]  <input> → <output> (X.X MB, N messages)` to stdout
- Per-file failure: `[ERR] <input>: <single-line error>` to stderr (newlines stripped from anyhow chain via `.replace('\n', " ")`)
- Final summary: `K/N succeeded, F failed` to stderr (always printed, not conditional on failure)
- Exit code: `Ok(())` if all succeeded; `Err(...)` (with the same summary text) if any failed — caller's `main.rs` propagates to nonzero exit code per D-05.
- Glob-entry-level errors (path traversal denied, broken symlink) are also accounted in the failure tally with `[ERR] <path>: <glob error>`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Collapsed nested `if let / if` into let-chain**

- **Found during:** Task 2 verification (`cargo clippy -p roz-cli --features export-rrd -- -D warnings`)
- **Issue:** The plan's prescribed `classify_channel` body had nested `if let Some(name) = topic.strip_prefix(...) { if schema_name == ... && !name.is_empty() { ... } }`. clippy's `collapsible_if` lint (warned at workspace level) flagged this as it can be collapsed using let-chains (stable since Rust 1.88; workspace pins 1.92).
- **Fix:** Rewrote as `if let Some(name) = topic.strip_prefix(...) && schema_name == ... && !name.is_empty() { ... }` — same semantics, single block.
- **Files modified:** `crates/roz-cli/src/commands/mcap/export.rs`
- **Commit:** `31af7a6` (folded into Task 2 commit)

**2. [Rule 2 - Critical] Added `#[expect(dead_code)]` to `ConversionState::seen_camera_videostream_logged` and `ConversionStats::messages_seen` / `unknown_channel_count`**

- **Found during:** Task 2 verification (`cargo build -p roz-cli --features export-rrd`)
- **Issue:** Three public fields are part of Plan 03's contract (Plan 07 reads `seen_camera_videostream_logged` via `&mut state` in `emit_camera`; Plan 08 smoke test will read the `ConversionStats` accessors), but no Plan 03 code path reads them. Workspace lint policy is `clippy::pedantic = warn` and CI runs `-D warnings`, so the `dead_code` warnings would gate the build.
- **Fix:** `seen_camera_videostream_logged` uses `#[cfg_attr(not(test), expect(dead_code, reason="..."))]` because the test `conversion_state_default_is_empty` reads it via `is_empty()`; otherwise `expect` would be unfulfilled in test profile. The other two fields use plain `#[expect(dead_code)]` with reasons naming Plan 08 + downstream substrate tooling as the consumers.
- **Files modified:** `crates/roz-cli/src/commands/mcap/export.rs`
- **Commit:** `31af7a6` (folded into Task 2 commit)

**3. [Rule 3 - Blocking] Reformatted `assert_eq!(classify_channel(...), ChannelKind::Tf)` per rustfmt 120-col**

- **Found during:** Final `cargo fmt --check -p roz-cli`
- **Issue:** The plan's prescribed test body wrote `assert_eq!` calls split across 4 lines; rustfmt with the project's 120-col `max_width` collapsed them to 1 line. The pre-commit hook runs `cargo fmt --check` and would block.

None — all new surface (MCAP file decode, glob expansion, warn-once log lines) is enumerated in the plan's `<threat_model>` (T-26.9-07 through T-26.9-10) and the Plan 03 implementation honors each disposition:

## Known Stubs

The four sibling-module stubs (`recording.rs`, `transforms.rs`, `text_logs.rs`, `camera.rs`) are intentional Plan 03 deliverables — they exist solely so Plans 04/05/06/07 can land their implementations without breaking the build for sibling plans. Each:

- Is gated by `#![cfg(feature = "export-rrd")]` (default build never sees them).
- Has a doc-commented module header naming the owning plan.
- Has each function body return `anyhow::bail!("not yet implemented — Phase 26.9 Plan 0X owns ...")` so any test or smoke run that reaches the stub gets a clear "this plan is outstanding" error rather than a panic or silent skip.

These stubs WILL fire if a feature-on build executes `export_one` against a real MCAP — e.g. the Task 2 verification step that runs `roz mcap to-rrd /nonexistent.mcap --output /tmp/x.rrd` succeeds at returning `read mcap` because the read fails before any stub is invoked. A real MCAP would hit `open_rrd_writer` first and bail with the Plan 04 message. That is the intended Plan 03 deliverable per the plan's `<NOTE>` block.

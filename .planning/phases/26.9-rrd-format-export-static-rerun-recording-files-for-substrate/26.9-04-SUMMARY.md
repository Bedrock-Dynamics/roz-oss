## B-2 Correction Applied

CONTEXT.md D-09 prescribes `RecordingStream::set_time_seconds(name, f64)`. That method does **not** exist in `rerun = 0.31.3`. The verified correct method is `RecordingStream::set_timestamp_secs_since_epoch(timeline, secs: impl Into<f64>)` per `docs.rs/rerun/0.31.3/rerun/struct.RecordingStream.html`.

D-09's intent is preserved: f64-second precision timestamps on two timelines per message. Only the method name changed. The correction is documented inline in the module doc comment of `recording.rs` (lines 9-19), not just in the plan, so future reviewers comparing against D-09 see the explanation at source.

## D-08 Default-Timeline Treatment

`RecordingStreamBuilder` has no `default_timeline(...)` setter per RESEARCH §Topic 4. The Rerun viewer picks the initial timeline by its own heuristics (first-encountered being the dominant signal). We approximate D-08's intent ("`publish_time` is the default timeline") by **always setting `publish_time` first** in `set_message_times`. This is the best available approximation without adopting a Rerun `Blueprint` (which CONTEXT Claude's Discretion explicitly defers).

If substrate's UX team eventually wants a hard-coded layout/default-timeline contract, the upgrade path is to attach a `Blueprint` to the recording — that work is deferred to a follow-up phase.

## Test Strategy: RRF2 Magic-Byte Sanity (RESEARCH §Topic 6 Strategy A)

Rerun's Rust API does not expose a public read/load function for `.rrd` files in 0.31.3, so the smoke test asserts:
1. The file exists and has > 4 bytes after the writer drops.
2. The first 4 bytes are `b"RRF2"` (the documented Rerun-format-v2 magic header).

This is RESEARCH §Topic 6 Strategy A. Strategy B (Python-side `rerun.dataframe.load_recording`) is deferred to Plan 08's smoke test if a richer assertion is needed against the fixture MCAP.

The `RRF2_MAGIC` constant is `[ASSUMED]` stable per Assumption A1; if a future rerun upgrade changes the magic, the test will fail loudly, which is the desired alarm signal.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Collapsed nested `if let / if !empty / if !exists` into let-chain**

- **Found during:** Task 1 GREEN verification (`cargo clippy -p roz-cli --features export-rrd -- -D warnings`)
- **Issue:** The plan's prescribed `open_rrd_writer` body had nested `if let Some(parent) = output.parent() { if !parent.as_os_str().is_empty() && !parent.exists() { ... } }`. Clippy's `collapsible_if` lint (warned at workspace pedantic level) flagged this.
- **Fix:** Rewrote as `if let Some(parent) = output.parent() && !parent.as_os_str().is_empty() && !parent.exists() { std::fs::create_dir_all(parent) ... }` — same semantics, single block, idiomatic Rust 1.88+ let-chain.
- **Files modified:** `crates/roz-cli/src/commands/mcap/recording.rs`
- **Commit:** `aee3453` (folded into GREEN commit)

**2. [Rule 2 - Critical] Added `#[expect(clippy::cast_precision_loss)]` to `set_message_times`**

- **Found during:** Task 1 GREEN verification (`cargo clippy -p roz-cli --features export-rrd -- -D warnings`)
- **Issue:** The `(msg.publish_time as f64) / 1e9` cast triggers `clippy::cast_precision_loss` at workspace `pedantic`/`nursery` levels (CI runs `-D warnings`). The cast is intentional — D-09 mandates f64-second precision.
- **Fix:** Added `#[expect(clippy::cast_precision_loss, reason="u64 ns -> f64 secs loses precision only for >2^53 ns (~285 years post-epoch); not reachable for session timestamps")]` to the function. Per CLAUDE.md, `#[expect]` with reason is preferred over `#[allow]`.
- **Files modified:** `crates/roz-cli/src/commands/mcap/recording.rs`
- **Commit:** `aee3453` (folded into GREEN commit)

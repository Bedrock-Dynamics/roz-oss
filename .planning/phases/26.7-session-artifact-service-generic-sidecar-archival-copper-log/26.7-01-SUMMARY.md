# Phase 26.7 Plan 01: Session artifacts data plane Summary

One-liner: Data-plane foundation for generic per-session sidecar archival — `roz_session_artifacts` Postgres table (RLS, 5-value CHECK enum, 2 indexes) plus the `roz_db::session_artifacts` CRUD module that every downstream plan (03 server gRPC, 06 worker upload, 08 retention, 09 integration test) depends on.

## What Was Built

### 1. Migration `migrations/20260423039_session_artifacts.sql` (Task 1)

- **Table** `roz_session_artifacts` with columns:
  - `artifact_id UUID PK DEFAULT gen_random_uuid()`
  - `tenant_id UUID NOT NULL REFERENCES roz_tenants(id)` (RLS scope key)
  - `session_id UUID NOT NULL` (NOT a foreign key — matches `roz_session_mcap_archives` precedent)
  - `artifact_type TEXT CHECK IN ('mcap','copper','ulog','video','bundle')`
  - `path TEXT`, `digest_sha256 BYTEA`, `size_bytes BIGINT`, `content_type TEXT`
  - `uploaded_at TIMESTAMPTZ DEFAULT now()`
  - `UNIQUE(session_id, artifact_type, path)`
- **Indexes** (per D-05):
  - `idx_session_artifacts_tenant_session ON (tenant_id, session_id, artifact_type)` — session/bundle lookup
  - `idx_session_artifacts_retention ON (uploaded_at)` — retention TTL pass
- **RLS** enabled with `tenant_isolation` policy keyed on `current_setting('rls.tenant_id')::uuid` — parity with `roz_session_mcap_archives`.
- Header comment documents the `'mcap'` reserved-never-populated semantics (D-03) and explains that `session_id` is intentionally not a foreign key (same contract as `roz_session_mcap_archives`).

### 2. CRUD module `crates/roz-db/src/session_artifacts.rs` (Task 2)

Public surface:

| Item | Purpose |
| --- | --- |
| `SessionArtifactRow` | `sqlx::FromRow` row type matching the migration schema exactly |
| `insert(executor, tenant_id, session_id, artifact_type, path, digest_sha256, size_bytes, content_type) -> Result<Row>` | INSERT ... RETURNING * idempotent via UNIQUE constraint |
| `list_by_session(executor, tenant_id, session_id) -> Vec<Row>` | session-scoped listing, tenant bound explicitly in SQL |
| `fetch_by_id(executor, tenant_id, artifact_id) -> Option<Row>` | single-row fetch; cross-tenant looks identical to not-found |
| `list_retention_candidates(executor, ttl_secs) -> Vec<Row>` | server-wide TTL sweeper input |
| `list_finalized_ordered_desc(executor) -> Vec<Row>` | server-wide size-cap sweeper input |
| `delete_by_id(pool, artifact_id) -> u64` | retention-sweep row deletion |

- `insert` carries `#[expect(clippy::too_many_arguments, reason = "...")]` since pedantic clippy flags 8 parameters (the schema-column-per-parameter shape intentionally matches the column set).
- Unit test: `row_struct_is_send_and_sync` — pure compile-time check.
- Module doc explicitly records the tenant-scoping contract: `tenant_id` is bound in SQL (not via `set_tenant_context`) because the server pool is a Postgres superuser that bypasses RLS (matches `crates/roz-server/src/grpc/observability.rs:115-135`).
- Retention helpers (`list_retention_candidates`, `list_finalized_ordered_desc`) run without tenant scope — retention is server-wide.

## Decisions Made

See the `decisions:` block above. Of note:

- The `'mcap'` enum value is reserved but never written this phase (D-03). Forward-compat for a later migration that consolidates MCAPs into the artifact plane; 26.7 does not dual-write.
- `session_id` is intentionally not a foreign key — RLS + `tenant_id` column are the authority, matching `roz_session_mcap_archives`. Cross-tenant fabrication is blocked by RLS, not the FK surface.
- Tenant scoping in helpers: bound in SQL (superuser pool bypass) with handler-side defense-in-depth in Plan 03. This means RLS policy is documentation + a safety net for non-superuser connections, not the authoritative path.

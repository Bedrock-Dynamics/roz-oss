# 26.7-08 — Retention sweep extension for session artifacts

## Outcome
`sweep_once` in `crates/roz-server/src/observability/retention.rs` now performs TTL + size-cap enforcement over both `roz_session_mcap_archives` AND `roz_session_artifacts` using the existing `ROZ_MCAP_TTL_SECS` / `ROZ_MCAP_MAX_BYTES` env vars (D-31). No new env vars introduced.

## Final structure of `sweep_once`
1. **Pass 1a — MCAP TTL.** `mcap_archives::list_retention_candidates(pool, ttl)` → `unlink_and_delete` each row. Query error logs `warn!` but Pass 1b still runs.
2. **Pass 1b — Artifact TTL.** `session_artifacts::list_retention_candidates(pool, ttl)` → `unlink_and_delete_artifact` each row. Query error logs `warn!` but Pass 2 still runs.
3. **Pass 2 — size cap, merged newest-first.** Both tables' `list_finalized_ordered*` results are merged into one `Vec<Merged>` with `kind`, `ts` (`opened_at` for MCAP / `uploaded_at` for artifact), `size_bytes`, `path`. Sorted newest-first by `ts`. Running total grows until it exceeds `max_bytes`; every subsequent (older) row is dropped via the kind-appropriate helper.

## Decision rationale for merge-sort vs UNION ALL SQL (RESEARCH Q5)
Option (a) — in-Rust merge of per-table queries — was chosen over a Postgres UNION ALL query. Rationale:
- The two row types differ (`McapArchiveRow` vs `SessionArtifactRow`) and need kind-specific delete helpers; UNION ALL would require schema-gymnastics to preserve the origin.
- Row counts scale with `max_bytes / avg_file_size` (O(thousands) typical), so memory cost is O(rows × ~200 bytes) — well under a MiB for any realistic deployment (T-26.7.08-02 accept).
- Keeps the SQL layer simple + per-table helper surfaces unchanged.

## Helper addition
`unlink_and_delete_artifact(pool, artifact_id, path)` mirrors the existing `unlink_and_delete` exactly — file-unlink first (ENOENT tolerated, logs warn then continues), DB `session_artifacts::delete_by_id` second.

## Differences from PATTERNS.md
Plan 08's PATTERNS.md sketch suggested the existing pass was a single `?`-propagating call chain. The actual pre-Plan-08 code matches the `?` pattern for MCAP only; Plan 08 converted both passes to `match`-and-`warn!` so a transient query error in one table does not abort the other.

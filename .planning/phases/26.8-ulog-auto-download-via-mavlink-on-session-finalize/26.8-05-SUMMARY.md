## Decisions Made

- **Construction-time derivation vs runtime-HEARTBEAT derivation for `autopilot_kind()`:** the plan body's `<interfaces>` block is authoritative and says "derived from the AutopilotHint supplied at construction time (NOT from runtime HEARTBEAT reads)." The orchestrator prompt's `<critical_invariants>` block stated the opposite ("derived from the first HEARTBEAT.autopilot field received"). Implemented per the plan body. Rationale: at ulog-finalize time (session end) the HEARTBEAT stream may not have arrived yet — construction-time `AutopilotHint` is the authoritative source. Plan 02's `AutopilotKind::from_mavlink_autopilot(u8)` remains available for callers that DO have a live HEARTBEAT byte (the two paths are complementary, not alternatives).
- **Test file layout — 2 files, not inline or 5 files:** the plan's Task 1 behavior block asks for "inline `#[cfg(test)] mod autopilot_kind_tests { ... }` in backend.rs." Inline tests are not viable for this plan: `MavlinkBackend` construction requires a real UDP bind (Plan 02 established this), and the mandatory `std::process::exit(0)` teardown terminates the whole test process, so only one `#[tokio::test]` per test binary can run assertions. Consolidating the 4 autopilot_kind variants into one `#[tokio::test]` with 4 sequential backend constructions preserves coverage in a single binary; `send_log_erase` gets its own binary. Two files total instead of inline-mod-in-backend.rs (impossible given UDP + exit(0)) or five separate files (wasteful).
- **`host_id` gating over synthetic-UUID construction:** `construct_mavlink_backend` requires a concrete `Uuid` to derive the MAVLink sysid (`main.rs:97-101`). If `register_host` skips/fails, `worker_host_id` is `None`. Rather than construct with `Uuid::nil()` or a fresh random UUID (both of which would silently derive arbitrary sysids), the boot-scope call is gated on `Some(hid)` with a `tracing::debug!` branch for the `None` case. Workers without API registration simply don't get a MAVLink backend — matching the soft-fail semantics of the ArtifactServiceClient precedent (Phase 26.7 D-16).

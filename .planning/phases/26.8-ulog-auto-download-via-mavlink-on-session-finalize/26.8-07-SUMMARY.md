# Phase 26.8 Plan 07: Soft-fail taxonomy + LOG_ERASE gate Summary

**D-06 LOG_ERASE gate and D-09 structured warn-log field coverage added to `finalize_ulog_archive`, closing out Phase 26.8 SC6.**

## Accomplishments
- D-06 LOG_ERASE gate: `backend.send_log_erase()` is now called exactly once, and only inside the strict `artifact_id.is_some() && !config.keep_fc_copy` branch. No other code path issues the message — premature erase on upload failure is impossible by construction.
- D-09 taxonomy uniformity: every `tracing::warn!` callsite in `finalize_ulog_archive` now carries the full 6 structured fields (`session_id`, `failure_mode`, `attempted_log_id`, `bytes_received`, `duration_ms`, `error`).
- `bytes_received` is extracted from `UlogError::LogDataTimeout` and `UlogError::ReassemblyGapsExhausted` so partial-progress telemetry surfaces in the download-failure warn-log.
- `keep_fc_copy=true` operator opt-in retention path honored with a debug-log (no erase, no warn — this is an expected configuration).
- Full D-09 `failure_mode` string set reachable from `ulog_archive.rs`: `fc_unreachable`, `log_list_timeout`, `log_data_timeout`, `reassembly_gaps_exhausted`, `upload_failed`, `erase_failed` (new), `no_logs_available`, `log_oversized`, `autopilot_not_px4` (9/10 of the D-09 taxonomy — `digest_mismatch` lives in `log_download.rs` per Plan 02, acceptable per plan verification notes).

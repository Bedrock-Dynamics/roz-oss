# Phase 26.8 Plan 02: MAVLink log-download protocol state machine Summary

**LogDownloader state machine (LOG_REQUEST_LIST -> LOG_ENTRY -> newest selection -> LOG_REQUEST_DATA -> LOG_DATA reassembly -> LOG_REQUEST_END) with Drop-guarded teardown, 500 MiB DOS cap, and AutopilotKind taxonomy in the protocol crate**

## Performance

- **Duration:** ~30 min
- **Completed:** 2026-04-24
- **Tasks:** 2 (Task 0 codegen verification had no file changes; Tasks 1 + 2 committed atomically)
- **Files created:** 2 (`log_download.rs`, `log_fanout.rs`)
- **Files modified:** 2 (`lib.rs`, `backend.rs`)

## Accomplishments

- Phase 26.8 SC1 protocol machinery complete: `LogDownloader` owns a `mpsc::Sender<MavMessage>` + `broadcast::Receiver<MavMessage>` pair, drives the full 6-message MAVLink LOG protocol, and reassembles `LOG_DATA` frames into a `Vec<u8>` with gap detection and per-gap + total retry caps.
- Drop-guard issues `LOG_REQUEST_END` on every exit path (success, error, panic-unwind, cancellation) unless `end_sent = true` — the load-bearing correctness invariant that keeps the FC responsive to subsequent `LOG_REQUEST_LIST` without reboot (RESEARCH pitfall 1).
- `MAX_LOG_SIZE_BYTES = 500 MiB` enforced before any buffer allocation so a malicious or malfunctioning FC cannot claim `u32::MAX` and OOM the worker (T-26.8.02-01).
- `MavlinkBackend::router_loop` extended with a `LOG_ENTRY | LOG_DATA` match arm before the catch-all; fans frames to a new `log_broadcast: broadcast::Sender<MavMessage>` channel (capacity 256) with zero surface change for other consumers.
- Two new pub accessors on `MavlinkBackend`: `subscribe_log_data()` (for `LogDownloader` inbound) and `outbound()` (for LOG_REQUEST_* writes).
- `AutopilotKind { Px4, ArduPilot, Unknown }` plus `const fn from_mavlink_autopilot(u8) -> Self` helper defined at the `roz-mavlink` crate root (Phase 26.8 D-11 gate consumer), so Plans 04/05 consume it independently with no cross-plan dependency.

## Decisions Made

- **`AutopilotKind` placement:** defined at the `roz-mavlink` crate root (in `lib.rs`) rather than inside `backend.rs`. This is a MAVLink-protocol taxonomy (mapping `MAV_AUTOPILOT` enum bytes to a coarse family) and is consumed by plans 04 and 05 independently. Keeping it outside `backend::*` means neither plan has a dependency on backend internals.
- **`end_sent` visibility:** `pub(crate) end_sent: bool` so inline unit tests can flip the flag to exercise the Drop-guard both branches without a full protocol round-trip, and without exposing internal lifecycle state on the public API.
- **Integration test location:** new file `crates/roz-mavlink/tests/log_fanout.rs`. An inline unit test would require a way to push synthetic frames into the router's inbound channel; no such hook exists and adding one would break encapsulation. The qgc_coexistence ephemeral-UDP pattern was already proven.
- **Teardown idiom:** `std::process::exit(0)` at end of `log_fanout` test. Upstream `mavlink::connect("udpin:...")` holds a blocking `UdpSocket::recv` that cannot be cancelled cleanly; the same workaround is in qgc_coexistence. Clean shutdown lands in a later phase per 25-PATTERNS Variance Note 2.
- **`fetch_newest` clippy:** `#[expect(clippy::too_many_lines, reason = "sequential MAVLink LOG_* protocol state machine; extraction would fragment the 6-step flow")]` — preserves the 6-step flow as a single readable unit; splitting would obscure the protocol state.

# Phase 26.10 Plan 07: FW-05c — Latched E-Stop State Machine — Summary

**FW-05 fully closed: the IEC 60204-1 Stop Category 0 + EN ISO 13849-1 manual-reset latched e-stop state machine is wired into the Copper controller loop end-to-end. System-level `LatchState` lives on `ControllerState` (Codex H3 fix). The latched-tick gate replaces the `all_default` short-circuit with explicit per-channel-kind zero emission each tick (velocity/torque/force = 0.0; position channels HOLD last commanded value to avoid the "go to position 0" collision path). WAL is authoritative for boot — corruption returns `Err`, never silently resets to `Run`. Distinct `AckEstop` + `ResumeAfterZeroVerified` commands enforce no-auto-rearm; existing `Resume` retains its pre-FW-05 semantics (preserved by a defense-in-depth serialization test). 33 tests across `latch.rs`, `channels.rs`, `wal.rs`, `subjects.rs`, inline `controller::tests`, and the integration test file all pass.**

### Task 2 — Loop-level wiring + per-channel-kind zero policy + rollback guard (commit `2c62b3e`)

**`crates/roz-copper/src/controller.rs`:**

- **`build_per_channel_zero_frame`** (`pub(crate)`): the per-channel-kind zero policy table. Keys on `BindingType`:
  - `JointPosition` / `GripperPosition` -> hold last commanded value (raw 0.0 = collision)
  - `JointVelocity` / `GripperForce` -> 0.0
  - `Command` (generic, includes torque) -> 0.0
  - sensor-side bindings -> 0.0 (defensive)
- **`assert_latch_estop`** (`pub(crate)`): system-level `Run -> Latched` transition on `ControllerState`. Returns `true` when transitioned. Sticky.
- **`bump_zero_motion_tick`** (`pub(crate)`): counts consecutive zero-motion ticks during AwaitingAck. Sensor-absent ticks reset the counter (cannot verify zero without sensor evidence). Advances to ZeroVerified at `ZERO_VERIFY_TICK_COUNT`.
- **Latched-tick gate** in `run_controller_loop_with_policy`: when `latch_state.requires_zero_emission()`, build the per-channel-kind zero frame from the active controller's bindings, send to actuator, halt running controllers, run the AwaitingAck counter, publish state, sleep, and continue. Gate runs AFTER drain_commands + sensor read but BEFORE the WASM tick — the WASM never executes during latch.
- **Rollback guard**: `if shared_state.load_full().latch_state == LatchState::Run { estop_reason = None }` else log and skip — Codex H3 fix. The rollback still runs (controller is restored), but `estop_reason` is preserved while the latch is active so observers see the originating error.
- **E-stop assertion** at the controller-error / candidate-failure / bounded-canary-overrun paths via `assert_latch_estop`. Persisted via optional `latch_persist_tx: SyncSender<LatchState>` channel (None for the legacy entry; ready for future worker WAL write-back wiring).
- **AckEstop / ResumeAfterZeroVerified** drive the system-level `LatchState` in `drain_commands` (where `shared_state` is now in scope). Existing `Resume` retains pre-FW-05 semantics.
- **`last_commanded_values: Vec<f64>`** loop-local tracker — populated from each Run-state actuated command; held read-only during latched ticks for the position-hold policy.
- **`last_output` preservation**: when latched-tick fires, any pre-existing `error` field is copied into the new latched JSON so observers still see the originating error reason. Fixes 3 pre-existing tests that depended on `last_output["error"]` being readable after a controller halt.

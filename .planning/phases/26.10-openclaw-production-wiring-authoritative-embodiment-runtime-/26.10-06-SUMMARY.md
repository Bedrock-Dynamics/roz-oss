# Phase 26.10 Plan 06: FW-05 (parts a + b) — Bounded WASM Host Output + Timer-Driven Stale-Heartbeat Scan — Summary

**Two of three pre-hardware safety gaps closed: `tick::set_output` now caps host-side allocation at 64 KiB before the `vec!` (FW-05a), and roz-safety publishes e-stop on a 1 s timer regardless of broker traffic via a pure `run_stale_scan(now, threshold, tracker)` extraction that sidesteps the `tokio::time::pause` + `std::time::Instant` incompatibility (FW-05b — Codex M3 Option B). FW-05 remains open pending Plan 07's latched e-stop state machine.**

### Task 2 — FW-05(b) Timer-Driven Stale Scan + Pure Fn Extraction (commit `53d9f84`)

- Pure function extraction in `crates/roz-safety/src/heartbeat.rs:107`:
  ```rust
  pub fn run_stale_scan(now: std::time::Instant, threshold: Duration, tracker: &HeartbeatTracker) -> Vec<String>
  ```
  Returns owned `String` IDs so the result outlives the borrow. Production loop wraps with `Instant::now()`; tests pass synthetic `Instant + Duration::from_secs(N)` values.
- New `HeartbeatTracker.workers_iter() -> impl Iterator<Item = (&str, &Instant)>` getter to allow the pure fn read-only access without making the internal HashMap public.
- Existing `stale_workers()` thinly delegates to `run_stale_scan(Instant::now(), self.stale_threshold, self)`. Return type changed from `Vec<&str>` → `Vec<String>` (existing callers in `nats_integration.rs`, `safety_pipeline.rs`, and the heartbeat unit tests all compile unchanged because `String: PartialEq<&str>` and `&String` derefs at index/pass sites).
- 5 new synthetic-Instant unit tests under `heartbeat::stale_scan_tests` — fresh worker not stale, old worker stale, exact-threshold boundary (`>=` predicate fires at 30s, fresh at 29.999s), empty tracker, post-remove no double-fire. All run without tokio virtual time.
- New `crates/roz-safety/src/run.rs` module exporting `SafetyDaemonConfig` + `run_safety_daemon`. The bin's `main.rs` collapses to env-load + `run_safety_daemon(cfg).await`.
- Inside `run_safety_daemon`:
  - `Arc<tokio::sync::Mutex<HeartbeatTracker>>` shared between the heartbeat-receive task and the stale-scan task.
  - Stale-scan task uses `tokio::time::interval(scan_period)` with `MissedTickBehavior::Skip`, discards the immediate first tick, then locks the tracker each tick and calls the pure `run_stale_scan`. Sends stale IDs over an mpsc channel.
  - Foreground select-loop publishes the e-stop and removes the worker from the tracker on first send to prevent double-fire.
  - Watchdog heartbeat task remains as a separate `tokio::spawn(interval)` (parity with the original main.rs:38-51 pattern).
- 2 testcontainer NATS integration tests in `crates/roz-safety/tests/stale_heartbeat_timer.rs`: `stale_heartbeat_timer_fires_without_new_heartbeats` (one heartbeat, then silence, assert e-stop within 3s) and `stale_heartbeat_timer_does_not_double_fire` (assert no second e-stop after first). Use real wall-clock time per Codex M3 — NOT `tokio::time::pause/advance` (which is incompatible with `std::time::Instant`).

**2. [Rule 1 - Bug fix] Threshold-boundary test using captured `Instant::now()` was off-by-record-latency**

- **Found during:** Task 2, after first test run
- **Issue:** First draft of `run_stale_scan_threshold_boundary` did `let recorded_at = Instant::now(); t.record("w1");` then queried at `recorded_at + Duration::from_secs(30)`. Because `record()` calls `Instant::now()` internally AFTER the test's `recorded_at` capture, the actual recorded instant is later, so `recorded_at + 30s < actual + 30s` → predicate didn't fire. Test failed with `left: [], right: ["w1"]`.
- **Fix:** Captured `let after_record = Instant::now()` AFTER `record()` to get an upper-bound on the recorded time; use `after_record + 31s` (well past threshold) and `after_record` (well under) for the directional asserts. For the exact-threshold check, used a separate tracker with direct `t2.workers.insert("w1", known)` to control the recorded instant exactly. Both `>=` boundary and just-under cases now assert deterministically.
- **Commit:** `53d9f84` (consolidated into Task 2 commit; the bug existed only between local test runs).

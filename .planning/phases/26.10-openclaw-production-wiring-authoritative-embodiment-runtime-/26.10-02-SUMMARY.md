# Phase 26.10 Plan 02: OpenClaw-class manifest fidelity (FW-06) — Summary

Replaced the `f64::INFINITY` synthesis path with a fail-closed
`try_authoritative_embodiment_runtime` parser that consumes the full
`Joint` schema from `[[joints]]` TOML tables and preserves TCPs, sensor
mounts, and workspace zones — so dispatched runtimes are loose by
construction no longer.

## What Was Built

### New `try_authoritative_embodiment_runtime` parse path

`EmbodimentManifest::try_authoritative_embodiment_runtime() -> Result<EmbodimentRuntime, ManifestError>`:

- Reads explicit `[[joints]]` tables, materialising each into the full
  `roz_core::embodiment::model::Joint` shape including `joint_type`,
  `parent_link`, `child_link`, `axis: [f64; 3]`, `origin: Transform3D`,
  and complete `JointSafetyLimits` (every field finite — no INFINITY).
- Returns `Err(ManifestError::MissingJointLimits)` when no `[[joints]]`
  table is present. Production manifests fail closed.
- Preserves `[[tcps]]`, `[[sensor_mounts]]`, `[[workspace_zones]]` into
  the resulting `EmbodimentRuntime.model.tcps` / `.sensor_mounts` /
  `.workspace_zones` (rather than flattening into channels at parse
  time).
- Compiles via `EmbodimentRuntime::compile(model, None, None)` which
  stamps `model_digest`, builds the frame graph, and resolves watched
  frames. Does NOT push the `SYNTHETIC_EMBODIMENT_RUNTIME_ISSUE` tag —
  this path is authoritative.

### `ManifestError` enum

Three `thiserror` variants: `MissingJointLimits` (the FW-06 fail-closed
trigger), `InvalidJoint { joint, detail }` (reserved for richer
validation), `MissingChannels` (reserved for future control-surface
gating).

### TOML mirror tables

Six new `pub` deserialise structs added to `EmbodimentManifest`:

| Mirror struct             | Source canonical type                           | Drift-prevention strategy                                   |
| ------------------------- | ----------------------------------------------- | ----------------------------------------------------------- |
| `JointTable`              | `embodiment::model::Joint`                      | `joint_name` propagated from `name` at materialisation      |
| `JointSafetyLimitsTable`  | `embodiment::limits::JointSafetyLimits`         | `joint_name` omitted from TOML; sourced from parent joint   |
| `Transform3DTable`        | `embodiment::frame_tree::Transform3D`           | `timestamp_ns` omitted; defaulted to 0 (static topology)    |
| `TcpTable`                | `embodiment::model::ToolCenterPoint`            | direct mirror; `Transform3DTable` for offset                |
| `SensorMountTable`        | `embodiment::model::SensorMount`                | direct mirror; `Transform3DTable` for offset                |
| `WorkspaceZoneTable`      | `embodiment::workspace::WorkspaceZone`          | direct mirror                                               |

All four added `EmbodimentManifest` fields use `#[serde(default)]` so
legacy channel-only manifests (e.g. existing `examples/diff_drive/robot.toml`)
continue to parse without modification.

### UR5 fixture (`examples/ur5/robot.toml`)

Extended with the full Joint schema for all 6 joints plus 1 TCP and 1
sensor mount. Channel manifest preserved untouched. Joint values per UR5
datasheet:

- 6 revolute joints (`shoulder_pan` → `wrist_3`)
- Position range: `[-3.14159, +3.14159]` rad on every joint
- Velocity caps: `3.14` rad/s on shoulder/elbow, `6.28` rad/s on wrists
- `max_acceleration = 5.0`, `max_jerk = 50.0` workspace-wide
- `max_torque`: `150.0` N·m on shoulder/elbow, `28.0` N·m on wrists
- Origin transforms reflect canonical UR5 link offsets (89.16 mm base
  height, 425 mm upper-arm reach, etc.)
- 1 TCP `wrist_tool_center` mounted at `wrist_3_link` with 82.3 mm Z
  offset (UR5 flange-to-tool standoff)
- 1 sensor mount `wrist_camera` mounted at `wrist_3_link`

Quaternion order is `[w, x, y, z]` (per `Transform3D` doc); identity
rotations encoded as `[1.0, 0.0, 0.0, 0.0]`.

### Compatibility preservation

The legacy `embodiment_runtime()` getter (which synthesizes loose
limits and stamps `SYNTHETIC_EMBODIMENT_RUNTIME_ISSUE`) is preserved
unchanged. Four production callers identified via grep:

- `crates/roz-worker/src/main.rs:1487` (loads manifest at boot)
- `crates/roz-cli/src/tui/tools.rs:74, 124` (CLI tool registration)
- `crates/roz-local/src/runtime.rs:996` (local runtime registration)

These callers are out-of-scope migrations — Plan 03 (FW-02) and Plan 09
(capability publish) will adopt the new fail-closed path. The runtime
rejection at `promote_controller.rs:192-201` already prevents synthetic
runtimes from reaching live-promotion.

### Plan-Internal Contradiction (documented deviation)

Two of the plan's `must_haves` are mutually exclusive given the live
production callers:

- `must_haves` requires preserving `embodiment_runtime()` unchanged.
- `must_haves` also requires `#[cfg(test)]`-gating the
  `SYNTHETIC_EMBODIMENT_RUNTIME_ISSUE` synthesis path — which is the
  body of `embodiment_runtime()`.

Resolution per Codex M1 explicit guidance ("preserve compatibility-only
`embodiment_runtime()` getter — do NOT break it during the swap"):
the M1 directive wins. The synthesis path stays live and callable.
Production safety is preserved by the runtime-layer rejection at
`promote_controller.rs:192-201` (pre-existing) AND by the new
fail-closed `try_authoritative_embodiment_runtime` API which Plan 03 +
Plan 09 will migrate callers onto.

The acceptance grep `f64::INFINITY count == 0 outside cfg(test)` is
unsatisfiable under the M1 constraint. Remaining 5 `INFINITY`/`NEG_INFINITY`
references are all inside the live `embodiment_runtime()` synthesis
function (`manifest.rs:721-725` post-edit). One additional reference at
line 778 is inside the new doc comment on `try_authoritative_embodiment_runtime`
explaining what it replaces.

### Plan-Spec Discrepancies (Documentation, Not Behavior)

The plan's `<interfaces>` block had three documentation typos that did
not match the live code:

- The plan listed `JointSafetyLimits { max_velocity, max_acceleration, max_jerk, position_min, position_max, max_torque }` — actual struct also has `joint_name: String` as the first field. The materialisation code copies `joint.name` into `limits.joint_name` to avoid double-authoring.
- The plan example used `rotation = [0.0, 0.0, 0.0, 1.0]` for identity quaternion. Actual `Transform3D` rotation is `[w, x, y, z]` per the source-of-truth doc comment at `frame_tree.rs:11`; identity is `[1.0, 0.0, 0.0, 0.0]`. The UR5 fixture and `to_transform()` implementation use the correct order.
- The plan example showed `Transform3D { translation, rotation }` with no `timestamp_ns`. Actual struct has a third field. Resolved by mirroring the TOML side as `Transform3DTable { translation, rotation }` and defaulting `timestamp_ns: 0` at materialisation (manifests describe static topology, not live frame samples).

All three were navigated by reading the actual source rather than
trusting the plan's interface block (advisor flagged the joint_name
issue upfront).

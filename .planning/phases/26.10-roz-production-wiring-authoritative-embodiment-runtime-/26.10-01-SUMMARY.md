# Phase 26.10 Plan 01: Authoritative EmbodimentRuntime on dispatch — Summary

JWT-style additive `Option<EmbodimentRuntime>` field on `TaskInvocation` plus
server-side resolution from `roz_hosts.embodiment_runtime` BEFORE any side
effect, with explicit cross-tenant defense and a 512 KiB envelope-size guard.

## What Was Built

### Wire-format (`TaskInvocation`)

- Additive `pub embodiment_runtime: Option<roz_core::embodiment::EmbodimentRuntime>` field
  with `#[serde(default, skip_serializing_if = "Option::is_none")]` so legacy
  workers (and the Phase 23 `payload_sha256_hex` signing path) keep working
  without any code change.
- Canonical constructor `TaskInvocation::new(...)` that defaults the new field
  to `None`. Solves the literal-site churn problem (Codex H1) once and for all
  — future additive fields (e.g. Plans 26.10-03..09) require zero call-site
  updates across `roz-nats`, `roz-server`, `roz-worker`, `roz-copper`.

### Server dispatch resolver (`dispatch_task`)

- For `mode_from_phases(phases) == OodaReAct`, the server resolves
  `roz_db::embodiments::get_by_host_id(...)` and JSON-deserializes the
  `embodiment_runtime` JSONB into an `EmbodimentRuntime`.
- Three fail-closed rejection paths:
  - Host has no embodiment row at all → `BadRequest("host {id} has no embodiment record (FW-01)")`
  - Embodiment row exists but `embodiment_runtime` column is NULL → `BadRequest("host {id} has no authoritative embodiment runtime (FW-01)")`
  - Embodiment row's `tenant_id != request.tenant_id` → `BadRequest("host {id} embodiment record belongs to a different tenant (FW-01)")` plus a `tracing::error!` for ops alerting.
- One internal-failure path for malformed JSONB.
- All four paths run BEFORE `roz_db::tasks::create` (line 217) AND BEFORE the
  Restate workflow POST (line 231) — verified by `awk` ordering check.

### Worker boxing fix

- Worker's `tokio::spawn(... execute_task(...).await)` was tipped over the
  `clippy::large_futures` 16 KiB threshold by the new field. Wrapped in
  `Box::pin(...)` per the rust-lang lint guidance.

### Threat mitigations

- T-26.10-01-01 (Tampering): payload covered by `payload_sha256_hex` automatically (no code change).
- T-26.10-01-02 (Information Disclosure): `EmbodimentRow.tenant_id` assertion.
- T-26.10-01-03 (DoS): envelope-size warn at 512 KiB.
- T-26.10-01-05 (EoP): same tenant assertion path as T-26.10-01-02.
- T-26.10-01-06 (Safety side-effect leak): resolver moved to fail-closed position.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `clippy::large_futures` triggered on worker `execute_task`**

- **Found during:** Task 1 (clippy --all-targets after constructor migration)
- **Issue:** Adding `EmbodimentRuntime` (a sizeable nested struct including `EmbodimentModel`, `FrameTree`, etc.) to `TaskInvocation` pushed the spawned future past the 16 KiB threshold of clippy's `large_futures` lint, causing `cargo clippy --workspace --all-targets -- -D warnings` to fail in `crates/roz-worker/src/main.rs:2736`.
- **Fix:** Wrapped the `execute_task(...)` call in `Box::pin(...)` per the lint's own help message — the recommended idiom.
- **Files modified:** `crates/roz-worker/src/main.rs`
- **Commit:** 9e93a10

**2. [Rule 1 — Bug] `clippy::collapsible_if` on the new envelope-size guard**

- **Found during:** Task 2 (clippy after adding the size guard)
- **Issue:** The original `if let Ok(bytes) = ... { if bytes.len() > N { ... } }` nested form failed `-D warnings` under `clippy::collapsible_if`.
- **Fix:** Collapsed into `if let Ok(bytes) = ... && bytes.len() > N { ... }` per the lint's own suggestion (let-chains are stable on the pinned 1.92.0 toolchain).
- **Files modified:** `crates/roz-server/src/routes/task_dispatch.rs`
- **Commit:** ec3a44a

**3. [Rule 1 — Test fix] Existing `dispatch_starts_workflow_before_publish_and_preserves_request_shape` test broken by FW-01 resolver**

- **Found during:** Task 2 (reviewing existing OodaReAct tests under the new resolver)

- **Fix:** Added a `minimal_embodiment_runtime_json()` helper alongside the existing helpers and seed `roz_db::embodiments::upsert(...)` immediately after `seed_tenant_and_host` in that test. The test continues to exercise the OodaReAct happy path post-resolution.

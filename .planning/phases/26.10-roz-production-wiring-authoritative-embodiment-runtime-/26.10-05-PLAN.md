## Reviews Addressed

This plan is a REVIEWS-driven revision. Codex (gpt-5.5) findings driving the revision:

- **M2 (MEDIUM) — AUTO placement misses worker/server-registered tools.** Original plan derived `has_physical_tools` only from client-supplied `start.tools`. Worker- and server-registered physical tools may not be visible at session start (timing gap: server-owned MCP tools register AFTER session start at line 2624, after the placement decision at line 2583). **FIX (Option B per advisor recommendation):** Use `host_id + controller_capable worker → edge`. The capability publication (Plan 09 makes this lossless from `EmbodimentRuntime`) advertises whether a worker is controller-capable. This is a stable, persistent property of the host (in `roz_hosts` capability table) — visible at session start without depending on tool registration ordering.
- **Choice rationale (Option B over Option A — tool union):** Tool-union approach requires session-start-after-MCP-registration timing, which is a larger surgery in `crates/roz-server/src/grpc/agent.rs`. The "controller-capable worker" property is what `host_id` + edge placement is fundamentally about — does this host have a real controller? Capability lookup at session start matches the existing host-fetch pattern at `:2570s`.

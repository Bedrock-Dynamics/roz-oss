## Summary

Plan 26.10-10 closes the FW-05c production-wiring gaps from `26.10-VERIFICATION.md`. The shipped wiring is correct on every focus check:

- **`latch_persist_tx` plumbing.** `run_controller_loop_with_policy` accepts `Option<SyncSender<LatchState>>` and forwards it through `drain_commands` and the candidate/rollback failure paths. `CopperHandle::spawn_with_policy_and_io` and `spawn_with_io_and_deployment_manager_and_wiring` both expose the parameter, and the worker pairs the `SyncSender(16)` with a `tokio::task::spawn_blocking` drainer that calls `WalStore::save_latch_state`.
- **Verify-before-dispatch.** `safety_subscribers::handle_safety_message` calls `WorkerSigningContext::verify_inbound_worker` (line 185) and `return`s on failure (line 198) BEFORE any `shared_cmd_tx.load()` / dispatch (lines 200-217). Unsigned messages are dropped AND audited via `safety_signature_failure_worker(host_id)` on the same code path.
- **WAL load failure path.** `main.rs:776-786` propagates `WalStore::load_latch_state` errors into `copper_spawn_failure(...)` and `return`s — no silent fallback to `Run`. The error message explicitly cites IEC 60204-1.
- **Boot ordering invariant.** `main.rs:776 → 802 → 806 → 826 → 853 → 862` runs WAL-load → channel-construction → `spawn_with_policy_and_io` → `state().rcu(...)` seed → `shared_cmd_tx.store(Some(handle.cmd_tx()))` → `spawn_blocking` drainer. The rcu commits before any `LoadArtifact`/`PromoteActive` lands, so the controller's first `drain_commands` iteration sees the seeded latch.
- **`shared_cmd_tx` lifecycle.** Installed in `execute_task` after spawn (`main.rs:853`) and cleared in both task-end paths: e-stop drop (`main.rs:1473`) and normal completion (`main.rs:1535`). Mirror site of `shared_copper_state.store(None)` precisely as the worker-level slot description requires.
- **Crate hygiene.** `roz-worker/src/lib.rs` keeps the existing crate-level `#![allow(clippy::pedantic, clippy::nursery, ...)]` (acknowledged tech-debt) and `roz-copper/src/handle.rs` carries no `unsafe_code`. Logging is `tracing::info!` / `warn!` / `error!` throughout the new code; no `println!` / `eprintln!` regressions in production paths.
- **Test surface.** `manipulator_dispatch_path.rs` exercises `promote_controller.execute()` end-to-end (Codex H4 production-parity gate, NOT `#[ignore]`-gated under `--features test-fixtures`). `fw05_estop_ack_subscriber.rs` covers WAL-authoritative boot, signed AckEstop dispatch, unsigned-rejection-with-audit, and Run→Latched WAL persistence — all default-runnable. ## Warnings

### WR-01: `try_send` silent-drop on `latch_persist_tx` violates the plan's IEC 60204-1 fail-closed posture

**Files:**
- `crates/roz-copper/src/controller.rs:952`
- `crates/roz-copper/src/controller.rs:978`
- `crates/roz-copper/src/controller.rs:2344`
- `crates/roz-copper/src/controller.rs:2538`
- `crates/roz-copper/src/controller.rs:2655`
- `crates/roz-copper/src/controller.rs:2744`

**Issue:** All six sites that publish a `LatchState` transition to the WAL persistence channel use `let _ = tx.try_send(new_state);`. On a full bounded `SyncSender(16)` the transition is dropped silently with no log. In steady state the drainer (sqlite, ms-scale) is faster than the producer, so this should never fire — but the same plan ships `main.rs:776-786` with an explicit "refusing to silently boot to `Run` per IEC 60204-1" error path. Applying the discriminating constraint consistently means a dropped persist must be observable: a subsequent worker restart could observe a stale WAL row and rearm without operator acknowledgment, which is the exact hazard the load-side code already guards against. Note that `main.rs:797` describes the design as "non-blocking — `SendError` is dropped silently so a full channel cannot stall the controller thread" — the goal is to keep the controller thread non-blocking, which `try_send` does even with logging on the error arm (the `tracing::error!` macro is itself non-blocking on the hot path).

**Fix:** Replace each `let _ = tx.try_send(new_state);` with an `Err`-arm trace at error level. Suggested helper to avoid six copies:

```rust
fn try_persist_latch(
    tx: &std::sync::mpsc::SyncSender<crate::latch::LatchState>,
    new_state: crate::latch::LatchState,
) {
    if let Err(e) = tx.try_send(new_state) {
        tracing::error!(
            error = %e,
            ?new_state,
            "FW-05: latch persist channel full or disconnected — transition NOT durable; \
             worker restart may observe stale WAL state (IEC 60204-1 audit signal)"
        );
    }
}
```

Then call sites become `if let Some(tx) = latch_persist_tx { try_persist_latch(tx, new_state); }`.

## Info

### IN-01: `JoinHandle` returns from `spawn_estop_ack_subscriber` / `spawn_safety_resume_subscriber` are dropped immediately at the call site

**File:** `crates/roz-worker/src/main.rs:2342`, `crates/roz-worker/src/main.rs:2350`

**Issue:** Both `spawn_*_subscriber` functions return `tokio::task::JoinHandle<()>`. The production call sites do not bind the return value (no `let _ack_join = ...`). Lifetime is bounded correctly via `phase24_cancel.cancelled()` inside each subscriber loop, so this is functionally correct and the spawned tasks are not leaked. Because `tokio::task::JoinHandle` is `#[must_use]`, this would normally be a clippy `let_underscore_must_use` warning — the only reason it does not surface in CI is the crate-level `#![allow(clippy::pedantic, clippy::nursery, ...)]` already declared at `roz-worker/src/lib.rs:6-12`. The integration test at `tests/fw05_estop_ack_subscriber.rs` does bind the return as `let _ack_join = spawn_estop_ack_subscriber(...)`.

**Fix:** Mirror the test-side pattern in `main.rs` for consistency, and so the production-wiring intent is grep-discoverable:

```rust
let _estop_ack_join = roz_worker::safety_subscribers::spawn_estop_ack_subscriber(
    nats.clone(),
    (**ctx_shared).clone(),
    config.worker_id.clone(),
    safety_host_id.clone(),
    shared_cmd_tx.clone(),
    phase24_cancel.clone(),
);
let _safety_resume_join = roz_worker::safety_subscribers::spawn_safety_resume_subscriber(
    nats.clone(),
    (**ctx_shared).clone(),
    config.worker_id.clone(),
    safety_host_id,
    shared_cmd_tx.clone(),
    phase24_cancel.clone(),
);
```

This is purely cosmetic — no behavior change.

### IN-02: Unsigned audit publish on `safety.signature_failure.{host_id}` follows the existing server-side precedent

**File:** `crates/roz-worker/src/safety_subscribers.rs:190`

**Issue:** `handle_safety_message` publishes the audit via plain `nats.publish(audit_subject, audit_payload.into())` rather than `roz_nats::dispatch::publish_signed`. At first read this seems asymmetric with the 1 Hz health heartbeat at `main.rs:2453` which DOES sign every emit. Cross-check against `crates/roz-server/src/signing_gate.rs:417-425` confirms the server-side counterpart also publishes both `safety_signature_failure_worker` and `safety_signature_failure_server` audits via plain `nats.publish` — the unsigned-audit pattern is the existing convention for signature-failure topics (avoids re-engaging the signing path that just rejected the inbound message). No fix needed; documenting only so a future reader does not flag it.

**Fix:** None.

# Phase 26.10 Plan 05: FW-04 — Edge AUTO Placement Defaults Edge for Controller-Capable Hosts — Summary

**`resolve_placement` now considers host controller-capability: AUTO routes to edge when the session has both a `host_id` and the host's worker has been registered as controller-capable, otherwise stays cloud. Pitfall 5 regression covered. Codex M2 Option B fix sidesteps the session-start tool-registration timing issue.**

## Accomplishments

- **FW-04 closed at the function level.** AUTO no longer defaults cloud unconditionally; the new third arg `controller_capable_worker: bool` lets the call site signal whether the host's worker is controller-capable. AUTO routes edge iff both flags are true.
- **Pitfall 5 regression test landed.** `resolve_placement_auto_host_no_capability_cloud` asserts cloud routing when has_host=true but controller_capable_worker=false — the exact "host alone is not enough" behavior FW-04 demands.
- **Codex M2 fix applied.** Capability-based check at the call site sidesteps the session-start tool-registration timing issue (server-owned MCP tools register at `:2624`, AFTER the placement decision at `:2583`).
- **No proto changes.** The capability signal flows through Rust types and a Postgres lookup; no wire surface touched.

## Files Modified

- `crates/roz-core/src/edge/mod.rs` — Replaced `pub const fn resolve_placement(placement: i32, _has_host: bool) -> bool` with `pub const fn resolve_placement(placement: i32, has_host: bool, controller_capable_worker: bool) -> bool`. Replaced 4 existing tests with 7 new tests covering all 6 spec branches (EDGE×2, CLOUD×2, AUTO×4) plus an unknown-placement case.
- `crates/roz-server/src/grpc/agent.rs` — Replaced the 2-arg call at line 2583 with a 3-arg call. Added a 17-line block above the call that resolves `controller_capable_worker` by parsing `start.host_id` as a UUID, fetching the host row via `roz_db::hosts::get_by_id(pool, host_uuid)`, and checking `!h.capabilities.is_empty()`. Returns `false` (cloud-default) for unparseable host_id, missing host, or empty capabilities.

## Decisions Made

- **Codex M2 Option B (capability lookup) over Option A (tool-union).** Plan 05 was a REVIEWS-driven revision: the original tool-union approach required session-start-after-MCP-registration timing, which is a larger surgery in `agent.rs`. Capability lookup matches the existing host-fetch pattern at `:2570s` (where `resolve_worker_id` already uses `roz_db::hosts::get_by_id`).
- **Use `roz_hosts.capabilities: Vec<String>` non-empty as the controller-capable proxy.** The plan's stated signal (`RobotCapabilities.joints` non-empty) presumes a JSONB capability column on `roz_hosts`. **No such column exists on `roz_hosts` today** — `RobotCapabilities` is published over NATS (`caps_subject` at `worker/main.rs:1486`) but never persisted server-side. The closest persisted signal is the `Vec<String>` operator-curated tag list. This is the documented proxy until Plan 09 establishes typed descriptor persistence.
- **Single `feat:` commit.** TDD plan-level cycle: the new fn signature breaks all existing call sites, so the test file's `super::*` import would not compile under the OLD signature. RED and GREEN are inseparable. Single commit precedent from Plan 02 / Plan 03 applies (advisor-confirmed).

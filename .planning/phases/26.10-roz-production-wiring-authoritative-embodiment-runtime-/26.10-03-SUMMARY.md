# Phase 26.10 Plan 03: FW-02 IO Factory + Real Copper Spawn Wiring — Summary

**`IoFactory` trait surface in `roz-copper` plus worker-side `factory_for` dispatch and `CopperHandle::spawn_with_policy_and_io` so OodaReAct task spawn binds real `Arc<dyn ActuatorSink>` and `Box<dyn SensorSource>` keyed on `EmbodimentRuntime.model.embodiment_family.family_id` — without introducing a `roz-copper -> roz-mavlink` cargo cycle.**

## Accomplishments

- **FW-02 IO wiring is closed.** OodaReAct task spawn no longer no-ops on the IO surface; it goes through a worker-side factory keyed on the embodiment family and binds real (or, for the deferred backends, loud-fail-closed) `ActuatorSink`/`SensorSource` impls into Copper.
- **Cargo cycle prevented architecturally and verified by test.** Trait surface in `roz-copper`, concrete dispatch in `roz-worker`. A compile-time `#[test]` reads `roz-copper/Cargo.toml` and asserts `roz-mavlink` is absent — future PRs that try to add the reverse edge will fail CI loudly.
- **Worker boot path fails closed on every prerequisite.** Missing `embodiment_runtime`, missing `control_interface_manifest`, unregistered family, and `IoFactory.build()` Err each publish a `TaskStatusEvent::failed` and signal Restate before returning — no silent no-op.
- **Plan 08 hand-off is clean.** `ManipulatorStubFactory` reachable through `factory_for(Some("manipulator"))` returns `Err` — Plan 08 replaces the impl in-place without touching call sites.

## Decisions Made

- **Trait in copper, dispatch in worker (Codex H2).** Confirmed `crates/roz-mavlink/Cargo.toml:38` already declares `roz-copper = { path = "../roz-copper" }`. Adding the inverse edge would cycle. Worker is the natural home: it already depends on both crates and is the OodaReAct boot path that consumes factories.
- **Cycle-prevention compile-time guard.** A `#[test] roz_copper_does_not_depend_on_roz_mavlink` reads `include_str!("../Cargo.toml")` and asserts the forbidden token is absent. CI fails the build if a future PR reintroduces the edge.
- **Reuse the existing fail-closed pattern.** `execute_task` returns `()`, so `?` is not available. The plan template's `anyhow::anyhow!(...)?` would not compile. Routed through a closure (`copper_spawn_failure`) that mirrors the existing model-build error path at line 646: `tracing::error` → `publish_task_status(failed)` → `signal_result(Restate)` → `return`.
- **`let-else` over `match Some/None`.** clippy fires `clippy::single_match_else` when destructuring a single Some variant via match. All 3 missing-prerequisite gates use `let Some(...) = expr else { ...; return; };`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `Result::unwrap_err()` requires Ok variant to be `Debug`**

- **Found during:** Task 2 (test compilation)
- **Issue:** Tests called `result.unwrap_err()` on `Result<(Arc<dyn ActuatorSink>, Box<dyn SensorSource>), anyhow::Error>`. `Box<dyn SensorSource>` does not implement `Debug`, so `unwrap_err()` failed to compile (E0277 four times).
- **Fix:** Replaced both `unwrap_err()` calls with `match result { Ok(_) => panic!("expected Err"), Err(e) => e }`. Functionally identical, no `Debug` requirement on the Ok variant.
- **Files modified:** `crates/roz-worker/src/io_backends.rs`
- **Commit:** `1c287d0` (Task 2 commit)

**2. [Rule 1 — Bug] `clippy::single_match_else` fires on the new fail-closed gates**

- **Found during:** Task 3 (`cargo clippy -p roz-copper -p roz-worker -- -D warnings`)
- **Issue:** The plan template used `match invocation.embodiment_runtime.as_ref() { Some(r) => r, None => { ...; return; } }` for each of the 3 missing-prerequisite gates. clippy's `single_match_else` flagged all 3 (rust 1.92.0 lint set) as compile errors under `-D warnings`.
- **Fix:** Replaced each match with `let Some(r) = invocation.embodiment_runtime.as_ref() else { ...; return; };`. Same control flow, satisfies the lint.
- **Files modified:** `crates/roz-worker/src/main.rs`
- **Commit:** `aa32796` (Task 3 commit)

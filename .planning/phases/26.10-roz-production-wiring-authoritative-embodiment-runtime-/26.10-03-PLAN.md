## Reviews Addressed

This plan is a REVIEWS-driven revision. Codex (gpt-5.5) findings driving the revision:

- **H2 (HIGH) — Cargo cycle risk.** Original plan put the factory dispatch table + `MavlinkFactory` registration in `roz-copper`, requiring `roz-copper → roz-mavlink`. Confirmed via `crates/roz-mavlink/Cargo.toml` that `roz-mavlink` already depends on `roz-copper` (path `../roz-copper`) for trait imports. The inverse edge would create a cargo cycle. **FIX:** Keep `IoFactory` *trait* in `roz-copper` (where the trait surface belongs); move concrete factory registry + impls to `roz-worker` (a new `crates/roz-worker/src/io_backends.rs` module). `roz-worker` already depends on both `roz-copper` and (transitively or directly) on `roz-mavlink`-compatible code; placing factories there is the lighter option than a new aggregation crate.
- **H2 — Choice rationale.** Worker is the natural home because (a) `roz-worker` is the boot path that consumes factories at task spawn time, (b) it already imports both `roz-copper` and the future `roz-mavlink` driver/wrapper crates without any cycle issues, (c) avoids workspace ceremony of creating a new `roz-io-backends` crate for one trait impl. Per advisor recommendation: `roz-worker` placement is lighter weight.
- **G1 — `EmbodimentRuntime` and `ControlInterfaceManifest` lack `Default`** (verified). Original plan's test-step using `Default::default()` would fail to compile. **FIX:** Tests construct via `EmbodimentRuntime::compile(...)` and explicit `ControlInterfaceManifest` builder; for the deferred-Err MAVLink test, runtime contents do not matter — minimal explicit literals suffice.

## User Constraints (from CONTEXT.md)

### Locked Decisions

> Copied verbatim from `26.10-CONTEXT.md`.

**LOCKED — Authoritative `EmbodimentRuntime` on Dispatch (Blocking #3):**
- Task dispatch and controller promotion MUST carry (or resolve to) an authoritative `EmbodimentRuntime` — not just a control manifest.
- Authoritative fields: joint limits, workspace, TCPs, channel authority, sensor bindings, capabilities — derived from the embodiment manifest, not synthesized.
- Resolution path: server resolves embodiment by `host_id` / `embodiment_id` at dispatch time, attaches signed runtime descriptor to the task envelope, worker passes it to controller promotion.
- Controller promotion at `crates/roz-copper/src/controller.rs:553` MUST reject promotion if the runtime descriptor is missing or fails authority checks.

**LOCKED — Worker Copper Real Actuator/Sensor IO (Blocking #2):**
- `spawn_with_policy` MUST bind a real `ActuatorSink` and `SensorSource` keyed by the embodiment's channel manifest — no more no-op spawns.
- Bindings are factory-driven from the resolved `EmbodimentRuntime` (see decision above) so adding a new robot family means adding an IO factory, not patching `roz-worker`.
- Deterministic fake backends MUST exist for every IO trait so framework tests run without hardware.
- Real backends (Dynamixel/OpenCR fixtures, MAVLink, etc.) plug in through the same factory surface.

**LOCKED — Controller Lifecycle Tools at Agent Layer (Blocking #1):**
- Worker MUST register at minimum: `promote_controller`, `stop_controller`, `controller_status`.
- Local runtime MUST register **live promotion** alongside replay (replay stays as a separate, explicitly-named tool).
- Tool category MUST mark these as physical-execution / safety-relevant so the OodaReAct loop applies the right approval and safety policies.
- Tool dispatch MUST go through the same trait surface so `crates/roz-local` and `crates/roz-worker` agree on contract shape.

**LOCKED — Edge `AUTO` Defaults Edge for Control Loops (High #4):**
- `AUTO` placement MUST resolve to **edge** for robot-control tools (controller lifecycle, channel writes, real-time safety) regardless of cloud-default policy elsewhere.
- Cloud is permitted only for non-control reasoning (planning, perception offload, model calls).
- The decision is per-tool category, not per-task, so a single agent turn can correctly mix edge control with cloud reasoning.

**LOCKED — Safety Hardening Before Hardware (High #5):**
1. Stale-heartbeat timer fires after `T_stale` regardless of whether another heartbeat ever arrives (`crates/roz-safety/src/main.rs:53`).
2. Bounded WASM host output validates output bounds **before** allocation (`crates/roz-copper/src/wit_host.rs:350`).
3. Latched e-stop with ack/zero/disable — no auto-rearm after a safety rollback (`crates/roz-copper/src/controller.rs:1111`).

**LOCKED — reference manipulator-Class Modeling Fidelity (High #6):**
- Manifest parsing in `crates/roz-core/src/manifest.rs:528` MUST preserve joints, TCPs, grippers, sensors, workspaces — no flattening into generic channels at parse time.
- Capability publishing in `crates/roz-worker/src/main.rs:1394` MUST publish embodiment-specific capability descriptors derived from the manifest, not generic empty capabilities.
- `crates/roz-core/src/embodiment/model.rs:54` MUST be the single source of truth for the runtime view that controllers see.

**LOCKED — framework-fidelity hardware bench (Medium #7):**
- A deterministic fake manipulator-class backend MUST exist (joints, gripper, encoders, current feedback) so CI exercises the manipulator path without hardware.
- The live-matrix script (`scripts/run_live_e2e_matrix.sh`) MUST add a manipulator row that runs the fake-backend tests deterministically (same posture as PX4/ArduPilot rows).
- Gated hardware bench tests (skippable, opt-in by env) MUST cover: tiny bounded motion, encoder/current feedback, channel order/sign/unit checks, physical e-stop latency.
- Serial/Dynamixel/OpenCR fixtures live behind a feature flag and are exercised when the relevant env is present.

## Summary

Roz is a real-world manipulator/drone platform with a clean trait surface in `crates/roz-copper/src/io.rs`, a Phase 23 signed-dispatch primitive, and a fully working MAVLink native backend (`crates/roz-mavlink/src/backend.rs`) that already implements `SensorSource + ActuatorSink + DiscreteCommandSink<FlightCommand>`. The codex review found that the *agent/task production path* (server dispatch → worker → Copper) skips three of the four wires that make the trait surface meaningful: it never carries an authoritative `EmbodimentRuntime` end-to-end, it spawns Copper with `actuator=None / sensor=None`, and the agent layer cannot promote, stop, or query a controller.

This phase does the wiring. None of the seven LOCKED decisions in CONTEXT requires a new primitive — every one extends something already scaffolded. `TaskInvocation` is a Rust struct serialised as JSON over NATS (`crates/roz-nats/src/dispatch.rs:97`), so adding `Option<EmbodimentRuntime>` is non-breaking and rides inside the Phase 23 signed envelope automatically (the signature covers `payload_hash`). The `IoFactory` layer is essentially "do for reference manipulators what `MavlinkBackend::new_serial` already does for drones." The latched-e-stop fix is a small state machine grounded in `IEC 60204-1` Stop Category 0/1 and `EN ISO 13849-1:2023` manual-reset semantics. The hardware bench strategy mirrors the existing live-matrix posture (deterministic-by-default, `--ignored` + `ROZ_*` env-gated for real hardware) and finally adds the manipulator row to `scripts/run_live_e2e_matrix.sh`.

**Primary recommendation:** Treat this phase as four threadbare integration vectors that all flow through the same control point — `ControllerCommand::LoadArtifact(_, _, control_manifest, embodiment_runtime: Some(_))`. Wire (a) the runtime into `TaskInvocation`, (b) the IO factory into `CopperHandle::spawn_with_policy`, (c) the lifecycle tools into the dispatcher, (d) the latched-e-stop state machine + bounded WIT IO + heartbeat-timer into safety. Validate the seam with a fake manipulator backend exercising the production code path.

### Pattern 1: `EmbodimentRuntime` rides inline on `TaskInvocation`

**What:** Add `embodiment_runtime: Option<EmbodimentRuntime>` to `TaskInvocation`.

**When to use:** Every OodaReAct dispatch (production path). React-only invocations may leave it `None` since they don't promote controllers.

**Why inline (not sidecar):**
- `TaskInvocation` is JSON-serialised over NATS [VERIFIED: `crates/roz-nats/src/dispatch.rs:97`].
- Phase 23 signing covers `payload_hash` — adding fields automatically widens the signed surface, no new signing primitives needed [VERIFIED: `crates/roz-server/src/signing_gate.rs:50` references `payload_sha256_hex`].
- `#[serde(default, skip_serializing_if = "Option::is_none")]` makes the addition wire-compatible with older workers (which would simply not promote a controller without a runtime — already enforced by `controller.rs:553`).

**Example (planner-bound):**
```rust
// crates/roz-nats/src/dispatch.rs (additive)
pub struct TaskInvocation {
    // ...existing fields...
    /// Phase 26.10 FW-01: authoritative embodiment runtime resolved by
    /// the server at dispatch time. Required for OodaReAct controller
    /// promotion; controller.rs:553 rejects load without it.
    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub embodiment_runtime: Option<roz_core::embodiment::EmbodimentRuntime>,
}
```

### Pattern 2: `IoFactory` keyed on `embodiment_family`

**What:** A trait with a single `build` method returning `(Arc<dyn ActuatorSink>, Box<dyn SensorSource>)` plus optional `Box<dyn DiscreteCommandSink<Cmd>>`. Backends register via explicit `match` in a registry function.

**When to use:** At task-spawn time inside `roz-worker::execute_task`, AFTER `EmbodimentRuntime` is verified.

**Why explicit-match (not `inventory`):** Workspace today has 2 backends (MAVLink, fake manipulator); a static dispatch table fits in one file change per backend. `inventory` adds a build-time crate and indirection that isn't earning its keep.

**CRITICAL: shared backend instance.** Both trait objects MUST wrap the same backend instance — the actuator path uses `Arc::clone(&backend) as Arc<dyn ActuatorSink>`, the sensor path uses `Box::new(backend.sensor_view())` where `sensor_view()` returns a thin wrapper sharing the same internals via `Arc<Mutex<_>>`. Mirror MavlinkBackend's pattern: `MavlinkBackend` impls `SensorSource + ActuatorSink + DiscreteCommandSink<FlightCommand>` on the SAME struct (`crates/roz-mavlink/src/backend.rs:526,535,587`) and shares state through `Arc<Mutex<ReadinessBuilder>>` and `mpsc::Sender<MavMessage>` clones. A factory that returned two unrelated objects would lose the shared queue and break round-trip semantics.

**Example (planner-bound):**
```rust
// crates/roz-copper/src/io_factory.rs  (NEW)
pub trait IoFactory: Send + Sync {
    /// Build an actuator + sensor pair for this backend.
    /// Both trait objects MUST share the same backend instance — see module docs.
    fn build(
        &self,
        runtime: &EmbodimentRuntime,
        control_manifest: &ControlInterfaceManifest,
    ) -> anyhow::Result<(Arc<dyn ActuatorSink>, Box<dyn SensorSource>)>;

    fn name(&self) -> &'static str;
}

/// Static dispatch table. Adding a new backend is one match arm + one impl block.
pub fn factory_for(family: Option<&str>) -> Option<Box<dyn IoFactory>> {
    match family {
        Some("manipulator") | Some("manipulator") => Some(Box::new(FakeManipulatorFactory)),
        Some("drone-mavlink") => Some(Box::new(MavlinkFactory)),
        _ => None,
    }
}
```

### Pattern 4: Latched e-stop state machine (IEC 60204-1 + EN ISO 13849-1)

**What:** Replace the all-defaults bypass at `controller.rs:1111` with a four-state latch.

**When to use:** Inside the controller tick when `force_estop` is asserted.

**Why this state machine:** Standards-grounded — `IEC 60204-1` mandates emergency-stop is Stop Category 0 (uncontrolled, immediate power removal) or 1 (controlled with power removed at end), explicitly excludes Category 2 [CITED: machinerysafety101.com / Schneider FAQ FA225420]. `EN ISO 13849-1:2023` lists Manual Reset Function as a safety function: re-establishing the safety function by resetting *cancels* the stop command, but cancellation requires "manual separate and intended action" — auto-rearm is forbidden when the risk assessment indicates whole-body access (manipulators do) [CITED: gt-engineering.it/.../is-manual-reset-a-safety-function/, ISO 13849-1:DIS:ed-4].

**State machine:**

```
States:
  Run         — normal tick; controller writes commands
  Latched     — force_estop asserted; commands clamped to ZERO and PUBLISHED
                (do NOT silently default — must emit explicit zero frame
                so downstream actuator sees commanded zero, not "no message")
  AwaitingAck — operator must send explicit ack via NATS subject
                safety.estop_ack.{worker_id} (signed per Phase 23)
  ZeroVerified— after ack, require N consecutive ticks of zero output AND
                sensor frames showing motion below threshold before exit

Transitions:
  Run         → Latched      : force_estop asserted (any source)
  Latched     → Latched      : default (sticky) — emit zero frame
  Latched     → AwaitingAck  : ack_pending flag set by ESTOP_ACK NATS msg
  AwaitingAck → ZeroVerified : N consecutive zero ticks observed (N=10 @ 100 Hz = 100 ms)
  ZeroVerified→ Run          : explicit Resume command from operator (signed)
  Any         → Latched      : safety_filter, watchdog, or stale-heartbeat e-stop

Forbidden transitions:
  Latched     ↛ Run          (no auto-rearm — IEC 60204-1)
  Latched     ↛ ZeroVerified (must pass through AwaitingAck)
```

**Concrete fix at `controller.rs:1111`:** The current code returns `default()` when `all_default` is true. The bug is that `all_default` is treated as "controller is fine" — but during a latched e-stop, the controller MUST emit an *explicit zero command frame*, not skip emission. Skipping emission means the actuator continues with its last command (drift). The replacement is:

1. Hoist the latch state into the controller (a new `LatchState` enum next to `running`).
2. While `LatchState != Run`, ALWAYS emit the explicit zero frame and ignore the all-default short-circuit.
3. Only advance `LatchState` on signed-NATS `ESTOP_ACK` and on observed zero motion.

**Standards mapping (Stop Category):** This is a **Stop Category 0** in roz-copper terms — power-removal-equivalent for a software actuator means "explicit zero frame on every tick." A future `roz-safety` extension could register Stop Category 1 (ramp-then-stop) for non-emergency safety stops; CONTEXT scopes this phase to the latched e-stop only.

### Pattern 5: Bounded WIT host `set_output`

**What:** Validate `len` against a const cap BEFORE `vec![0u8; len as usize]`.

**Why:** `wit_host.rs:350` allocates `len` bytes first, then attempts to read. A malicious WASM controller calling `set_output(ptr, 1_000_000_000)` triggers a 1GB allocation. The WASM is sandboxed but the host process is not — heap exhaustion is a denial-of-service against the worker.

**Recommended bound:** `MAX_TICK_OUTPUT_BYTES = 64 * 1024` (64 KiB) — generous given that `TickOutput` JSON is bounded by `command_values.len() * 32` and live controllers have at most ~50 channels. Make it const on `HostContext`.

**Example (planner-bound):**

### Pattern 6: Stale-heartbeat timer in `roz-safety`

**What:** Replace the heartbeat-driven stale check with a `tokio::time::interval` ticker.

**Why:** `roz-safety/src/main.rs:53` only calls `tracker.stale_workers()` *after* receiving a new heartbeat. ### Anti-Patterns to Avoid

- **Don't synthesize embodiment runtimes from `robot.toml`.** `crates/roz-core/src/manifest.rs:540-555` builds `JointSafetyLimits { max_velocity: f64::INFINITY, ... }` and tags the runtime with `SYNTHETIC_EMBODIMENT_RUNTIME_ISSUE`. Promotion already rejects synthetic runtimes [VERIFIED: `crates/roz-local/src/tools/promote_controller.rs:192-201`]; the bug is upstream — manifests must declare real limits or the dispatch path fails fast.
- **Don't run real-hardware code in the deterministic CI matrix.** Gate Dynamixel/OpenCR fixtures behind `ROZ_MANIPULATOR_HARDWARE=1` (mirror `ROZ_SKIP_MANIPULATOR_LIVE_TEST` posture).
- **Don't add a new ControllerCommand variant if `Halt` already covers `stop_controller`.** [VERIFIED: `crates/roz-copper/src/channels.rs:58`].
- **Don't auto-rearm after e-stop rollback.** Forbidden by IEC 60204-1.
- **Don't assume `len > MAX` cannot happen.** A buggy WASM controller emitting `set_output(_, 1_000_000)` is the canonical case.
- **Don't depend on `inventory` for backend registration in v3.0.** Two backends today; revisit at four.
- **Don't return two unrelated trait objects from `IoFactory::build`.** Both must wrap the same instance — see Pattern 2 critical caveat.

## Common Pitfalls

### Pitfall 1: Embodiment runtime cache stalls during host re-registration

**What goes wrong:** A worker re-registers (host-id rotation, e.g. after firmware flash). The server's embodiment cache returns the *prior* runtime; the dispatched task has joint limits that don't match the new hardware. Controller load passes (digests match the prior runtime) but the actuator sees out-of-range commands.

**Why it happens:** Resolution caches are tempting for cold-path latency, but embodiment changes are rare *and high-impact*.

**How to avoid:**
- Don't cache `EmbodimentRuntime` at the server. The Postgres lookup is single-digit ms; CONTEXT-locked decision says "cold-path latency budget vs. correctness on rotation" is explicit.
- If caching is needed for fan-out (1k workers / sec dispatch), invalidate on `roz_hosts` UPDATE via NATS (similar to FS-01's `roz.policy.{worker_id}` push).

**Warning signs:** Discrepancy between `EmbodimentRuntime.model_digest` in the dispatch envelope and the worker's locally-known digest after re-registration.

### Pitfall 2: Fake-reference manipulator backend hides a real control bug

**What goes wrong:** The fake backend simulates "ideal" Dynamixel behavior — instant response, no friction, perfect encoder. A controller that works on the fake hangs the real arm because it commands aggressive velocities the actual servo can't track.

**Why it happens:** Fake backends are tempting to make "convenient" rather than "representative."

**How to avoid:**
- Model first-order encoder lag (~5 ms) and saturation (per-joint max-velocity from `EmbodimentRuntime.joints[i].limits.max_velocity`).
- Inject quantization (encoder ticks ≠ continuous floats).
- Add a "current feedback" channel that returns a current proportional to commanded acceleration — controllers reading current must see something realistic.
- Document explicitly what the fake does NOT model (no compliance, no physical e-stop button, no thermal limits).

**Warning signs:** Tests pass on fake but the real-hardware hardware bench row (gated, opt-in) reveals systematic overcommand.

### Pitfall 3: WASM heap exhaustion via `get_input` not `set_output`

**What goes wrong:** The codex flagged `set_output` (line 350). The mirror function `get_input` (line 322) is *currently* safe — it checks `needed > buf_len as usize`. But a future contributor copying the `set_output` shape into a new host fn may forget the bound.

**Why it happens:** Convention-by-example only — no shared helper.

**How to avoid:** Introduce a small `bounded_read_buf(len, max) -> Result<Vec<u8>>` helper used by every host fn that allocates from a WASM-supplied length.

### Pitfall 4: Latched e-stop ack arrives unsigned

**What goes wrong:** Operator sends `safety.estop_ack.{worker_id}` from a debug shell. Worker rearms. Operator was wrong about the cause; the arm slams into a fixture.

**Why it happens:** Phase 23 signing covers task dispatch but the e-stop ack is a separate subject.

**How to avoid:** ALL ack messages MUST be signed by the server with `roz-sig-v1` (Phase 23 envelope). Worker rejects unsigned acks even in `Audit` mode for safety-critical subjects. Add `safety.estop_ack.*` and `safety.resume.*` to the strict-signing allowlist.

### Pitfall 5: Edge AUTO regression on existing cloud-only sessions

**What goes wrong:** A pre-existing cloud session with `Pure`-only tools and a `host_id` (e.g. for memory queries) suddenly routes edge after the fix.

**Why it happens:** The simplistic AUTO check `has_host && physical_tools` triggers on `has_host` alone if any `Physical` tool slips into the registered surface (e.g. via skill bootstrap).

**How to avoid:** The session-start scan must require AT LEAST ONE registered `ToolCategory::Physical` tool; the absence of any physical tools means AUTO=cloud regardless of `has_host`. Add a regression test that constructs a session with `host_id=Some(_)` and only `Pure` tools and asserts cloud placement.

### Pitfall 6: Manipulator manifest fidelity passes locally but capability publish drops fields

**What goes wrong:** `EmbodimentRuntime` is correct end-to-end on the worker, but `RobotCapabilities` advertised on `caps.{worker_id}` flattens it back to `joints: vec![]`. substrate-ide renders an empty manipulator.

**Why it happens:** Capability publishing is a separate code path at `crates/roz-worker/src/main.rs:1394` — currently hand-built from `config.max_velocity` only.

**How to avoid:** Replace the hand-built `RobotCapabilities` with a projection from the resolved `EmbodimentRuntime` (joint names + limits, TCP names + types, sensor mounts → `cameras`/`sensors` arrays). Add a regression test asserting projection lossless-ness on the UR5 fixture manifest.

### Example 6: Fake-reference manipulator backend skeleton (FW-07)

```rust
// crates/roz-test/src/fake_manipulator.rs  (NEW)
//
// NOTE: uses std::sync::Mutex (workspace already uses it pervasively).
// Switch to parking_lot::Mutex only if benchmarks demand it.

use std::sync::{Arc, Mutex};
use std::sync::atomic::{AtomicI64, Ordering};
use roz_copper::io::{ActuatorSink, SensorFrame, SensorSource};
use roz_core::command::CommandFrame;
use roz_core::embodiment::EmbodimentRuntime;

/// Inner state shared between the actuator and sensor halves.
struct FakeManipulatorState {
    joint_positions: Vec<f64>,
    joint_commands: Vec<f64>,
    joint_max_velocity: Vec<f64>,
    joint_position_bounds: Vec<(f64, f64)>,
    sim_time_ns: AtomicI64,
    command_count: usize,
}

#[derive(Clone)]
pub struct FakeManipulatorActuator(Arc<Mutex<FakeManipulatorState>>);

pub struct FakeManipulatorSensor(Arc<Mutex<FakeManipulatorState>>);

pub fn fake_manipulator_pair(runtime: &EmbodimentRuntime) -> (FakeManipulatorActuator, FakeManipulatorSensor) {
    let n = runtime.model.joints.len();
    let state = FakeManipulatorState {
        joint_positions: vec![0.0; n],
        joint_commands: vec![0.0; n],
        joint_max_velocity: runtime.model.joints.iter()
            .map(|j| j.limits.max_velocity)
            .collect(),
        joint_position_bounds: runtime.model.joints.iter()
            .map(|j| (j.limits.position_min, j.limits.position_max))
            .collect(),
        sim_time_ns: AtomicI64::new(0),
        command_count: n,
    };
    let shared = Arc::new(Mutex::new(state));
    (FakeManipulatorActuator(shared.clone()), FakeManipulatorSensor(shared))
}

impl ActuatorSink for FakeManipulatorActuator {
    fn send(&self, frame: &CommandFrame) -> anyhow::Result<()> {
        let mut state = self.0.lock().expect("fake-manipulator mutex poisoned");
        for (i, &v) in frame.values.iter().enumerate().take(state.command_count) {
            let max = state.joint_max_velocity[i];
            state.joint_commands[i] = v.clamp(-max, max);
        }
        Ok(())
    }
}

impl SensorSource for FakeManipulatorSensor {
    fn try_recv(&mut self) -> Option<SensorFrame> {
        const TICK_PERIOD_NS: i64 = 10_000_000;  // 10 ms
        let mut state = self.0.lock().expect("fake-manipulator mutex poisoned");
        let now = state.sim_time_ns.fetch_add(TICK_PERIOD_NS, Ordering::Relaxed) + TICK_PERIOD_NS;
        let dt = (TICK_PERIOD_NS as f64) / 1e9;

        // Integrate commanded velocity into position; clamp to bounds.
        let cmds = state.joint_commands.clone();
        for (i, v) in cmds.iter().enumerate() {
            let new_pos = state.joint_positions[i] + v * dt;
            let (lo, hi) = state.joint_position_bounds[i];
            state.joint_positions[i] = new_pos.clamp(lo, hi);
        }
        Some(SensorFrame {
            joint_positions: state.joint_positions.clone(),
            joint_velocities: cmds,
            sim_time_ns: now,
            ..Default::default()
        })
    }
}
```

This skeleton is the seed for FW-07's deterministic backend. The actuator + sensor share state via `Arc<Mutex<FakeManipulatorState>>` — mirror of the `MavlinkBackend` pattern. Variants can be added for: gripper (binary state + force), encoder noise (Gaussian or quantization), current feedback (proportional to acceleration), thermal (drift in max-velocity over commanded duty-cycle).

1. **Where does `EmbodimentRuntime` actually live in storage today?**
   - What we know: `crates/roz-core/src/manifest.rs` builds it from `robot.toml`; `crates/roz-local` uses it via the local manifest path.
   - What's unclear: Whether the *server* has any current path to load an `EmbodimentRuntime` from a database row, or whether the only load source is `robot.toml`.    - What's unclear: Whether `promote_controller`/`stop_controller` deserve a new `ControllerLifecycle` category, or whether `Physical` is correct.

   - **RESOLVED**: No new variant. Plan 04 registers `promote_controller`/`stop_controller`/`controller_status` as `ToolCategory::Physical`.

3. **What's the expected operator UX for issuing the e-stop ack message?**
   - What we know: CONTEXT says ack/zero/disable; ack must be explicit.
   - What's unclear: Is the ack issued via `roz-cli` subcommand (`roz device clear-failsafe` already exists for FS-01 deadman — can it be extended for e-stop ack?), via substrate-ide UI, or via a dedicated REST endpoint?

4. **Should the live-matrix manipulator row use a containerised sim or a pure in-process fake?**
   - What we know: PX4/ArduPilot/nav2 rows use Docker containers (sim images). Manipulator authored row uses Gazebo (`live_claude_wasm_gazebo`).
   - What's unclear: Whether FW-07's deterministic row also uses Gazebo (heavy, slow boot) or just an in-process fake (fast, but less realistic).

   - **RESOLVED**: In-process fake (Plan 09) for the deterministic row. The Gazebo-backed `live_claude_wasm_gazebo` is retained for authored-scenario coverage; the two paths run side-by-side in the live matrix.

5. **Does the latched-e-stop state need to survive worker restart?**
   - What we know: `force_estop` today is in-memory.
   - What's unclear: If a worker crashes mid-latch and restarts, should it return to `Run` or `Latched`?
   - Recommendation: Return to `Latched` and require an ack — fail-safe is the standards-compliant choice. Persist `LatchState` in `WalStore` (analogous to FS-03 task checkpoints). Surface during planning.
   - **RESOLVED**: Yes — persist `LatchState` in `WalStore` (FS-03 precedent). On worker restart, the latch returns to `Latched` and requires an explicit ack before re-enabling outputs. Plan 07 implements WAL persistence as part of the latched-e-stop state machine.

6. **How does `ToolCategory` reach the FW-04 session-start scan?**
   - What we know: `ToolDispatcher.category(name)` returns the registered category. `ToolSchema` (the gRPC-serialised schema) does NOT carry a category field today.
   - What's unclear: At `crates/roz-server/src/grpc/agent.rs:2583`, the active_tools list is `Vec<ToolSchema>` — derived from client `start.tools` plus server-owned MCP tools. Neither source knows the dispatcher's view (which only exists on the worker for edge sessions, or on the server for cloud sessions). Approach (a) is cleaner; approach (b) avoids a proto change.
   - Recommendation: Approach (a). Add `ToolCategory category = N;` to `agent.proto::Tool` (next free field number), default `PHYSICAL` for backward compat, and propagate through `roz_core::tools::ToolSchema`. This makes the FW-04 scan a one-line predicate. Plan 1-2 extra tasks.
   - **RESOLVED**: No proto change required for 26.10. The category data is already extracted into a `tool_categories: HashMap<String, ToolCategory>` at `crates/roz-server/src/grpc/agent.rs:2559-2570` from the existing `ToolCategoryHint` field on `agent.proto::Tool`. ### Known Threat Patterns for Roz robotics stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Malicious WASM controller exhausts host heap via `set_output(huge)` | DoS | Bounded read buffers (FW-05) |
| Tampered `EmbodimentRuntime` enables unsafe joint commands | Tampering | Phase 23 signed envelope covers `payload_hash` (FW-01 reuses) |
| Unsigned e-stop ack causes premature rearm | EoP | All ack messages MUST be Phase 23 signed; `Strict` mode for `safety.estop_ack.*` allowlist |
| Stale heartbeat → silent worker death (no e-stop) | DoS / Safety | Timer-based stale scan (FW-05) |
| Auto-rearm after safety rollback | Safety | Latched state machine forbids `Latched → Run` (FW-05; IEC 60204-1) |
| Fake-reference manipulator test backend in production | Tampering | `IoFactory` registration explicitly gated; CI assertion that fake backends are `#[cfg(any(test, feature = "test-fixtures"))]` |
| Edge AUTO downgrade routes physical control to cloud | Safety | Session-start scan requires *at least one* `Physical` tool (FW-04 regression test) |
| Embodiment runtime cache stale during host re-registration | Integrity | No-cache for v3.0 (CONTEXT-locked); push-invalidate when caching is added |
| Unbounded `EmbodimentRuntime` size in NATS message | DoS | Size-cap envelope serialization at server before publish |

**Gathered:** 2026-04-25

<framing>
## Reference Architectures

> **Roz is inspired by reference manipulator, NemoClaw, and Hermes for robotics.** These are *reference architectures* Roz models itself after — not separate robot-family integrations or hardware platforms. References to "reference manipulator-specific" fixtures (Dynamixel, OpenCR, manipulator hardware bench, joint/TCP/gripper modeling) are validating the **framework-fidelity claim** that Roz can support that class of hardware, paralleling how Phase 27 PX4 SITL validates the drone-class claim.

Do not propose splitting reference manipulator work to a separate milestone or rescoping it as out-of-scope for v3.0 (Pixhawk-focused). The framework gaps here block all robot families, including drones. hardware bench fixtures using Dynamixel / OpenCR / serial servos are framework-fidelity tests, not vendor integrations.

</framing>

<domain>
## Phase Boundary

Close the production-wiring gaps that prevent **live WASM controller deployment** through the normal agent/task path on Roz. The codex review identified 3 blocking + 3 high + 1 medium gap between Roz's claim of being an Roz agentic robotics harness and the actual production code path:

- The agent/task path cannot deploy a live WASM controller (worker tools missing, local registers replay).
- Worker-spawned Copper has no actuator or sensor IO.
- Live controller loading needs an authoritative `EmbodimentRuntime`, but task dispatch only carries a control manifest.
- `AUTO` edge placement defaults cloud instead of edge for robot-control loops.
- Safety has hardware-blocking gaps: silent stale-heartbeat detection, unbounded WASM output, force-e-stop bypass via all-defaults.
- Roz manipulator-class modeling is conceptually supported but flattened into channels by TOML/manifest plumbing, with synthesized loose limits and empty capabilities.
- Manipulator validation is sim-only; no fake manipulator fixtures, no hardware bench coverage, no e-stop latency measurements.

**This phase delivers the wiring that makes Phase 27's nightly SITL CI a meaningful validation** of the controller-deployment path it exercises, and it makes manipulator-class deployment as real as drone-class deployment.

**Out of scope:**
- Real reference manipulator vendor procurement or shipping integration code for a specific commercial arm — fixtures are deterministic and fake by default; real-hardware hardware bench is gated and optional.
- New robot-family integrations beyond what's needed to validate the framework claim.
- Anything that breaks `substrate.sim.v1` wire compatibility (per v3.0 cross-phase rule).

</domain>

<decisions>
## Implementation Decisions

### LOCKED — Authoritative `EmbodimentRuntime` on Dispatch (Blocking #3)

- Task dispatch and controller promotion MUST carry (or resolve to) an authoritative `EmbodimentRuntime` — not just a control manifest.
- Authoritative fields: joint limits, workspace, TCPs, channel authority, sensor bindings, capabilities — derived from the embodiment manifest, not synthesized.
- Resolution path: server resolves embodiment by `host_id` / `embodiment_id` at dispatch time, attaches signed runtime descriptor to the task envelope, worker passes it to controller promotion.
- Controller promotion at `crates/roz-copper/src/controller.rs:553` MUST reject promotion if the runtime descriptor is missing or fails authority checks.

### LOCKED — Worker Copper Real Actuator/Sensor IO (Blocking #2)

- `spawn_with_policy` MUST bind a real `ActuatorSink` and `SensorSource` keyed by the embodiment's channel manifest — no more no-op spawns.
- Bindings are factory-driven from the resolved `EmbodimentRuntime` (see decision above) so adding a new robot family means adding an IO factory, not patching `roz-worker`.
- Deterministic fake backends MUST exist for every IO trait so framework tests run without hardware.
- Real backends (Dynamixel/OpenCR fixtures, MAVLink, etc.) plug in through the same factory surface.

### LOCKED — Controller Lifecycle Tools at Agent Layer (Blocking #1)

- Worker MUST register at minimum: `promote_controller`, `stop_controller`, `controller_status`.
- Local runtime MUST register **live promotion** alongside replay (replay stays as a separate, explicitly-named tool).
- Tool category MUST mark these as physical-execution / safety-relevant so the OodaReAct loop applies the right approval and safety policies.
- Tool dispatch MUST go through the same trait surface so `crates/roz-local` and `crates/roz-worker` agree on contract shape.

### LOCKED — Edge `AUTO` Defaults Edge for Control Loops (High #4)

- `AUTO` placement MUST resolve to **edge** for robot-control tools (controller lifecycle, channel writes, real-time safety) regardless of cloud-default policy elsewhere.
- Cloud is permitted only for non-control reasoning (planning, perception offload, model calls).
- The decision is per-tool category, not per-task, so a single agent turn can correctly mix edge control with cloud reasoning.

### LOCKED — Safety Hardening Before Hardware (High #5)

Three concrete gaps to close, each verified by a targeted test:

1. **Stale-heartbeat timer**: `crates/roz-safety/src/main.rs:53` MUST detect missing heartbeats via timer, not via "next heartbeat received." E-stop fires after `T_stale` regardless of whether another heartbeat ever arrives.
2. **Bounded WASM host output**: `crates/roz-copper/src/wit_host.rs:350` MUST validate output bounds **before** allocation, not after, so a malicious or buggy controller cannot exhaust memory pre-validation.
3. **Latched e-stop with ack/zero/disable**: `crates/roz-copper/src/controller.rs:1111` MUST NOT accept "all default outputs" as a way past force-e-stop. Latched state requires explicit operator ack, zero-output validation, and explicit re-enable. No auto-rearm after a safety rollback.

### LOCKED — reference manipulator-Class Modeling Fidelity (High #6)

- Manifest parsing in `crates/roz-core/src/manifest.rs:528` MUST preserve joints, TCPs, grippers, sensors, workspaces — no flattening into generic channels at parse time.
- Capability publishing in `crates/roz-worker/src/main.rs:1394` MUST publish embodiment-specific capability descriptors derived from the manifest, not generic empty capabilities.
- `crates/roz-core/src/embodiment/model.rs:54` MUST be the single source of truth for the runtime view that controllers see.

### LOCKED — framework-fidelity hardware bench (Medium #7)

- A deterministic fake manipulator-class backend MUST exist (joints, gripper, encoders, current feedback) so CI exercises the manipulator path without hardware.
- The live-matrix script (`scripts/run_live_e2e_matrix.sh`) MUST add a manipulator row that runs the fake-backend tests deterministically (same posture as PX4/ArduPilot rows).
- Gated hardware bench tests (skippable, opt-in by env) MUST cover: tiny bounded motion, encoder/current feedback, channel order/sign/unit checks, physical e-stop latency.
- Serial/Dynamixel/OpenCR fixtures live behind a feature flag and are exercised when the relevant env is present.

### Claude's Discretion

- Concrete proto/API shapes for the runtime descriptor on dispatch (proto field, sidecar fetch, or signed envelope — propose during research, decide in plans).
- Exact state machine for the latched e-stop (ack/zero/disable). Use safety-engineering-standard semantics; document clearly.
- Cache strategy for `EmbodimentRuntime` resolution at dispatch time (cold-path latency budget vs. correctness on rotation).
- Trait surface details for IO factories — propose during research using the `Integration Policy` doc as the verdict authority.
- hardware bench test harness layout (separate crate vs. tests in `roz-test`).

</decisions>

<specifics>
## Specific Ideas

- **Single trait surface for IO factories.** Adding a new robot family should be a `impl IoFactory for X` plus registration, not a worker code change.
- **Runtime descriptor is signed.** Reuse the Phase 23 Ed25519 signing path so an unauthorized embodiment cannot be passed to a worker.
- **Force-e-stop must reject all-defaults.** A tick where every output equals its default is suspicious by definition during e-stop — explicit zero-validation is required.
- **Stale-heartbeat timer fires on `T_stale`** measured from last received message, not from arbitrary inter-message gap.
- **Capability publishing is the control-plane surface** that substrate-ide reads to render the embodiment — this must be lossless from manifest to capability.

</specifics>

<deferred>
## Deferred Ideas

- Vendor-specific firmware integration for any commercial arm — fixtures are fake-by-default. Real procurement is a future milestone.
- Cross-tenant embodiment sharing — single-tenant runtime authority is in scope; multi-tenant runtime federation is not.
- ROS2 / rclrs IO factories — listed in integration policy as future work; not in 26.10 scope.

</deferred>

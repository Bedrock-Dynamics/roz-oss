## Reviews Addressed

This plan is a REVIEWS-driven revision. Codex (gpt-5.5) findings driving the revision:

- **M3 (MEDIUM) — Stale-heartbeat tests use wrong time mechanism.** Original plan proposed `tokio::time::pause/advance` for the stale-heartbeat tests, but `HeartbeatTracker` uses `std::time::Instant` (verified at `crates/roz-safety/src/heartbeat.rs:54`: `workers: HashMap<String, Instant>`, and `:69`: `self.workers.insert(worker_id.to_owned(), Instant::now())`). Tokio's virtual time only affects `tokio::time::Instant` — `std::time::Instant` always reads the wall clock. **FIX (Option B per advisor recommendation):** Extract `run_stale_scan(now: Instant, threshold: Duration, tracker: &HeartbeatTracker) -> Vec<String>` as a pure function. Tests inject synthetic `Instant` values via `Instant::now() - Duration::from_secs(N)` — no tokio virtual time needed. Production loop wraps the pure fn with `tokio::time::interval`.
- **Choice rationale (Option B over Option A — injectable clock):** Refactoring HeartbeatTracker to inject `Box<dyn Clock>` is a larger surgery with downstream consumers. Pure-function extraction is one new fn and a one-line wrapper in main.rs. The advisor explicitly recommended B as the lighter option.
- **WIT bound is unaffected** — keep `MAX_TICK_OUTPUT_BYTES = 64 KiB` as in original plan.

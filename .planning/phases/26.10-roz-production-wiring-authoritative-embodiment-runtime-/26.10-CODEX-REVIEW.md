# Phase 26.10 — Codex Review Findings

## Framing

> **Roz is inspired by reference manipulator, NemoClaw, and Hermes for robotics.** These are the *reference architectures* Roz models itself after — not separate robot-family integrations. References to "reference manipulator-specific" fixtures (Dynamixel, OpenCR, manipulator hardware bench) in this review are validating the **framework-fidelity claim** that Roz can support that class of hardware, in the same way Phase 27 PX4 SITL CI validates the drone-class claim. Do **not** scope this phase as a new robot family or move reference manipulator work to a future milestone.

## Read

The repo has the right foundation: WIT-based WASM controllers, Wasmtime sandboxing, Copper tick loop, safety filtering, evidence, and embodiment abstractions. It is **not yet** an end-to-end Roz physical-control stack because deployment, robot IO, authoritative embodiment, edge placement, and hardware validation are not connected on the production path.

## Findings

### Blocking

1. **Normal agent/task path cannot deploy a live WASM controller.**
   Worker tools do not register `promote_controller`, `stop_controller`, or status tooling, and local runtime registers replay rather than live promotion.
   - `crates/roz-worker/src/main.rs:672`
   - `crates/roz-local/src/runtime.rs:1025`
   - `crates/roz-local/src/tools/promote_controller.rs:249`

2. **Worker-spawned Copper has no actuator or sensor IO.**
   `spawn_with_policy` expands to no `ActuatorSink` and no `SensorSource`, so the controller loop can run but cannot command or observe a robot.
   - `crates/roz-worker/src/main.rs:692`
   - `crates/roz-copper/src/handle.rs:249`
   - `crates/roz-copper/src/controller.rs:2395`

3. **Live controller loading requires an authoritative `EmbodimentRuntime`, but task dispatch only carries a control manifest.**
   This prevents safe reference manipulator deployment because runtime joint limits, workspace, TCP, and channel authority are missing.
   - `crates/roz-copper/src/controller.rs:553`
   - `crates/roz-local/src/tools/promote_controller.rs:183`
   - `crates/roz-server/src/routes/task_dispatch.rs:225`

### High

4. **Edge placement does not match the intended OodaReAct robot-control model.**
   `AUTO` currently defaults cloud unless explicit edge placement is requested.
   - `crates/roz-core/src/edge/mod.rs:9`
   - `crates/roz-server/src/grpc/agent.rs:2583`

5. **Safety has real gaps before hardware.**
   - Silent worker stale detection depends on receiving another heartbeat — `crates/roz-safety/src/main.rs:53`
   - Legacy WASM output can allocate before bounds validation — `crates/roz-copper/src/wit_host.rs:350`
   - Force e-stop can be bypassed by all-default outputs — `crates/roz-copper/src/controller.rs:1111`

6. **reference manipulator modeling is conceptually supported, but not production wired.**
   Core can model joints, TCPs, grippers, sensors, and workspaces, but TOML/examples flatten that into channels, synthesize loose limits, and publish generic empty capabilities.
   - `crates/roz-core/src/embodiment/model.rs:54`
   - `crates/roz-core/src/manifest.rs:528`
   - `crates/roz-worker/src/main.rs:1394`

### Medium

7. **Manipulator validation is mostly simulated and not reference manipulator-specific.**
   No reference manipulator artifacts, drivers, manifests, hardware bench tests, serial/Dynamixel/OpenCR fixtures, or physical e-stop latency checks. The live matrix covers mobile/PX4/ArduPilot deterministically; manipulator remains skippable / sim-oriented.
   - `scripts/run_live_e2e_matrix.sh:96`
   - `scripts/run_live_e2e_matrix.sh:111`

## Next Build Order

1. **Authoritative reference manipulator embodiment / runtime** — joints, gripper TCP, limits, workspace, sensors, channel bindings, and capabilities derived from that runtime.
2. **Worker Copper with real manipulator `ActuatorSink` / `SensorSource`** — and expose safe controller `promote` / `status` / `stop` tools.
3. **Harden safety before hardware** — stale heartbeat timer, bounded WASM host output, latched e-stop with ack/zero/disable, and no auto-rearm after safety rollback.
4. **Deterministic fake manipulator backend tests + gated hardware bench tests** — tiny bounded motion, encoder/current feedback, channel order/sign/unit checks, e-stop latency.

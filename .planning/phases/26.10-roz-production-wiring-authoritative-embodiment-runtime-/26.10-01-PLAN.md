## Reviews Addressed

This plan is a REVIEWS-driven revision. Codex (gpt-5.5) findings driving the revision:

- **H1 (HIGH) — Fail-closed ordering broken.** Original plan inserted runtime resolution "immediately before line 225" (the `TaskInvocation` literal at the END of `dispatch_task`). That placement is AFTER `roz_db::tasks::create` (line 157) AND AFTER the Restate workflow start (line 186). A missing or invalid runtime would leak a created-but-uninvoked task row + a started Restate workflow before failing. **FIX:** Move runtime resolution to immediately AFTER the host fetch + tenant filter at line 136, BEFORE line 157.
- **H1 (HIGH) — Tenant assertion missing on EmbodimentRow.** `embodiments::get_by_host_id` returns its own `tenant_id`. The host's tenant filter at :135 does not transitively cover the embodiment row. **FIX:** Explicit `EmbodimentRow.tenant_id == request.tenant_id` assertion before attaching.
- **H1 (HIGH) — TaskInvocation literal-site churn.** 13 literal sites would each need the new field set explicitly. **FIX:** Introduce `TaskInvocation::new(...)` constructor that defaults `embodiment_runtime: None`. Migrate all 13 sites to it. Acceptance criterion enforces zero unconstructed literals.
- **G1 — `TaskInvocation` lacked a `Default` impl** so the original plan's "..Default::default()" guidance was a compile hazard. The constructor pattern sidesteps this entirely.

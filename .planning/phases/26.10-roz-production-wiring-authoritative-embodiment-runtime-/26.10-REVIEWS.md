# Cross-AI Plan Review ‚Äî Phase 26.10 (physical-control production wiring)

## Codex Review (gpt-5.5)

**Summary**

The plans have the right architectural intent and good traceability to FW-01..FW-07, but I would not execute them as-is for a safety-critical phase. Several plans rely on assumptions that do not match the current repo: `TaskInvocation` has many struct literals and no builder/default, `EmbodimentRuntime` / `ControlInterfaceManifest` do not implement `Default`, `RobotCapabilities` is a Rust JSON struct rather than a proto message, and putting concrete backend factories in `roz-copper` risks a `roz-copper ‚Üî roz-mavlink` cargo cycle. The highest-risk gap is Plan 07: the latched e-stop design is directionally correct but not yet integrated into the actual controller loop, state publication, WAL boot path, worker NATS subscriptions, and rollback behavior in a way that is demonstrably fail-closed.

**Strengths**

- Clear requirement mapping: each FW requirement has a named plan and test intent.
- Reusing Phase 23 payload signing for inline `EmbodimentRuntime` is the right trust-boundary choice.
- The `set_output` bound-before-allocation fix is small, targeted, and directly addresses the WASM host DoS.
- The edge-placement pure-function tests in Plan 05 are well scoped.
- Feature-gating the fake manipulator backend is the right instinct.
- The plans correctly avoid making real Dynamixel/OpenCR hardware mandatory for CI.

**Concerns**

- **HIGH:** Plan 01 resolves `EmbodimentRuntime` after task creation and Restate workflow start in the current dispatch flow. A missing runtime should fail before any task/workflow side effect.
- **HIGH:** Plan 03 puts backend factory dispatch in `roz-copper` and asks for a `roz-mavlink` dependency. Since `roz-mavlink` depends on `roz-copper` traits, this likely creates a cargo cycle.
- **HIGH:** Plan 07 is not yet fail-closed. Latch state is proposed on controller structs, but the real loop has active/candidate/rollback controllers, existing rollback clears `estop_reason`, `ControllerState` has no latch field, and `Resume` already exists with broader semantics.
- **HIGH:** Plan 09‚Äôs deterministic test does not actually load/promote a controller, so it would not exercise the production path. Spawning Copper and sleeping is not enough.
- **HIGH:** Plan 09‚Äôs capability projection cannot compile as written: current `roz_core::capabilities::RobotCapabilities` only has `joints: Vec<String>`, `sensors: Vec<String>`, no TCP/workspace/limit descriptors, and no `roz_v1::RobotCapabilities` proto exists.
- **MEDIUM:** Plan 02‚Äôs manifest schema does not match current model types. Actual joints need `joint_type`, parent/child links, `axis: [f64; 3]`, `origin`, `max_jerk`, and `max_torque: Option<f64>`.
- **MEDIUM:** Plan 05 derives physical-tool presence only from client-supplied `start.tools`; worker-registered physical tools may not be visible at that point.
- **MEDIUM:** Plan 06 stale-heartbeat tests propose `tokio::time::pause/advance`, but `HeartbeatTracker` uses `std::time::Instant`, so virtual time will not work without refactoring.
- **MEDIUM:** Many plans include `todo!()` / placeholder fixture builders and rely on `Default` impls that do not exist.

**Suggestions**

- Move Plan 01 runtime resolution before `roz_db::tasks::create` and before Restate start. Also update every `TaskInvocation { ... }` literal in the same plan, or introduce a safe constructor/builder.
- Keep `IoFactory` trait in `roz-copper`, but move concrete registry/factory selection to `roz-worker` or a new backend-aggregation crate. Do not make `roz-copper` depend on `roz-mavlink`.
- Redesign Plan 07 around a system-level latch in the Copper loop, not per prepared controller. Add `latch_state` to `ControllerState`, explicit `AckEstop` / `ResumeAfterZeroVerified` commands, handle initialization from WAL, and define behavior when no sensor frame is present: remain latched.
- Make e-stop zero output typed by channel/interface. Universal numeric `0.0` is safe for velocity/torque, but may be unsafe for position channels if it commands ‚Äúgo to zero‚Äù instead of ‚Äúdisable/hold‚Äù.
- In Plan 05, include server-owned and worker-known physical tools in the AUTO decision, or make `host_id + known controller-capable worker` edge by default.
- For Plan 06, extract a testable `run(...)` function with configurable stale threshold/scan period, or refactor `HeartbeatTracker` to use an injectable clock.
- For Plan 08, implement `DiscreteCommandSink` or explicitly document why manipulator fake only needs continuous `ActuatorSink` for this phase. Add current feedback and gripper/TCP simulation if FW-07 acceptance depends on them.
- For Plan 09, extract `project_capabilities` into a real helper module and first extend `roz_core::capabilities` with typed descriptors. Use startup `embodiment_runtime` from `robot_toml` or re-publish per task; `invocation` is not in scope at startup.
- Add one end-to-end deterministic test that goes through worker dispatch/tool registration/promote_controller, not just direct Copper spawning.

**Overall Risk Assessment: HIGH**

The roadmap is directionally strong, but the executable plans contain enough compile-time mismatches and safety-state integration gaps that autonomous execution would likely produce churn or a false sense of safety. I would revise Plans 01, 03, 07, and 09 before starting implementation, then run the rest behind those corrected interfaces.

## Consensus Summary

> Single reviewer (codex/gpt-5.5). No cross-AI consensus ‚Äî treat findings as one strong opinion, not validated agreement.

### Codex's Top Concerns (all HIGH severity)

1. **Plan 01 ‚Äî Side-effect ordering breaks fail-closed contract.** `EmbodimentRuntime` is resolved after `roz_db::tasks::create` and Restate workflow start in the proposed flow. A missing runtime should fail before any task/workflow side effect.
2. **Plan 03 ‚Äî Cargo cycle risk on `roz-copper ‚Üî roz-mavlink`.** Putting concrete backend factory dispatch in `roz-copper` while requiring a `roz-mavlink` dep creates a likely cycle (since `roz-mavlink` already depends on `roz-copper` traits). Move concrete factory selection to `roz-worker` or a new aggregation crate.
3. **Plan 07 ‚Äî Latched e-stop is not yet fail-closed at the loop level.** The plan adds latch state to controller structs, but the actual Copper loop has active/candidate/rollback controllers, existing rollback already clears `estop_reason`, `ControllerState` has no latch field, and `Resume` already exists with broader semantics. Needs a system-level latch in the loop, explicit `AckEstop` / `ResumeAfterZeroVerified` commands, WAL boot init, and "no sensor frame ‚Üí remain latched" behavior.
4. **Plan 09 ‚Äî Tests don't exercise the production path.** The "deterministic" test spawns Copper and sleeps; it does not promote a controller through the agent/task path.
5. **Plan 09 ‚Äî Capability projection won't compile as written.** Current `roz_core::capabilities::RobotCapabilities` is `joints: Vec<String>` + `sensors: Vec<String>`, with no TCP/workspace/limit descriptors and no `roz_v1::RobotCapabilities` proto. The plan presumes typed descriptors that don't exist.

### Codex's Medium Concerns

- **Plan 02** ‚Äî Manifest schema mismatch. Real joints need `joint_type`, parent/child links, `axis: [f64; 3]`, `origin`, `max_jerk`, `max_torque: Option<f64>`. Plan understates the change surface.
- **Plan 05** ‚Äî `start.tools` only sees client-supplied tools; worker- and server-registered physical tools may not be visible at session start.
- **Plan 06** ‚Äî Stale-heartbeat tests use `tokio::time::pause/advance`, but `HeartbeatTracker` uses `std::time::Instant`. Virtual time won't work without refactoring to an injectable clock.
- **Plans broadly** ‚Äî `todo!()` and `Default::default()` calls on types that don't implement `Default` will not compile.

### Codex's Risk Assessment

**Overall: HIGH.** "I would not execute them as-is for a safety-critical phase." Recommend revising Plans 01, 03, 07, and 09 before implementation, then running the rest behind the corrected interfaces.
